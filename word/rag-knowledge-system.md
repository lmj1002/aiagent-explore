# RAG（检索增强生成）全流程知识体系深度解析

> 本文档系统梳理 RAG（Retrieval-Augmented Generation）的完整技术栈，涵盖离线索引管线、在线推理管线、六大场景模块，并重点深挖切片方法、向量数据库选型、相似性算法三大核心技术领域。资料整合自 2025-2026 年最新实践与学术研究。

---

## 一、RAG 概述与核心架构

### 1.1 什么是 RAG？

RAG（Retrieval-Augmented Generation，检索增强生成）是一种将**信息检索**与**大语言模型生成**相结合的架构范式。其核心思想是：在 LLM 生成回答之前，先从外部知识库中检索相关文档片段，将其作为上下文注入 Prompt，从而让模型基于"最新、最相关、可溯源"的信息生成回答。

### 1.2 RAG 的三代演进

| 阶段 | 时期 | 特征 | 局限性 |
|------|------|------|--------|
| **Naive RAG** | 2023 | 简单的"索引-检索-生成"线性管道 | 召回率低、上下文割裂、无法处理复杂查询 |
| **Advanced RAG** | 2024 | 引入查询重写、混合检索、重排序 | 仍为单次检索，缺乏迭代能力 |
| **Modular RAG** | 2025-2026 | 组件化、可插拔的模块化架构，各环节独立可替换 | 架构设计复杂度提升 |

### 1.3 核心公式

```
RAG = Retriever + Generator

Retriever = Chunking + Embedding + VectorDB + Search
Generator = Prompt + Context + LLM
```

## 二、离线索引管线（Indexing Pipeline）

> 离线管线在查询到来**之前**完成，关注高吞吐而非低延迟。处理质量直接影响在线检索效果。

### 2.1 文档预处理与内容提取

原始文档来源多样（PDF、HTML、Markdown、数据库行、API 响应），预处理是质量的第一道关。

| 步骤 | 操作 | 目的 |
|------|------|------|
| **格式标准化** | 统一编码、去除特殊字符 | 减少噪声 |
| **内容清洗** | 移除页眉页脚、广告、导航栏 | 保留正文 |
| **多模态提取** | OCR 提取图片文字、表格解析 | 覆盖所有信息载体 |
| **元数据提取** | 来源、作者、日期、部门、页码 | 辅助检索过滤 |
| **版本管理** | 追踪文档变更历史 | 支持回滚与增量更新 |

### 2.2 切片方法（Chunking Strategy）——重点深度剖析

> 切片是 RAG 系统中**对检索质量影响最大的单一因素**。切分过细导致上下文断裂，切分过粗导致检索精度下降。

#### 2.2.1 切片方法全景一览

| 方法 | 原理 | 复杂度 | 语义保持 | 适用场景 |
|------|------|:---:|:---:|------|
| **固定大小切片** | 按硬 Token 上限切割 | ★ | 低 | 简单、均匀内容 |
| **递归切片** | 按优先级分隔符逐级切割 | ★★ | 中 | **通用首选**，LangChain 默认 |
| **语义切片** | 用 Embedding 相似度检测主题边界 | ★★★★ | 高 | 专业化文档、需要语义连贯 |
| **Max-Min 语义切片** | 用块内 max/min 相似度决定边界 | ★★★★ | 很高 | 语义优先的高精度场景 |
| **Query-Adaptive 切片** | 根据用户查询动态调整分块 | ★★★★★ | 最高 | 查询感知的智能检索 |
| **文档结构切片** | 按 Markdown 标题/PDF 页码/代码函数 | ★★★ | 高 | 技术文档、法律文件 |
| **Agentic 切片** | LLM Agent 自主判断逻辑边界 | ★★★★★ | 很高 | 高质量需求，计算成本极高 |
| **Late Chunking** | 先整篇 Embedding 再在 Token 级切分 | ★★★★ | 很高 | ColBERT 风格的迟交互范式 |

#### 2.2.2 核心方法详解

**（1）固定大小切片（Fixed-Size Chunking）**

最朴素的策略。设定一个硬上限（如 512 Token），累计到阈值即切分。

```
优点：实现简单，速度最快
缺点：常在句子中间切断，语义碎片化
推荐参数：chunk_size=500-800 Token, overlap=10-20%
```

**（2）递归切片（Recursive Chunking）——通用首选**

LangChain 和 LlamaIndex 的默认策略。自上而下用优先级分隔符逐层切割：

```
英文分隔符优先级：
  "\n\n" → "\n" → ". " → " " → ""

中文分隔符优先级：
  "\n\n" → "\n" → "。" → "！" → "？" → "；" → "，" → " "
```

> **业界共识：递归切片是效果与复杂度的最佳折衷，初学者从它开始。**

**（3）语义切片（Semantic Chunking）——前沿方向**

核心思路：计算相邻句子的 Embedding 余弦相似度，在相似度"陡降"处切分——这些位置即是主题自然边界。

```
流程：
1. 将文档拆为句子级
2. 为每个句子生成 Embedding
3. 计算相邻句子间的余弦相似度
4. 识别相似度断崖（dip point）作为切分边界
5. 合并断裂点之间的句子形成语块
```

**（4）Max-Min 语义切片（Milvus, 2025.12）**

语义切片的进阶版。引入块内质量评估：

```
核心规则：
  如果 新句子与当前块中句子的最大相似度 ≥ 当前块内所有句子对的最小相似度
  那么 该句子被纳入当前块
  否则 在当前位置切分，开启新块
```

优势：确保块内语义一致性下限不会持续下降，形成"质量门控"。

**（5）Query-Adaptive Semantic Chunking (QASC, 2026.04)**

将用户查询嵌入到切片阶段，实现**查询感知的动态切片**：

| 步骤 | 描述 |
|------|------|
| **种子句识别** | 用查询 Embedding 在文档中找到最相关句 |
| **上下文窗口扩展** | 从种子句向两侧扩展，纳入关联上下文 |
| **块级得分聚合** | 按查询相关性对分块评分 |

> **实验数据**：F1 达 0.85，比固定切片高 18-27%

**（6）Late Chunking（迟切片/Token-Level 嵌入）**

突破传统"先切后嵌"范式，改为"先嵌后切"：

```
传统方式：文档 → 切块 → 每个块独立Embedding → 存库
Late Chunking：文档 → 整篇Token级Embedding → 检索时按Token切分
```

这种方式保留了 Token 间的完整语义关系，避免了"孤儿块"问题。ColBERT 是此范式的代表。

#### 2.2.3 切片参数配置指南

| 参数 | 推荐起始值 | 调整方向 |
|------|-----------|----------|
| **chunk_size** | 500-800 Token | 精确问答→缩小；需要上下文→放大 |
| **chunk_overlap** | 10-20% | 语义连贯性要求高→增大 |
| **分隔符优先级** | 段落→句子→字符 | 结构化文档→优先按标题/MD层级 |
| **Small2Big** | 检索用小块，返回用大块 | 平衡精度与上下文丰富度 |

#### 2.2.4 切片策略决策树

```
文档类型？
├── 结构化技术文档（Markdown/代码）→ 文档结构切片 / 递归切片
├── 非结构化长文本（论文/报告）→ 语义切片 / Max-Min 语义切片
├── 对话/客服记录 → 按轮次切片 / 固定大小
├── 法律/合规文档 → 文档结构切片 + overlap 增大
├── 混合文档集合 → Adaptive Chunking（自动选择）
└── 需要最高精度 → Query-Adaptive / Late Chunking
```

### 2.3 元数据增强与上下文富化

> 解决"孤儿块"（Orphan Chunk）问题——脱离文档上下文后，纯文字块语义不完整。

**增强策略：**

| 策略 | 方法 | 示例 |
|------|------|------|
| **文档标题前缀** | 在每个块前添加文档标题和章节路径 | `[《劳动合同法》> 第三章 > 第39条] 用人单位单方解除...` |
| **关键实体标注** | LLM 自动提取实体标签 | `[实体: 竞业限制, 违约金]` |
| **相邻块摘要** | 在块中添加前后块的简要描述 | 提升跨块检索的上下文感知 |
| **问题生成** | 为每个块生成可能匹配的用户问题 | "这个块可以回答：竞业限制的期限是多久？" |

**2025 年研究数据：** 递归切片 + TF-IDF 加权 Embedding 可实现 **82.5% 精确率**；朴素切片 + Prefix-Fusion 的 NDCG 达 **0.813**。

### 2.4 嵌入模型（Embedding Model）选型

| 模型 | 维度 | 语言 | 优势 | 适用场景 |
|------|:---:|------|------|----------|
| **OpenAI text-embedding-3-small** | 1536 | 多语言 | 性价比高，质量稳定 | 生产级通用 RAG |
| **OpenAI text-embedding-3-large** | 3072 | 多语言 | 最高精度 | 高质量要求的金融/法律 |
| **BGE bge-large-zh-v1.5** | 1024 | 中英 | 开源、中文表现优异 | 中文 RAG 首选 |
| **BGE-M3** | 1024 | 多语言(100+) | 支持稠密+稀疏混合 | 多语言场景 |
| **Cohere Embed v3** | 1024-4096 | 多语言 | 企业级特性 | 全球多语种平台 |
| **all-MiniLM-L6-v2** | 384 | 英文 | 轻量快速 | 原型验证、资源受限 |
| **Jina Embeddings v2** | 768 | 中英 | 支持 8K 上下文窗口 | 长文档嵌入 |

**选型原则：**
1. 中文场景优先选择 BGE 系列（bge-large-zh 或 BGE-M3）
2. 多语种商业产品选 OpenAI 或 Cohere
3. 资源受限选 all-MiniLM-L6-v2（384维，内存友好）
4. 长文档优先选支持长上下文的 Jina v2

### 2.5 向量数据库选择——重点深度剖析

> 向量数据库是 RAG 系统的核心基础设施。选型错误会导致延迟失控、成本暴涨、检索质量下降。

#### 2.5.1 六大主流方案全景对比

| 维度 | **Chroma** | **Pinecone** | **Qdrant** | **Milvus** | **Weaviate** | **pgvector** |
|------|------------|-------------|------------|------------|-------------|-------------|
| **开源协议** | Apache 2.0 | 商业闭源 | Apache 2.0 | Apache 2.0 | BSD-3 | PostgreSQL |
| **开发语言** | Python | — | **Rust** | Go + C++ | Go | C |
| **部署方式** | 嵌入式 | 全托管 SaaS | 自托管/Cloud | 自托管/Zilliz Cloud | 自托管/Cloud | PG 插件 |
| **最大规模** | 百万级 | 亿级 | 亿级 | **十亿+** | 千万-亿级 | 百万-千万级 |
| **GPU 加速** | ❌ | ❌ | ❌ | ✅ 10x | ❌ | ❌ |
| **混合搜索** | ❌ | ✅ (2025) | ✅ | ⚠️ 有限 | ✅ **原生** | ✅ (SQL+向量) |
| **权限控制** | ❌ | ✅ | ✅ v1.10+ | ✅ | ✅ 企业版 | ✅ PG RBAC |

#### 2.5.3 逐库深度解析

**Chroma — 原型验证首选**

| 项目 | 说明 |
|------|------|
| 核心优势 | 5行代码启动；与 LangChain/LlamaIndex 深度集成；支持内存/DuckDB/ClickHouse 后端 |
| 核心劣势 | 不适合大规模生产（建议<10万向量）；无内置权限、审计、备份 |
| 适用场景 | PoC 验证、教学、个人项目、本地开发测试 |
| 一句话 | **快速跑通 RAG 流程的最佳选择，但不建议直接上生产** |

**Pinecone — 零运维全托管**

| 项目 | 说明 |
|------|------|
| 核心优势 | 零运维；自动扩展、高可用 SLA；Serverless 架构 |
| 核心劣势 | 成本高（$0.20/百万次查询起）；厂商锁定；数据存 AWS us-east |
| 适用场景 | 快速上线 SaaS 产品、无 DevOps 团队 |
| 一句话 | **预算充足、追求省心上线的首选，注意数据主权和成本** |

**Qdrant — 高性能性价比之选**

| 项目 | 说明 |
|------|------|
| 核心优势 | Rust 实现，P99 <20ms；内存效率比 Milvus 低 40%；Payload 过滤优异；支持量化压缩（降内存 4-8x）；原生全文+向量混合检索 |
| 核心劣势 | 生态工具较少（相比 Milvus）；中文文档较弱；不支持 GPU 加速 |
| 适用场景 | 延迟敏感的实时对话系统、推荐引擎、生产级 RAG |
| 2025 亮点 | DiskANN + HNSW 混合索引，更大规模低成本部署 |
| 一句话 | **中大规模生产 RAG 的首选，性能与成本的绝佳平衡** |

**Milvus — 企业级十亿规模**

| 项目 | 说明 |
|------|------|
| 核心优势 | 支持十亿级向量；GPU 加速（10×提速）；插件化 ANN 算法；完整监控；国产开源，中文文档完善 |
| 核心劣势 | 架构复杂（依赖 etcd/MinIO/Pulsar）；学习曲线陡峭；小规模过度设计 |
| 适用场景 | 超大规模企业平台、金融/电商/广告系统、多模态 RAG |
| 一句话 | **数据量过亿、需要 GPU 加速的规模化场景，没有对手** |

**Weaviate — 知识图谱+向量融合**

| 项目 | 说明 |
|------|------|
| 核心优势 | 原生语义图谱（对象间可建关系）；内置 BM25 关键词检索；混合搜索开箱即用；Schema-first 设计 |
| 核心劣势 | 内存占用高（Go+HNSW）；集群配置较复杂 |
| 适用场景 | 需要语义关联+知识图谱的复杂知识库 |
| 一句话 | **关键词+向量混合搜索、或知识图谱关联查询时的最佳选择** |

**pgvector — 零迁移成本的 PostgreSQL 方案**

| 项目 | 说明 |
|------|------|
| 核心优势 | 在现有 PG 上开插件即可用；ACID + SQL 过滤 + 向量搜索一体；零新增运维 |
| 核心劣势 | 十亿级性能不及专用向量库；索引构建较慢 |
| 适用场景 | 已深度使用 PG 的团队、中小规模 RAG |
| 一句话 | **如果已有 PG 基础设施，这是最务实的起点** |

### 2.6 索引更新策略

| 策略 | 描述 | 适用场景 |
|------|------|----------|
| **全量重建** | 定期清空重建全部索引 | 数据量小、变动频繁 |
| **增量更新** | 仅对变更文档重新索引 | 大规模知识库 |
| **触发式更新** | 文档新增/变更时自动触发索引 | 实时性要求高 |
| **版本快照** | 保留索引历史版本用于回滚 | 合规审计场景 |
| **流式处理** | Kafka/Pulsar 实时流处理新数据 | 实时数据源 |

## 三、在线推理管线（Inference Pipeline）

> 在线管线每次查询实时执行，关注低延迟、高精度、可解释性。

### 3.1 查询预处理

| 技术 | 原理 | 适用场景 |
|------|------|----------|
| **查询重写** | 扩展缩写、改写为更清晰的表达 | 用户输入质量不稳定 |
| **Step-Back Prompting** | 先生成更抽象的"回退问题"再检索 | 复杂推理问题 |
| **HyDE** | LLM 生成假设性答案 → Embed 答案 → 用答案向量去检索 | 查询-文档语义鸿沟大 |
| **子查询分解** | 复杂问题拆分为多个简单子问题分别检索后合并 | 多跳推理、对比分析 |
| **多轮改写** | 结合对话历史将指代消解为完整查询 | 多轮对话场景 |

**HyDE（Hypothetical Document Embeddings）原理：**
```
用户查询 → LLM生成一个假想的答案文档 → Embed这个答案 → 用答案向量检索真实文档
```
"假答案"的 Embedding 往往比"短查询"的 Embedding 更接近真实文档的 Embedding 分布。

### 3.2 检索策略

#### 3.2.1 三种检索范式

| 方法 | 原理 | 优势 | 劣势 |
|------|------|------|------|
| **稠密检索 (Dense)** | 查询 Embedding → 向量相似度搜索 | 语义匹配，同义词泛化 | 精确术语匹配弱 |
| **稀疏检索 (Sparse)** | BM25 词频-逆文档频率 | 精确关键词匹配 | 词汇鸿沟 |
| **混合检索 (Hybrid)** | 稠密+稀疏并行 → RRF 融合 | **两者互补，生产标配** | 复杂度略增 |

#### 3.2.2 Reciprocal Rank Fusion（RRF）——混合检索的融合引擎

**公式：**
```
RRF_score(d) = Σ (1 / (k + rank_i(d)))
```
其中 `k` 为平滑常数（通常取 60），`rank_i(d)` 是文档 d 在第 i 个检索列表中的排名。

**工作流程：**
```
用户查询
    │
    ├──→ 稠密向量检索 → Top-50 → 排名列表A
    │
    └──→ BM25稀疏检索 → Top-50 → 排名列表B
                    │
                    ▼
            RRF 融合排名 → Top-25
                    │
                    ▼
              送入重排序器
```

> **2025 年共识：混合检索 + RRF 是企业生产的标准配置，纯向量搜索已无法满足复杂检索需求。**

---

### 3.3 相似性算法（Similarity Metrics）——重点深度剖析

> 相似性度量是检索质量的"无声杀手"——选错指标不会导致系统崩溃，但会缓慢侵蚀回答的相关性和准确性，且很难被察觉。

#### 3.3.1 四大核心指标数学原理

**（1）余弦相似度（Cosine Similarity）——业界默认标准**

```
cos(A, B) = (A · B) / (||A|| × ||B||)
         = Σ(Aᵢ × Bᵢ) / (√ΣAᵢ² × √ΣBᵢ²)
```

| 特性 | 说明 |
|------|------|
| **测量对象** | 向量间的**夹角**（方向），忽略模长 |
| **取值范围** | [-1, 1]（1=完全相同方向，0=正交，-1=完全相反） |
| **核心优势** | **尺度不变性**——无论文档长短，只比较语义方向 |
| **适用场景** | 自然语言文本检索（RAG 默认首选） |

**（2）点积 / 内积（Dot Product / Inner Product）**

```
dot(A, B) = Σ(Aᵢ × Bᵢ)
```

| 特性 | 说明 |
|------|------|
| **测量对象** | 方向 × 模长（综合得分） |
| **取值范围** | (-∞, +∞) 无上界 |
| **核心优势** | 计算最快（省去除法步骤） |
| **关键陷阱** | **对向量模长敏感**——长文档块自动获得更高分数，可能淹没短而相关的结果 |

> **⚠️ 使用前提：向量必须 L2 归一化到单位长度。归一化后 Dot Product ≈ Cosine Similarity。**

**（3）欧几里得距离（Euclidean Distance / L2）**

```
L2(A, B) = √Σ(Aᵢ - Bᵢ)²
```

| 特性 | 说明 |
|------|------|
| **测量对象** | 向量间**绝对空间距离** |
| **取值范围** | [0, +∞)（0=完全相同） |
| **在 RAG 中的弱点** | 高维空间（768+维）中"维数灾难"明显——最近邻与最远邻的距离趋于相同 |
| **适用场景** | 低维数据、图像特征、几何数据 |

**（4）曼哈顿距离（Manhattan Distance / L1）**

```
L1(A, B) = Σ|Aᵢ - Bᵢ|
```

| 特性 | 说明 |
|------|------|
| **测量对象** | 各维度绝对差异之和 |
| **核心优势** | 对**异常值**比 L2 更鲁棒 |
| **适用场景** | 稀疏向量、离散特征空间、结构化数据 |

#### 3.3.2 综合对比表

| 指标 | Cosine | Dot Product | Euclidean | Manhattan |
|------|:---:|:---:|:---:|:---:|
| **关注维度** | 方向（语义） | 方向+强度 | 绝对距离 | 绝对差异 |
| **模长敏感** | ❌ 不敏感 | ⚠️ **敏感** | ✅ 敏感 | ✅ 敏感 |
| **高维表现** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐ | ⭐⭐ |
| **计算速度** | 快 | **最快** | 中等 | 中等 |
| **RAG 推荐度** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐ | ⭐⭐ |
| **语义理解** | 最优 | 良好 | 差 | 差 |

#### 3.3.3 选型决策树

```
你的场景是？

├── 文本语义检索（RAG默认）→ Cosine Similarity ✅
│
├── 向量已 L2 归一化 + 追求极致速度 → Dot Product ✅
│   ⚠️ 必须确认归一化！否则长文档主导结果
│
├── 低维几何/物理特征数据 → Euclidean (L2) ✅
│
├── 稀疏向量 + 需要异常鲁棒 → Manhattan (L1) ✅
│
└── 不确定？ → 用模型训练时指定的指标 ✅
    （大多数文本 Embedding 模型用 Cosine 训练）
```

#### 3.3.4 学术基准数据（EvaRAG 2025 研究）

| 配置 | 指标 | Coverage | Correctness | Faithfulness | 延迟 |
|------|:---:|:---:|:---:|:---:|:---:|
| HNSW + InnerProduct + Fusion + minilm | IP | **0.942** | **0.909** | **0.970** | 中等 |
| IVF + L2 + Hierarchical | L2 | 较低 | 较低 | 较低 | **最低 1.7ns** |

> **结论：无单一最优配置。需根据"精度 vs 延迟"的实际优先级选择。**

#### 3.3.5 无声的失败模式

> "你的系统会'正常运转'——仍会返回答案，仍会检索出一些块，LLM 仍会自信地生成。但相关性在缓慢衰减：稍差的回答、微妙的事实错误、'感觉不太对'的回复。因为没有任何告警，所以没人注意到。"

**防护清单：**
1. 确认 Embedding 模型的训练指标，检索时保持一致
2. 如果用 Dot Product，验证向量是否已 L2 归一化
3. 在上线前用评估集对比不同指标的实际效果
4. 切换指标后做 A/B 测试，不要只看离线评估

---

### 3.4 重排序（Re-ranking）

> "检索 50 个候选 → 重排序 → 保留 Top 5-10"已成为生产 RAG 的标准模式。

| 重排序器 | 类型 | 优势 |
|----------|------|------|
| **BGE-Reranker-v2** | Cross-encoder | 开源、中英表现优秀 |
| **Cohere Rerank** | API 服务 | 多语言、零部署 |
| **ms-marco-MiniLM-L-6-v2** | Cross-encoder | 轻量快速 |
| **MMR (最大边际相关性)** | 算法级 | 确保结果多样性 |

**重排序为什么重要？**
- Bi-encoder（Embedding 模型）将查询和文档**独立**编码，无法捕捉精细交互
- Cross-encoder 将"查询-文档对"**联合**编码，精度远高于 Bi-encoder，但速度较慢
- 两阶段策略结合了二者优势：粗筛用 Bi-encoder（快），精选用 Cross-encoder（准）

### 3.5 上下文组装与 Prompt 构造

```
System Prompt:
"你是一个基于知识库的问答助手。请仅根据以下提供的上下文回答用户问题。
如果上下文不包含足够信息，请明确告知，不要编造。"

检索到的上下文：
[Chunk 1] 来源: 《XX手册》第3章 | ...内容...
[Chunk 2] 来源: 《YY白皮书》第5节 | ...内容...

用户问题：...

请给出答案并标注信息来源。
```

**组装技巧：**
- 最相关的块放在开头和结尾（LLM 倾向于关注起始和末尾位置）
- 每个块附带来源元数据，用于引文生成
- 控制总 Token 数在模型上下文窗口的 60-70%，为推理留出空间

### 3.6 后处理

| 步骤 | 方法 | 目的 |
|------|------|------|
| **事实校验** | 逐句对照检索上下文验证 | 减少幻觉 |
| **引文格式化** | 自动附加 `[来源: 文档名, 页码]` | 可溯源 |
| **安全审查** | PII 检测、内容策略合规 | 防止泄露与违规 |
| **置信度标注** | 基于检索得分给出置信度 | 提示用户交叉验证 |

## 四、不同场景独立模块

> RAG 不是一个"一刀切"的方案。不同场景需要不同的架构选型和策略组合。

### 4.1 基础问答系统（Naive → Advanced RAG）

**适用场景：** 单轮 FAQ、企业知识库问答、文档搜索

**推荐架构：**
```
递归切片 + BGE Embedding + Qdrant + 混合检索(RRF) + BGE-Reranker + GPT-4o
```

**关键参数：**
- chunk_size: 500-800 Token
- overlap: 15%
- top_k 检索: 25 → 重排序 → 5
- 温度: 0-0.2

**进阶优化：**
- 查询重写 + 多轮对话历史感知
- 大块返回上下文（Small2Big）
- 基于元数据的检索过滤（按部门/日期/文档类型）

---

### 4.2 代码检索（Code RAG）

**技术要点：**

| 维度 | 策略 |
|------|------|
| **切片方式** | 按函数/类边界切片，保留 AST 结构信息 |
| **嵌入模型** | 代码专用模型（如 CodeBERT、UniXcoder）或通用模型（text-embedding-3） |
| **检索增强** | 代码块不参与语义图构建，仅做向量检索（避免噪声） |

**行业案例：**
- **Morgan Stanley DevGen.AI**：基于 GPT-4 的代码分析代理，已审查 **900万行代码**，节省 **28万开发小时**

**推荐工具链：**
- LlamaIndex CodeSplitter（AST-aware）
- 结合代码仓库结构（文件路径、import 关系）作为检索过滤

---

### 4.3 多模态 RAG（Multimodal RAG）

**两条技术路线：**

| 路线 | 方法 | 优势 | 劣势 |
|------|------|------|------|
| **多模态 Embedding** | CLIP/BLIP 统一映射图文到同一向量空间 | 端到端语义一致 | 模型选择有限，训练成本高 |
| **LLM 图像摘要** | GPT-4V/GPT-4o 将图像转为文字描述，统一文本检索 | 实现简单，兼容现有文本 RAG | 图像语义信息有损 |

**性能数据：**
- **MMed-RAG**（医疗多模态）：事实准确率提升 **43.8%**
- **RULE**（EMNLP 2024）：事实准确率提升 **47.4%**
- 实施多模态 RAG 的企业报告生产力提升 **30-42%**

**生产挑战：** 高计算成本、跨模态幻觉、集成复杂度高。2025 年仍处于"早期采用者"阶段。

---

### 4.4 知识图谱 + GraphRAG

**核心原理：** 将文档中的实体和关系提取出来构建知识图谱，支持多跳推理和全局主题查询。

**构建流程：**
```
文档 → 实体识别 → 关系抽取 → 图构建 → 社区检测 → 图谱存储
```

**GraphRAG 三阶段：**

| 阶段 | 操作 | 输出 |
|------|------|------|
| **图构建** | LLM 提取实体和关系；Leiden 社区检测 | 实体-关系网络 + 社区层级 |
| **图谱索引** | 为每个社区生成摘要；社区+实体分别 Embedding | 两个可查询的向量空间 |
| **图谱检索** | 局部搜索（实体邻域）/ 全局搜索（社区摘要） | 结构化上下文 |

**性能对比：**
| 指标 | GraphRAG | 纯向量 RAG |
|------|:---:|:---:|
| 多跳推理准确率 | **54.2%** | 16.7% |
| 提升倍数 | — | **3.3×** |

**适用场景：**
- ✅ 合规审查（"这批合同中哪些存在合规风险？"）
- ✅ 企业知识管理（多源异构数据）
- ✅ 研究分析（"最近有哪些关于XX的新发现？"）
- ❌ 简单单文档问答
- ❌ 实时性要求极高的场景

**LazyGraphRAG（2024.11）突破：**
- 索引成本降低 **99.9%**
- 查询成本降低 **700倍以上**
- 使 GraphRAG 在生产中的可行性大幅提升

**主流图数据库：** Neo4j、NebulaGraph

### 4.5 Agentic RAG

**架构原理：** LLM 作为 Agent 嵌入 RAG 管道，自主决策检索策略、评估结果、迭代改进。

**核心能力：**
- 判断是否需要检索（某些问题可从参数化知识直接回答）
- 选择数据源和检索策略
- 分解复杂问题为子查询链
- 评估检索结果是否充分，决定是否追加检索

**主流框架：**
- **LangGraph**：状态图编排，适合复杂多步工作流
- **LlamaIndex AgentQueryEngine**：内置 Agent 推理循环
- **Microsoft AutoGen**：多 Agent 对话式协作

**⚠️ 关键数据：90% 的 Agentic RAG 项目在生产中失败**

| 失败原因 | 说明 |
|---------|------|
| **链式失败累积** | 10步工作流每步95%成功率 → 整体仅60% |
| **不确定性难调试** | Agent 的自主决策路径难以复现和调试 |
| **成本/延迟爆炸** | 多轮检索+推理的 Token 消耗和响应时间远超预期 |
| **过度工程化** | 简单问题不需要复杂 Agent |

**成功模式共同特征：**
- 限定高价值、有边界的场景
- 分步骤验证工具可靠性
- 加入人在回路（Human-in-the-Loop）
- 充分的监控和可观测性

---

### 4.6 长文档 RAG（LongRAG）

**核心挑战：** 传统 RAG 在处理超长文档（如数百页 PDF 合同、学术论文）时面临语义切割失效和上下文窗口限制。

**解决方案矩阵：**

| 方案 | 描述 |
|------|------|
| **RAPTOR** | 递归摘要树——底层是切片，上层是摘要，支持从不同层级检索 |
| **LongRAG** | 长上下文 LLM（128K+ Token）直接处理完整文档，省去切割步骤 |
| **层级检索** | 先检索文档，再在文档内定位具体段落 |
| **Small2Big** | 检索用小粒度块（精确），返回时扩展为大粒度块（上下文完整） |

**选型建议：**
- 文档数少但每篇长 → LongRAG（直接用长上下文 LLM）
- 文档数多且需跨文档查询 → RAPTOR + 层级检索

## 五、评估体系与可观测性

### 5.1 三维评估框架

| 维度 | 指标 | 方法 |
|------|------|------|
| **检索质量** | Precision@K, Recall@K, MRR, NDCG | 对比检索结果与标注的相关块 |
| **生成质量** | Faithfulness（忠实度）, Answer Relevance, Context Precision | RAGAS / TruLens / LLM-as-Judge |
| **端到端体验** | 用户满意度, 任务完成率 | 线上 A/B 测试、反馈采集 |

### 5.2 关键评估工具

| 工具 | 用途 | 特点 |
|------|------|------|
| **RAGAS** | 自动化 RAG 评估 | Faithfulness + Relevance + Precision 三位一体 |
| **TruLens** | 反馈函数驱动评估 | 支持自定义评估链 |
| **LangSmith / LangFuse** | 全链路追踪 | 调试 + 监控 + 评估统一平台 |
| **Phoenix (Arize)** | LLM 可观测性 | 检索+生成的端到端可视化 |
| **DeepEval** | RAG 单元测试 | 类 pytest 的开发体验 |

### 5.3 Golden Dataset 方法

构建 `(问题, 期望答案, 相关块ID)` 三元组集合：

1. 人工标注 50-100 个代表性查询
2. 记录对应的正确答案和应检索到的文档块
3. 每次系统变更后运行评估
4. 追踪 Recall@K 和 Faithfulness 的变化趋势

### 6.2 安全与治理

| 关注点 | 措施 |

## 八、总结

### 核心认知

> **RAG 的质量瓶颈不在模型，而在检索管线。切片策略 + 向量数据库选择 + 相似性算法的组合，决定了检索精度的上限。LLM 只能在这个上限之下工作。**

### RAG 系统质量公式

```
RAG_Quality = Chunking(0.30) + Retrieval(0.35) + Generation(0.20) + Evaluation(0.15)

其中：
  Chunking    = 切片策略 × 参数调优 × 上下文富化
  Retrieval   = 向量DB选择 × 相似性算法 × 混合检索 × 重排序
  Generation  = Prompt工程 × LLM选择 × 上下文组装
  Evaluation  = 评估框架 × Golden Dataset × 持续监控
```

### 实战起步建议

1. **第一周**：递归切片(500Token/15%overlap) + Chroma + Cosine + GPT-4o → 跑通基础流程
2. **第二周**：加入混合检索(BM25+RRF) + BGE-Reranker → 检索质量跃升
3. **第一个月**：替换为 Qdrant/Milvus + RAGAS 评估 → 生产基线建立
4. **第二个月**：根据场景引入语义切片或 GraphRAG → 场景化深度优化
