### 1.2 Buffer Pool 内存结构

```
Buffer Pool (innodb_buffer_pool_size)
│
├── LRU List (old/new 分区, 3/8 比例)
│   ┌────────────────────────────────────────────────────┐
│   │  NEW Sublist (5/8)       │  OLD Sublist (3/8)     │
│   │  [热点数据页]   ← 提升 ←  │  [新读入的页, 1s窗口]  │
│   │  midpoint = LRU长度 × 5/8                          │
│   └────────────────────────────────────────────────────┘
│   首次读入: 插入 OLD 区头部 (innodb_old_blocks_time=1000ms 窗口保护)
│   再次访问: 如果在 OLD 区超过 1s → 提升到 NEW 区；否则留在 OLD 区
│   全表扫描保护: 大表扫描页停留在 OLD 区，不污染 NEW 区
│
├── Free List: 空闲页链表，分配新页时从此获取
│
├── Flush List: 脏页链表 (按 oldest_modification LSN 排序)
│   Checkpoint 时从此链表尾部刷盘
│
└── 关键配置:
    innodb_buffer_pool_size        = 系统内存的 50%-80%
    innodb_buffer_pool_instances   = 8-16 (减少并发竞争)
    innodb_old_blocks_pct          = 37 (OLD区占比)
    innodb_old_blocks_time         = 1000 (ms, 窗口保护)
```

### 🔥 高频面试题

**Q1: Buffer Pool 为什么使用 LRU 链表，且要分为 OLD/NEW 两个子链表？**

> A: 防止全表扫描污染缓冲池。全表扫描读入的大量页只会被访问一次，如果不分区，这些"一次性"页会挤掉真正热点数据。MySQL 采用 midpoint insertion strategy: 新读入的页插入 OLD 区头部(而非 NEW 区)，只有在 OLD 区停留超过 `innodb_old_blocks_time`(默认1s)后被再次访问才提升到 NEW 区。这样全表扫描的页刚进 OLD 区就被移出，不污染 NEW 区热点数据。

**Q2: Change Buffer 只适用于二级索引而不适用于聚簇索引，为什么？**

> A: 聚簇索引要求数据物理有序，且可能存在唯一性约束。写入聚簇索引必须立即检查唯一性，必须读取原始页。而二级非唯一索引不需要检查唯一性，可以先将变更缓存起来，稍后批量 merge。这样随机 IO 变成顺序 IO，大幅提升写入性能。

│                                                            │
│  BNL (Block Nested Loop, 8.0.20 后被 Hash Join 取代):     │
│    将驱动表分批读入 Join Buffer (join_buffer_size)         │

### 🔥 高频面试题

**Q1: 一条 SQL 在 MySQL 中的完整执行过程是怎样的？**

> A: 连接器认证 → (8.0前: 查询缓存) → 解析器生成 AST → 预处理器语义分析 → 优化器 CBO 生成执行计划(选择索引/Join 算法/子查询优化) → 执行器调用存储引擎 Handler API 逐行处理 → 返回结果集。UPDATE 额外经过 Undo Log 记录 → Buffer Pool 修改 → Redo Log 2PC 提交流程。

**Q2: 为什么 MySQL 8.0 移除了 Query Cache？**

> A: Query Cache 以整个 SQL 语句为 key，表数据更新后整个表所有关联缓存全部失效。在高并发写入场景下，QC 的锁竞争(所有查询/失效都需要全局锁)比它带来的性能提升更大，反而成为瓶颈。

## 3. 事务与 MVCC 原理

### 3.1 MVCC 三要素

```
┌──────────────────────────────────────────────────────────┐
│                   MVCC 多版本并发控制                      │
│                                                           │
│  三要素:                                                   │
│                                                           │
│  ① 隐藏列 (每行记录自带)                                   │
│  ┌──────────────────────────────────────────────────────┐ │
│  │  DB_ROW_ID    │  DB_TRX_ID    │  DB_ROLL_PTR         │ │
│  │  行ID(6字节)   │  事务ID(6字节) │  回滚指针(7字节)      │ │
│  │  (无主键时存在) │  (最后修改本行的事务ID) │ (指向Undo Log)│ │
│  └──────────────────────────────────────────────────────┘ │
│                                                           │
│  ② Undo Log 版本链                                        │
│  ┌──────────────────────────────────────────────────────┐ │
│  │ 当前行 (trx_id=100)                                   │ │
│  │   DB_ROLL_PTR ──────────────────────────┐             │ │
│  │                                          ▼             │ │
│  │                        Undo Log (trx_id=90)            │ │
│  │                           DB_ROLL_PTR ────────┐        │ │
│  │                                                ▼        │ │
│  │                                  Undo Log (trx_id=80)  │ │
│  │                                     DB_ROLL_PTR → ...  │ │
│  └──────────────────────────────────────────────────────┘ │
│                                                           │
│  ③ ReadView (快照读时生成)                                 │
│  ┌──────────────────────────────────────────────────────┐ │
│  │  m_ids[]      : 创建快照时 活跃(未提交)事务ID 列表      │ │
│  │  min_trx_id   : m_ids 中最小的 ID                     │ │
│  │  max_trx_id   : 下一个即将分配的事务ID                  │ │
│  │  creator_trx_id: 创建此 ReadView 的事务ID              │ │
│  └──────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────┘
```

### 3.2 ReadView 可见性判断算法

```
┌──────────────────────────────────────────────────────────┐
│              可见性判断 (trx_id 是行版本的事务ID)          │
│                                                           │
│  if (trx_id == creator_trx_id):                           │
│      → 可见 (自己修改的)                                  │
│                                                           │
│  if (trx_id < min_trx_id):                                │
│      → 可见 (修改此行的事务在快照前已提交)                  │
│                                                           │
│  if (trx_id >= max_trx_id):                               │
│      → 不可见 (修改此行的事务在快照后才开始)                │
│                                                           │
│  if (trx_id in m_ids[]):                                  │
│      → 不可见 (修改事务在快照时刻尚未提交)                  │
│      → 沿 DB_ROLL_PTR 链查找上一个版本                    │
│                                                           │
│  else:                                                    │
│      → 可见 (修改事务在快照时刻已提交)                     │
└──────────────────────────────────────────────────────────┘
```

### 3.3 RC vs RR — ReadView 生成时机差异

```
┌──────────────────────────────────────────────────────────────┐
│                                                               │
│  READ COMMITTED (RC): 每次 SELECT 都生成新的 ReadView         │
│  ─────────────────────────────────────────────────────────   │
│  SELECT 1 → ReadView_1 (看到已提交，包括事务中提交的其他事务)  │
│  SELECT 2 → ReadView_2 (可能看到不同数据 → 不可重复读)         │
│                                                               │
│  时间线示例:                                                   │
│  T1: BEGIN; SELECT count=10 (生成ReadView1, 看到10)           │
│  T2: UPDATE SET count=20; COMMIT;                             │
│  T1: SELECT count=20 (生成ReadView2, 看到20★ 不可重复读!)      │
│                                                               │
│  ─────────────────────────────────────────────────────────   │
│                                                               │
│  REPEATABLE READ (RR): 只在第一次 SELECT 时生成 ReadView       │
│  ─────────────────────────────────────────────────────────   │
│  SELECT 1 → ReadView (整个事务复用此视图)                      │
│  SELECT 2 → 复用同一 ReadView → 可重复读                       │
│                                                               │
│  时间线示例:                                                   │
│  T1: BEGIN; SELECT count=10 (生成ReadView, 复用)              │
│  T2: UPDATE SET count=20; COMMIT;                             │
│  T1: SELECT count=10 (使用原ReadView, 仍是10★ 可重复读!)       │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

### 3.5 Redo Log vs Undo Log 协作

```
┌──────────────────────────────────────────────────────────┐
│              Redo Log 与 Undo Log 分工                     │
│                                                           │
│  Redo Log (重做日志)                                       │
│  ├── 职责: Crash-Safe (崩溃恢复), 保证已提交事务不丢       │
│  ├── 记录: "对哪个页的哪个偏移量做了什么修改"              │
│  ├── 写入: 顺序写 (快!), WAL (Write Ahead Log)            │
│  ├── 大小: ib_logfile0 + ib_logfile1 (固定, 循环写)       │
│  └── Group Commit: 3 阶段 (flush → sync → commit)         │
│                                                            │
│  Undo Log (回滚日志)                                       │
│  ├── 职责: 事务回滚 + MVCC 多版本                         │
│  ├── 记录: "修改前的值" (insert undo / update undo)       │
│  ├── 写入: 随机写 (与数据页在一起)                         │
│  ├── 生命周期: 事务回滚时使用 → MVCC不再需要时 Purge       │
│  └── 长事务危害: Undo Log 无法 Purge → ibdata1 膨胀       │
│                                                            │
│  崩溃恢复协作:                                              │
│  启动 → 检查 Redo Log → 重放已提交事务(Redo)              │
│       → 回滚未提交事务(Undo)                               │
│       → Doublewrite 修复部分写坏页                         │
│       → 开放数据库                                         │
└──────────────────────────────────────────────────────────┘
```

### 🔥 高频面试题

**Q1: MySQL 的 MVCC 是如何实现的？与 PostgreSQL 的 MVCC 有什么区别？**

> A: MySQL InnoDB 通过隐藏列(DB_TRX_ID/DB_ROLL_PTR) + Undo Log 版本链 + ReadView 可见性判断实现 MVCC。Undo Log 存旧版本，新数据直接在原行更新。PostgreSQL 则是 append-only，UPDATE 产生新行，旧行标记为过期，通过 VACUUM 清理。MySQL 方式写入更集中但版本链追溯代价更大，PG 方式空间换时间但需要 VACUUM。

**Q2: 为什么要用 2PC (两阶段提交)？只用 Redo Log 不行吗？**

> A: 因为 MySQL 有两套日志系统：InnoDB 的 Redo Log 和 Server 层的 Binlog。如果不用 2PC：
> - 先写 Redo 再写 Binlog：如果写 Binlog 前崩溃，从库没收到此事务，但主库已提交→主从不一致
> - 先写 Binlog 再写 Redo：如果写 Redo 前崩溃，主库没此事务，但从库已收到→主从不一致
> 2PC 的 prepare-commit 确保了 Redo Log 和 Binlog 的原子性。

### 4.1 锁体系全景

│                    InnoDB 锁体系                               │

│  │                                                        │   │

│  │  Gap Lock:         锁定索引间隙 (防插入)                │   │
│  │  Next-Key Lock:    Record + Gap (前开后闭区间)         │   │
│  │  Insert Intention: 插入意向锁 (Gap锁的一种)            │   │
│  │  AUTO-INC Lock:    自增列锁 (轻量级互斥)               │   │
│  └───────────────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────────┘
```

### 4.2 Next-Key Lock 锁范围图解

```
假设索引值为: 5, 10, 15, 20, 25

┌──────────────────────────────────────────────────────────┐
│                                                           │
│  WHERE id = 15 FOR UPDATE (主键/唯一索引等值查询):        │
│       .....[ 10 ].....(gap).....[ 15 ].....(gap).....[20]│
│                                   ↑                       │
│                            只锁 Record Lock (15)          │
│       (精确命中 → 降级为 Record Lock)                     │
│                                                           │
│  WHERE id = 13 FOR UPDATE (主键等值但不存在):             │
│       .....[ 10 ].....(gap).....[ 15 ].....(gap).....[20]│
│                     ↑ Gap Lock (10,15)                    │
│       (不存在的值 → Gap Lock 防止插入)                    │
│                                                           │
│  WHERE age = 15 FOR UPDATE (非唯一索引等值查询):          │
│       .....[ 10 ].....(gap).....[ 15 ].....(gap).....[20]│
│              ↑ Next-Key (10,15] + Gap (15,20)             │
│       (非唯一索引 → 防止幻读, 锁当前值+下一个间隙)        │
│                                                           │
│  WHERE id > 15 AND id < 25 FOR UPDATE (范围查询):         │
│       .....[ 15 ].....(gap).....[ 20 ].....(gap).....[25]│
│                ↑ Next-Key (15,20] + Next-Key (20,25]      │
│       (范围查询 → 锁定每个匹配行+行间间隙)                │
│                                                           │
│  核心: 在 RR 下, Next-Key Lock 是默认加锁方式              │
│        RC 下可以不需要 Gap Lock (binlog_format=ROW)       │
└──────────────────────────────────────────────────────────┘
```

### 4.3 加锁行为矩阵

```
┌────────────────────────────────────────────────────────────────────┐
│                    SQL 加锁行为矩阵 (RR 隔离级别)                    │
│                                                                     │
│  条件类型        │ 示例                          │ 加锁行为          │
│  ───────────────┼────────────────────────────────┼───────────────── │
│  主键等值命中    │ WHERE pk=10 AND pk EXISTS      │ Record Lock(pk=10)│

│                 │                                │ + 主键 Record Lock│
│  唯一索引未命中  │ WHERE unique_key IS NULL       │ Gap Lock锁定间隙  │
│  非唯一索引等值  │ WHERE idx=15                   │ Next-Key Lock    │
│                 │                                │ (前开后闭)        │

│                 │                                │ 直到supremum     │

### 4.4 死锁检测 — wait-for-graph

```
┌──────────────────────────────────────────────────────────┐
│                   wait-for-graph 死锁检测                  │
│                                                           │
│  事务A: UPDATE t SET v=1 WHERE id=10;                     │
│         → 持有 id=10 的 X 锁                              │
│         → 等待 UPDATE t SET v=2 WHERE id=20; (等待id=20)  │
│                                                           │
│  事务B: UPDATE t SET v=3 WHERE id=20;                     │
│         → 持有 id=20 的 X 锁                              │
│         → 等待 UPDATE t SET v=4 WHERE id=10; (等待id=10)  │
│                                                           │
│  wait-for-graph:                                          │
│       TrxA ───等待───→ TrxB                               │
│         ↑                │                                │
│         └──────等待──────┘                                │
│        = 环路! → 死锁检测 → 回滚代价最小的事务            │
│                                                           │
│  死锁检测算法:                                             │
│  - 深度优先遍历 wait-for-graph                            │
│  - 检测到环 → 选择 undo 量最小的事务回滚                  │
│  - innodb_deadlock_detect = ON (默认)                     │
│  - 高并发下 O(n²) 可能成为瓶颈                            │
│  - 8.0.18+: innodb_deadlock_detect_delay (毫秒)           │
│                                                           │
│  降低死锁概率:                                             │
│  1. 固定加锁顺序 (按主键升序)                             │
│  2. 减少事务粒度, 缩短事务时间                            │
│  3. 合理使用索引, 避免全表扫描锁多行                      │
│  4. RC 隔离级别 + binlog_format=ROW (减少Gap Lock)        │
└──────────────────────────────────────────────────────────┘
```

### 🔥 高频面试题

**Q1: MySQL 的 Gap Lock 在什么情况下生效？RC 隔离级别有 Gap Lock 吗？**

> A: Gap Lock 仅在 RR 隔离级别下生效，用于解决幻读问题。RC 隔离级别默认只使用 Record Lock (等值命中) 或不加 Gap Lock。生产环境如果使用 RC + binlog_format=ROW，可以完全避免 Gap Lock，减少死锁。

**Q2: 为什么 RR 隔离级别可以解决幻读？**

> A: 通过 Next-Key Lock (Record + Gap Lock) 实现。例如 `SELECT * FROM t WHERE id BETWEEN 10 AND 20 FOR UPDATE`，不仅锁 id=10,20 的行，还锁住 (5,10], (10,15], (15,20], (20,25] 的所有间隙，使得其他事务无法在这些间隙插入新行，从而阻止幻读。MVCC 的 ReadView 快照也能解决"快照读"的幻读，但对于"当前读"(FOR UPDATE/LOCK IN SHARE MODE)，必须配合 Next-Key Lock。

### 5.4 索引下推 ICP (Index Condition Pushdown)

```

│                                                           │
│  ┌──────────── 有 ICP (5.6+) ──────┐                     │
│  │                                   │                    │
│  │  Storage Engine:                  │                    │
│  │  索引扫描 → 根据 name 前缀       │                    │

│  │        ↓ 少量 rows               │                    │
│  │  Server Layer:                    │                    │
│  │  直接返回结果                     │                    │
│  │                                    │                    │

```

### 🔥 高频面试题

**Q1: 为什么 InnoDB 使用 B+Tree 而不是 BST/AVL/红黑树?**

> A: ① B+Tree 是多路搜索树，每个节点可存储多个 key，树高度低(3-4 层即可存千万数据)，减少磁盘 IO；② 叶子节点形成有序双向链表，支持高效范围查询(BST 需要中序遍历)；③ 非叶子节点只存 key 不存数据，一个节点能存更多 key，进一步降低树高；④ 相比 B-Tree，B+Tree 的查询性能更稳定（所有数据访问路径等长）。

**Q2: 什么是索引下推(ICP)？为什么能提高性能？**

> A: ICP 将 Server 层的 WHERE 过滤条件下推到存储引擎层，在索引扫描阶段就过滤不符合条件的行。对于联合索引(a,b)，如果 WHERE a>10 AND b>5，传统做法是索引只过滤 a>10，返回所有行再由 Server 过滤 b>5；ICP 则在索引扫描时直接判断 b>5，减少回表和 Server 层处理的数据量。

### 🔥 高频面试题

**Q1: MySQL 半同步复制(Semi-Sync)的工作流程？和异步复制的区别？**

> A: 异步复制：Master 事务提交后立即返回客户端，不等 Slave。Semi-Sync：Master 事务提交后，至少等待一个 Slave 收到并写入 Relay Log (ACK 确认) 后才返回。5.7 引入 `rpl_semi_sync_master_wait_point=AFTER_SYNC` (推荐)，在 binlog 写完后 sync binlog 后等待 ACK，而非等整个 commit 完成，减少了延迟。损失: 如果 Slave 故障且没降级为异步，Master 写入会 hang。

**Q2: MGR 和传统主从复制有什么区别？**

> A: 传统主从是单向复制(Master→Slave)，故障切换需要外部工具(Orchestrator/MHA)。MGR 是无共享的集群，多节点互为副本，使用 Paxos 协议自动选主和故障切换(秒级)。核心优势：自动故障转移、多主写入(可选)、强一致性(流控)。代价：至少 3 节点，延迟比异步更高，适合核心高可用场景。

### 🔥 高频面试题

**Q1: 线上出现大量慢查询，你的排查思路是什么？**

> A: ① 确认范围：`SHOW FULL PROCESSLIST` + `information_schema.processlist` 找到正在执行的慢 SQL；② 查看执行计划：`EXPLAIN` / `EXPLAIN ANALYZE` (8.0.18+) 确认是否索引失效、全表扫描、文件排序；③ 锁等待分析：`sys.innodb_lock_waits` 查看锁等待链；④ 资源层面：磁盘 IO、CPU、内存使用情况；⑤ 定位到具体瓶颈后针对性优化：加索引、改写 SQL、调参数、扩容。

**Q2: 大表 DDL 在生产环境如何安全执行？**

> A: ① 使用 `pt-online-schema-change` (Percona Toolkit): 创建影子表→触发器同步增量→批量拷贝→原子切换(rename)；② MySQL 8.0 支持 `ALGORITHM=INSTANT`（仅改元数据）和 `ALGORITHM=INPLACE`（不改表结构，在线执行）；③ 避免直接 `ALGORITHM=COPY`（重建表，锁表）；④ 在从库先执行→主从切换→原主库执行；⑤ 选择低峰期，并设置 `lock_wait_timeout`。
