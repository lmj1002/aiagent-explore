```
│  │    ↓                                                      ││
│  │  分发写响应任务 ─────────────────┐                        ││
│  │    │                             │                        ││
│  │    │  ┌────────────┬─────────────┤                        ││
│  │    │  ▼            ▼             ▼                        ││
│  │    │ IO Thread0  IO Thread1  IO Thread2                   ││
│  │    │ (写响应)    (写响应)    (写响应)   ← 并行!            ││

│                                                               │
│  核心结论:                                                     │
│  - 读写 Socket 可以多线程 (IO 密集)                            │

```


### 🔥 高频面试题

**Q1: Redis 为什么单线程还这么快？6.0 多线程又做了什么？**

> A: 快的原因：① 基于内存操作（微秒级）；② IO 多路复用 epoll（非阻塞）；③ 精心设计的数据结构（SDS/skiplist/quicklist/dict）；④ 单线程避免锁竞争和上下文切换。6.0 多线程只用于 Socket IO 的读/写，命令执行仍然是单线程。因为瓶颈不是 CPU 而是网络 IO，多线程解除了这部分瓶颈。

**Q2: Redis 如何处理客户端命令的原子性？**

> A: 单线程事件循环保证。所有命令在同一个线程中按顺序执行，不会出现并发修改。这也是 Redis 不需要复杂锁机制的原因。注意：多个客户端命令之间是穿插执行的（非事务场景），但每个命令本身执行时不被打断。

### 2.3 SkipList (跳表) 结构

```
┌──────────────────────────────────────────────────────────────┐
│                   SkipList (跳表) 结构                         │
│                                                               │
│  Level 3:  [头] ──────────────────────→ [99] ─→ NULL         │
│  Level 2:  [头] ──────→ [30] ──────────→ [99] ─→ NULL        │
│  Level 1:  [头] → [10] → [30] → [50] → [99] → NULL          │
│  Level 0:  [头]→[5]→[10]→[25]→[30]→[50]→[75]→[99]→ NULL     │
│                                                               │
│  本质: 基于多层链表实现的概率平衡数据结构                       │
│                                                               │
│  插入流程:                                                     │
│  ┌──────────────────────────────────────────────────────┐    │
│  │ 1. 随机生成层高 (概率: 每次1/4, 最大64层)            │    │
│  │ 2. 从最高层开始查找插入位置, 记录每层的前驱节点(update[])│ │
│  │ 3. 从底层开始逐层插入新节点                           │    │
│  │ 4. 更新前驱节点的 forward 指针                        │    │
│  └──────────────────────────────────────────────────────┘    │
│                                                               │
│  查找流程:                                                     │
│  ┌──────────────────────────────────────────────────────┐    │
│  │ 从最高层开始 → 比较 forward->score                   │    │
│  │   ≤ 目标 → 向前移动                                   │    │
│  │   > 目标 → 下降到下一层                               │    │
│  │ 循环直到 Level 0 确定精确位置                         │    │
│  │ 复杂度: O(logN) 期望, O(N) 最坏                       │    │
│  └──────────────────────────────────────────────────────┘    │
│                                                               │
│  ZSet 双索引设计:                                              │
│  ┌──────────────────────────────────────────────────┐        │
│  │ skiplist: 按 score 排序, 支持范围查询 ZRANGE等    │        │
│  │   节点: {member, score, backward/forward/levels} │        │
│  │ dict: member → score 映射                        │        │
│  │   支持 ZSCORE member 的 O(1) 查找                │        │
│  │ 两个数据结构共享 member 指针, 不浪费内存          │        │
│  └──────────────────────────────────────────────────┘        │
└──────────────────────────────────────────────────────────────┘
```

### 2.4 渐进式 Rehash

```
┌──────────────────────────────────────────────────────────────┐
│                 Dict 渐进式 Rehash 流程                         │
│                                                               │
│  触发条件:                                                     │
│  - 扩容: used/size >= dict_can_resize (无子进程时=1, 否则=5)  │
│  - 缩容: used/size < HASHTABLE_MIN_FILL (10%)                  │
│                                                               │
│  ┌──────────────────────────────────────────────────────┐    │
│  │  dict:                                                │    │
│  │  ┌───────────────┐     ┌───────────────┐             │    │
│  │  │ ht[0] (旧表)   │     │ ht[1] (新表)   │             │    │
│  │  │ size=4        │ →→→ │ size=8        │             │    │
│  │  │ used=4        │     │ used=0        │             │    │
│  │  │ rehashidx=2   │     │               │             │    │
│  │  └───────────────┘     └───────────────┘             │    │
│  │   ↑ rehashidx 表示已迁移的桶索引                      │    │
│  │  [0] [1] 已完成   [2] 正在迁移   [3] 未迁移           │    │
│  └──────────────────────────────────────────────────────┘    │
│                                                               │
│  分配机制:                                                     │
│  ┌──────────────────────────────────────────────────────┐    │
│  │ 1. 每次 CRUD 操作时, 调用 dictRehash(d, 1) 迁移1个桶  │    │
│  │ 2. serverCron 中调用 dictRehashMilliseconds(1ms)     │    │
│  │    → 每次尽量迁移 100 个桶 (不超1ms)                  │    │
│  │ 3. 查询时先查 ht[0], 再查 ht[1] (都扫描)            │    │
│  │ 4. 新增只写 ht[1]                                    │    │
│  │ 5. rehashidx == -1 表示 rehash 完成                  │    │
│  │ 6. 完成后 free ht[0], ht[1] 变成 ht[0]              │    │
│  └──────────────────────────────────────────────────────┘    │
│                                                               │
│  优点: 避免一次性大量 CPU 占用导致服务卡顿                       │
└──────────────────────────────────────────────────────────────┘
```

### 🔥 高频面试题

**Q1: Redis ZSet 为什么使用跳表而不是红黑树/B+Tree？**

> A: ① 跳表实现简单，代码量远少于红黑树；② 范围查询(ZRANGE)跳表只需 O(logN+M)，跟平衡树相当；③ 跳表不存在树旋转导致的锁粒度问题（虽然 Redis 单线程没这问题）；④ 跳表天然支持按 score 顺序遍历，B+Tree 只对磁盘友好，对内存操作优势不大。

**Q2: Redis 的渐进式 Rehash 解决了什么问题？**

> A: 如果一次性将 ht[0] 全部迁移到 ht[1]，当 Hash 有百万级 key 时会导致长达数百毫秒甚至秒级的卡顿。渐进式 Rehash 将迁移工作分散到每次操作和 serverCron 定时任务中，每次只迁移少量数据，保证不阻塞正常命令处理。

## 3. 持久化深度

### 3.1 RDB (Redis Database) 快照

```
┌──────────────────────────────────────────────────────────────┐
│                    RDB 持久化流程                               │
│                                                               │
│  save (主进程, 阻塞):                                          │
│  ┌──────────────────────────────────────────────────────┐    │
│  │ Redis 主进程 → 直接执行 rdbSave()                     │    │
│  │ → 遍历所有 DB → 序列化到 dump.rdb                    │    │
│  │ → 期间所有客户端命令被阻塞!                           │    │
│  │ → 不推荐生产使用                                     │    │
│  └──────────────────────────────────────────────────────┘    │
│                                                               │
│  bgsave (fork 子进程, 非阻塞):                                 │
│  ┌──────────────────────────────────────────────────────┐    │
│  │                                                       │    │
│  │  父进程 (Redis Server)         子进程 (rdbSave)      │    │
│  │       │                              │                │    │
│  │  fork() ────────────────→   fork() 返回 0            │    │
│  │       │                              │                │    │
│  │  继续处理命令             遍历 DB 的所有数据          │    │
│  │  (利用 COW, 父进程        序列化写入 dump.rdb         │    │
│  │   修改页时复制新页)                 │                 │    │
│  │       │                    写入完成, exit()           │    │
│  │  收到 SIGCHLD ← ───────────────────┘                 │    │
│  │  更新 dirty counter                                  │    │
│  │                                                       │    │
│  │  COW (Copy On Write) 原理:                            │    │
│  │  父进程内存100MB → fork() → 父子共享内存页            │    │
│  │  父进程写入某页 → MMU 触发 page fault                │    │
│  │  → OS 复制该页 → 父进程修改副本                       │    │
│  │  → fork 时内存使用 ≈ 当前数据量的 10%-30%            │    │
│  │  → 必须预留足够内存!                                │    │
│  └──────────────────────────────────────────────────────┘    │
│                                                               │
│  自动触发条件 (save配置):                                      │
│  save 900 1   ← 900秒内至少1次修改                            │
│  save 300 10  ← 300秒内至少10次修改                            │
│  save 60 10000 ← 60秒内至少10000次修改                         │
└──────────────────────────────────────────────────────────────┘
```

### 3.2 AOF (Append Only File)

```
┌──────────────────────────────────────────────────────────────┐
│                    AOF 持久化全链路                             │
│                                                               │
│  写入流程:                                                     │
│  ┌──────────────────────────────────────────────────────┐    │
│  │                                                       │    │
│  │  Client 执行 SET key value                            │    │
│  │       │                                               │    │
│  │       ▼                                               │    │
│  │  ① 执行命令 (修改内存数据)                            │    │
│  │       │                                               │    │
│  │       ▼                                               │    │
│  │  ② 追加命令到 AOF 缓冲区 (server.aof_buf)             │    │
│  │     "*3\r\n$3\r\nSET\r\n$3\r\nkey\r\n$5\r\nvalue\r\n" │    │
│  │       │                                               │    │
│  │       ▼  (事件循环 beforesleep/每次循环)               │    │
│  │  ③ 将 AOF 缓冲写入 OS 内核缓冲区 (write)              │    │
│  │       │                                               │    │
│  │       ▼  (根据 appendfsync 策略)                       │    │
│  │  ④ fsync() 强制刷盘                                   │    │
│  │       │                                               │    │
│  │       ▼                                               │    │
│  │  AOF 文件持续增长 → 触发 AOF 重写                      │    │
│  └──────────────────────────────────────────────────────┘    │
│                                                               │
│  appendfsync 策略对比:                                         │
│  ┌──────────┬──────────────────┬─────────────┬─────────────┐│
│  │ 策略     │ 行为             │ 数据安全    │ 性能        ││
│  ├──────────┼──────────────────┼─────────────┼─────────────┤│
│  │ always   │ 每条命令后 fsync │ 最高(不丢)  │ 最低(QPS 100)││
│  │ everysec │ 每秒 fsync 一次  │ 最多丢1秒   │ 中间(推荐)  ││
│  │ no       │ 交给OS决定       │ 不可控      │ 最高        ││
│  └──────────┴──────────────────┴─────────────┴─────────────┘│
│                                                               │
│  AOF 重写 (BGREWRITEAOF, fork子进程):                          │
│  ┌──────────────────────────────────────────────────────┐    │
│  │ 目的: 压缩 AOF (合并重复操作, 去过期键等)             │    │
│  │                                                       │    │
│  │ 父进程                    子进程                      │    │
│  │ fork() ──────────────→ 读取内存数据                   │    │
│  │ │                      生成最小命令集                  │    │
│  │ │                      写入 temp-rewriteaof-bg-xxx.aof│    │
│  │ │ 同时: 父进程将新命令   │                            │    │
│  │ │       追加到 aof_buf   完成写入 → exit()            │    │
│  │ │       也追加到 aof_rewrite_buf ←──┘                │    │
│  │ │                                                    │    │
│  │ 收到 SIGCHLD:                                        │    │
│  │   1. 将 aof_rewrite_buf 追加到新 AOF 文件            │    │
│  │   2. rename 新AOF → 旧AOF  (原子替换)                │    │
│  │   3. 完成!                                           │    │
│  └──────────────────────────────────────────────────────┘    │
└──────────────────────────────────────────────────────────────┘
```

### 🔥 高频面试题

**Q1: Redis RDB fork 子进程时系统需要多少额外内存？为什么？**

> A: Redis 手册建议预留当前使用内存的 50%-100%。因为 fork 基于 COW，如果父进程在子进程写 RDB 期间有大量写入，频繁触发 COW 复制页，极端情况可能导致内存翻倍。生产经验：监控 `used_memory` 和 `used_memory_rss`，确保系统可用内存 > 2× used_memory。

**Q2: AOF 重写期间，新的写入命令如何保证不丢失？**

> A: 父进程同时将新命令写入两个缓冲区：`aof_buf`(正常 AOF 缓冲，按 fsync 策略刷盘) 和 `aof_rewrite_buf`(重写缓冲)。子进程完成后，父进程将 `aof_rewrite_buf` 中积累的增量追加到子进程生成的新 AOF 文件末尾，然后 rename 原子替换旧 AOF，确保不丢任何命令。

## 4. 主从复制全链路

### 4.1 全量复制 (Full Resynchronization)

```
┌──────────────────────────────────────────────────────────────┐
│                    全量复制时序图                               │
│                                                               │
│  Slave                                        Master          │
│    │                                            │             │
│    │ ──── PSYNC ? -1 ────────────────────────→ │             │
│    │     (首次连接, replid=?, offset=-1)        │             │
│    │                                            │             │
│    │ ←─── +FULLRESYNC <replid> <offset> ─────── │             │
│    │     (告诉Slave: 全量复制)                   │             │
│    │                                            │             │
│    │                                    Master fork()         │
│    │                                    开始 BGSAVE          │
│    │                                            │             │
│    │                              Master 将所有新写入命令     │
│    │                              累积到 replication buffer   │
│    │                                            │             │
│    │ ←─── 传输 RDB 文件 ─────────────────────── │             │
│    │     (Slave 接收并加载到内存)                │             │
│    │                                            │             │
│    │  Slave 清空旧数据, 加载 RDB                │             │
│    │                                            │             │
│    │ ←─── 传输累积在 buffer 中的增量命令 ─────── │             │
│    │     (Slave 执行增量, 追平数据)              │             │
│    │                                            │             │
│    │  数据同步完成, 进入增量复制阶段              │             │
│    │  Slave 持续接收 Master 的写命令流           │             │
│                                                               │
│  replication buffer 大小:                                      │
│  - 默认 client-output-buffer-limit slave 256mb 64mb 60        │
│  - 如果 RDB 传输慢 + 写入量大, buffer 可能溢出                  │
│  - 溢出后复制断开, Slave 重新全量同步 (恶性循环)               │
│  - 解决方案: repl-backlog-size 调大 + repl-diskless-sync      │
└──────────────────────────────────────────────────────────────┘
```

### 4.2 增量复制 (Partial Resynchronization) — PSYNC

```
┌──────────────────────────────────────────────────────────────┐
│                  增量复制 (PSYNC) 原理                          │
│                                                               │
│  三要素:                                                       │
│  ┌──────────────────────────────────────────────────────┐    │
│  │ 1. Replication ID (replid): 标识 Master 身份          │    │
│  │    Master 每次重启/晋升会生成新 replid                │    │
│  │ 2. Replication Offset: Master/Slave 各自的复制偏移量  │    │
│  │    Master: master_repl_offset                        │    │
│  │    Slave:  slave_repl_offset                         │    │
│  │ 3. Replication Backlog: Master 端的环形缓冲区         │    │
│  │    大小: repl-backlog-size (默认 1MB, 建议 64MB+)     │    │
│  └──────────────────────────────────────────────────────┘    │
│                                                               │
│  ┌──────────────────────────────────────────────────────┐    │
│  │           Replication Backlog (环形缓冲区)            │    │
│  │                                                       │    │
│  │  offset:  ... 100 101 102 ... 200 201 ...             │    │
│  │            ┌───┬───┬───┬───┬───┬───┬───┬───┐        │    │
│  │            │ A │ B │ C │ D │   │   │   │   │        │    │
│  │            └───┴───┴───┴───┴───┴───┴───┴───┘        │    │
│  │             ↑repl_backlog_off         ↑master_repl_off│   │
│  │                                                       │    │
│  │  Slave 重连: "我的 offset=99, replid=xxx"              │    │
│  │  Master: 检查 offset 是否在 backlog 范围内:            │    │
│  │    99 >= repl_backlog_off (105) → 否! 已覆盖 → 全量复制│   │
│  │    150 >= repl_backlog_off (105) → 是! → 增量复制      │    │
│  └──────────────────────────────────────────────────────┘    │
│                                                               │
│  PSYNC 成功条件:                                               │
│  1. Slave 的 replid 与 Master 匹配                             │
│  2. Slave 的 offset 在 Master 的 backlog 范围内                │
│                                                               │
│  backlog 大小估算:                                              │
│  backlog_size = master_write_QPS × 每条命令大小 × 断线最长秒数 │
│  例: 10K QPS × 256 bytes × 60s ≈ 150MB                        │
└──────────────────────────────────────────────────────────────┘
```

### 🔥 高频面试题

**Q1: Redis 全量复制过程中，如果 Master 写入量很大，会出现什么问题？**

> A: ① `client-output-buffer-limit` 溢出导致复制中断→重连→全量复制→再溢出，恶性循环；② fork BGSAVE 期间内存因 COW 暴增，可能 OOM；③ 多个 Slave 同时全量同步导致 Master IO 打满。解决方案：调大 buffer 限制、使用 diskless sync、错峰 Slave 同步、使用 `wait` 命令控制同步节奏。

## 5. Sentinel 故障转移

### 5.1 Sentinel 架构

```
┌──────────────────────────────────────────────────────────────┐
│                   Sentinel 集群架构                            │
│                                                               │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐                 │
│  │Sentinel 1│   │Sentinel 2│   │Sentinel 3│  (≥3, 奇数)     │
│  │ :26379   │   │ :26380   │   │ :26381   │                 │
│  └────┬─────┘   └────┬─────┘   └────┬─────┘                 │
│       │              │              │                         │
│       └──────────────┼──────────────┘                         │
│                      │ 各自直连监控                            │
│         ┌────────────┼────────────┐                           │
│         ▼            ▼            ▼                           │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐                   │
│  │  Master  │  │ Slave 1  │  │ Slave 2  │                   │
│  │ :6379    │←─│ :6380    │←─│ :6381    │                   │
│  └──────────┘  └──────────┘  └──────────┘                   │
│                                                               │
│  Sentinel 四重身份 (功能分离):                                  │
│  ① 监测(Monitoring): 定期 PING → 判断节点存活                  │
│  ② 通知(Notification): 通知管理员/其他程序事件                 │
│  ③ 自动故障转移(Automatic Failover)                           │
│  ④ 配置提供者(Configuration Provider): 客户端获取Master地址   │
└──────────────────────────────────────────────────────────────┘
```

### 5.2 故障判定流程 (SDOWN → ODOWN)

```
┌──────────────────────────────────────────────────────────────┐
│               SDOWN → ODOWN 判定流程                           │
│                                                               │
│  SDOWN (Subjective Down, 主观下线):                            │
│  ┌──────────────────────────────────────────────────────┐    │
│  │ Sentinel 每秒 PING Master                             │    │
│  │   如超过 down-after-milliseconds 无响应               │    │
│  │    → 该 Sentinel 判定为 SDOWN                         │    │
│  │    → 此时不发起故障转移!                              │    │
│  └──────────────────────────────────────────────────────┘    │
│                                                               │
│  ODOWN (Objective Down, 客观下线):                             │
│  ┌──────────────────────────────────────────────────────┐    │
│  │ SDOWN的Sentinel → 向其他Sentinel发送                 │    │
│  │   SENTINEL is-master-down-by-addr <ip> <port>        │    │
│  │   询问: "你们也认为Master挂了吗?"                     │    │
│  │                                                       │    │
│  │ 收集投票 → ≥ quorum 个 Sentinel 认为 SDOWN            │    │
│  │    → 判定为 ODOWN → 启动故障转移!                     │    │
│  └──────────────────────────────────────────────────────┘    │
│                                                               │
│  quorum 参数的意义:                                             │
│  - quorum=2: 至少2个Sentinel同意才判定ODOWN和发起故障转移        │
│  - 不能简单设置为多数 (如5个Sentinel设quorum=3)                 │
│  - quorum 决定了故障转移的敏感度 (太小→误判, 太大→延迟)        │
└──────────────────────────────────────────────────────────────┘
```

### 5.3 故障转移完整流程

```
┌──────────────────────────────────────────────────────────────┐
│                  故障转移完整时序                               │
│                                                               │
│  ① ODOWN → Leader Sentinel 选举                               │
│  ┌──────────────────────────────────────────────────────┐    │
│  │ 最先发现 ODOWN 的 Sentinel 向其他 Sentinel 发送       │    │
│  │ SENTINEL is-master-down-by-addr (带 runid + epoch)   │    │
│  │ → Raft 式投票选出 Leader (Epoch 最大者)              │    │
│  └──────────────────────────────────────────────────────┘    │
│           │                                                    │
│           ▼                                                    │
│  ② Leader Sentinel 选择新 Master                               │
│  ┌──────────────────────────────────────────────────────┐    │
│  │ 过滤: 排除 SDOWN/ODOWN/超时断连的 Slave              │    │
│  │ 排序 (优先级从高到低):                                │    │
│  │   1. slave-priority (值越小越优先)                   │    │
│  │   2. 复制偏移量 (offset 越大→数据越新)               │    │
│  │   3. runid (字典序, 保底)                             │    │
│  └──────────────────────────────────────────────────────┘    │
│           │                                                    │
│           ▼                                                    │
│  ③ 晋升 Slave 为 Master                                       │
│  ┌──────────────────────────────────────────────────────┐    │
│  │ SLAVEOF NO ONE → 断开旧Master→变为独立Master         │    │
│  └──────────────────────────────────────────────────────┘    │
│           │                                                    │
│           ▼                                                    │
│  ④ 重新配置其他 Slave                                         │
│  ┌──────────────────────────────────────────────────────┐    │
│  │ SLAVEOF <新Master IP> <新Master Port>                │    │
│  │ → 开始从新 Master 复制                                │    │
│  └──────────────────────────────────────────────────────┘    │
│           │                                                    │
│           ▼                                                    │
│  ⑤ 更新 Sentinel 配置 + 通知客户端                            │
│  ┌──────────────────────────────────────────────────────┐    │
│  │ 修改 sentinel.conf 中的 Master 地址                   │    │
│  │ 发布 +switch-master 频道消息                          │    │
│  │ 客户端订阅获得通知 → 更新连接                          │    │
│  └──────────────────────────────────────────────────────┘    │
│                                                               │
│  旧 Master 恢复后:                                              │
│  → 被 Sentinel 配置为 Slave → SLAVEOF 新 Master               │
└──────────────────────────────────────────────────────────────┘
```

### 🔥 高频面试题

**Q1: Sentinel 选举 Leader 的过程是怎样的？**

> A: 采用类 Raft 协议的投票机制。每个 Sentinel 有 epoch(纪元)和 runid。发起故障转移的 Sentinel 递增 epoch，向其他 Sentinel 请求投票(包含自己的 runid 和 epoch)，先到先得。得票 ≥ max(quorum, majority) 的 Sentinel 成为 Leader，负责执行故障转移。如果超时未选出，epoch 递增重新选举。

**Q2: Sentinel 集群为什么至少需要 3 个节点？**

> A: ① 防止脑裂：2 个 Sentinel 挂掉一个就只剩一个，无法区分"Master 挂了"还是"Sentinel 本身网络问题"；② 多数派原则：3 个 Sentinel 可以容忍 1 个故障，5 个容忍 2 个；③ quorum 设置灵活性：3 节点设 quorum=2，既保证判定可靠性又不过度敏感。

## 6. Cluster 集群分片

### 6.1 数据分布模型

```
│  Hash Tag (强制同一slot):                                      │
│  {user:1001}:name  → CRC16("user:1001")                       │
│  {user:1001}:age   → CRC16("user:1001")  ← 同一slot!          │
│  只有 {} 之间的部分参与 CRC16 计算                             │
└──────────────────────────────────────────────────────────────┘
```

### 6.2 MOVED 与 ASK 重定向

```
┌──────────────────────────────────────────────────────────────┐
│              MOVED (永久) vs ASK (临时) 重定向                  │
│                                                               │
│  MOVED 重定向:                                                 │
│  ┌──────────────────────────────────────────────────────┐    │
│  │ Client → Node A: "GET key1"                          │    │
│  │ Node A: 计算 slot → 不属于我 → 返回                   │    │
│  │   -MOVED 3999 192.168.1.2:6379                       │    │
│  │ Client: 更新 slot 路由表                              │    │
│  │ Client → Node B (192.168.1.2:6379): "GET key1"       │    │
│  │ Node B: 返回 value                                    │    │
│  │                                                       │    │
│  │ 特点: slot 已永久迁移到 Node B, 客户端应更新本地路由表 │    │
│  └──────────────────────────────────────────────────────┘    │
│                                                               │
│  ASK 重定向 (slot 正在迁移中):                                  │
│  ┌──────────────────────────────────────────────────────┐    │
│  │ slot 3999 正在从 Node A → Node B 迁移                 │    │
│  │ Client → Node A: "GET key1"                          │    │
│  │ Node A: key1 已经被迁移到 Node B                       │    │
│  │   -ASK 3999 192.168.1.2:6379                         │    │
│  │ Client: 先向 Node B 发送 ASKING                       │    │
│  │ Client → Node B: "ASKING" + "GET key1"               │    │
│  │ Node B: 返回 value                                    │    │
│  │                                                       │    │
│  │ 特点: 只是临时的, slot 迁移完后不再需要 ASK           │    │
│  │ 客户端不应更新路由表! (因为迁移尚未完成)              │    │
│  └──────────────────────────────────────────────────────┘    │
└──────────────────────────────────────────────────────────────┘
```

### 6.3 Gossip 协议与故障检测

```
┌──────────────────────────────────────────────────────────────┐
│                Gossip 协议 节点通信                              │
│                                                               │
│  四种消息类型:                                                  │
│  ┌──────────────────────────────────────────────────────┐    │
│  │ MEET:  邀请新节点加入集群                              │    │
│  │ PING:  周期性发送 (每秒选几个随机节点)                │    │
│  │ PONG:  响应 PING/MEET, 携带自身信息                  │    │
│  │ FAIL:  广播某个节点故障 (由 PFAIL 升级而来)           │    │
│  └──────────────────────────────────────────────────────┘    │
│                                                               │
│  PFAIL (Possible Failure) → FAIL (确认故障):                   │
│  ┌──────────────────────────────────────────────────────┐    │
│  │ 1. Node A PING Node B → 超时无响应                   │    │
│  │    → Node A 标记 Node B 为 PFAIL (疑似下线)           │    │
│  │ 2. Node A 与其他节点 Gossip 时带上 Node B 的 PFAIL    │    │
│  │ 3. 当 cluster-node-timeout/2 内, 超过半数 Master     │    │
│  │    都认为 Node B PFAIL                                 │    │
│  │    → Node A 广播 FAIL 消息 → 全集群标记 Node B 为 FAIL│    │
│  │ 4. 如果 Node B 是 Master → 触发 Slave 升主(故障转移) │    │
│  └──────────────────────────────────────────────────────┘    │
│                                                               │
│  cluster-node-timeout 作用:                                    │
│  - 超时判定基准: Master PING 超时 → PFAIL                      │
│  - 故障转移触发时间: 2 × cluster-node-timeout                  │
│  - 默认 15000ms, 不建议调太小 (误判风险)                       │
└──────────────────────────────────────────────────────────────┘
```

### 6.4 Slot 迁移流程

```
┌──────────────────────────────────────────────────────────────┐
│              Slot 迁移 (Resharding) 状态机                      │
│                                                               │
│  将 slot 3999 从 Node A → Node B:                              │
│                                                               │
│  ① 目标节点 (Node B) 状态:                                     │
│     CLUSTER SETSLOT 3999 IMPORTING <Node A ID>                │
│     → Node B 进入 IMPORTING 状态, 接收 slot 3999 的查询        │
│                                                               │
│  ② 源节点 (Node A) 状态:                                       │
│     CLUSTER SETSLOT 3999 MIGRATING <Node B ID>                │
│     → Node A 进入 MIGRATING 状态                               │
│     → 查询: key 在 → 正常返回; key 不在 → ASK 重定向           │
│                                                               │
│  ③ 逐个迁移 Key:                                               │
│     CLUSTER GETKEYSINSLOT 3999 <count>                        │
│     → 批量获取 slot 3999 中的 N 个 key                        │
│     MIGRATE <host> <port> <key> <db> <timeout>                │
│     → 原子操作: Node A dump key→RESTORE to Node B→Node A DEL │
│                                                               │
│  ④ 完成通知:                                                   │
│     CLUSTER SETSLOT 3999 NODE <Node B ID>                     │
│     → 所有节点更新 slot 归属: Node B                           │
└──────────────────────────────────────────────────────────────┘
```

### 🔥 高频面试题

**Q1: Redis Cluster 如何保证高可用？Master 宕机后如何恢复？**

> A: ① 每个 Master 至少配 1 个 Slave；② Master 被标记 FAIL 后，其 Slave 通过 Raft 协议选举新 Master（要求 N/2+1 个 Master 同意）；③ 新 Master 接管 slot → 广播配置变更；④ Gossip 协议确保所有节点更新路由表。关键参数 `cluster-node-timeout` 控制故障检测和转移速度。

**Q2: Redis Cluster 的 Gossip 协议为什么用 16384 个 slot 而不是更多？**

> A: ① 16384 可以确保每个节点的心跳消息中携带所有 slot 位图（16384/8/1024 = 2KB），如果扩容到 65536 就是 8KB；② 2KB 的心跳包在局域网中可以接受；③ 集群规模通常在 1000 节点以内，16384 slot 足够均匀分布；④ CRC16 产生 16 位哈希值，取模 16384 保持了较好的分布特性。

## 7. 内存管理与淘汰

### 7.1 过期删除策略

```
┌──────────────────────────────────────────────────────────────┐
│                   过期删除策略                                  │
│                                                               │
│  惰性删除 (Lazy Expiration):                                   │
│  ┌──────────────────────────────────────────────────────┐    │
│  │ 每次访问 key 时检查是否过期:                           │    │
│  │   expireIfNeeded() → 过期 → 删除key → 返回空          │    │
│  │ 优点: 对 CPU 友好                                      │    │
│  │ 缺点: 过期 key 如果一直不被访问 → 内存泄漏!           │    │
│  └──────────────────────────────────────────────────────┘    │
│                                                               │
│  定期删除 (Active Expiration):                                 │
│  ┌──────────────────────────────────────────────────────┐    │
│  │ serverCron → activeExpireCycle()                     │    │
│  │   每次随机抽取 20 个带过期时间的 key                   │    │
│  │   删除其中过期的                                       │    │
│  │   如果过期比例 > 25% → 继续处理这个DB                 │    │
│  │   (避免大量过期key堆积)                               │    │
│  │   限制: 每次执行 ≤ 25ms (ACTIVE_EXPIRE_CYCLE_SLOW)   │    │
│  │ 优点: 集中清理, 防止内存泄漏                          │    │
│  │ 缺点: 过期集中时 CPU 负载增加                          │    │
│  └──────────────────────────────────────────────────────┘    │
│                                                               │
│  两者结合: 惰性删除保证访问效率 + 定期删除防止内存泄漏            │
└──────────────────────────────────────────────────────────────┘
```

### 7.2 8 种内存淘汰策略

```
┌──────────────────────────────────────────────────────────────┐
│                   内存淘汰策略全景                               │
│                                                               │
│  maxmemory 配置触发:                                           │
│                                                               │
│  ┌────────────────────────────────────────────────────────┐  │
│  │ 不淘汰:                                                 │  │
│  │   noeviction: 拒绝写入(读可继续), 返回OOM error         │  │
│  │                                                         │  │
│  │ 全键空间 (allkeys):                                      │  │
│  │   allkeys-lru:   近似LRU, 淘汰最近最少使用的key          │  │
│  │   allkeys-lfu:   近似LFU, 淘汰最不频繁使用的key (4.0+)  │  │
│  │   allkeys-random: 随机淘汰                               │  │
│  │                                                         │  │
│  │ 有过期时间的键 (volatile):                               │  │
│  │   volatile-lru:    近似LRU, 只淘汰有过期时间的key        │  │
│  │   volatile-lfu:    近似LFU, 只淘汰有过期时间的key (4.0+)│  │
│  │   volatile-random: 随机淘汰有过期时间的key               │  │
│  │   volatile-ttl:    淘汰剩余TTL最小的key                  │  │
│  └─────────────────────────────────────────────────────────┘  │
│                                                               │
│  近似 LRU 算法:                                                 │
│  ┌──────────────────────────────────────────────────────┐    │
│  │ 1. 随机采样 N 个 key (maxmemory-samples, 默认5)       │    │
│  │ 2. 从采样中选择 idle time 最大的 (最久未被访问)        │    │
│  │ 3. 淘汰该 key → 释放内存                               │    │
│  │ 4. 如果仍不够 → 继续采样淘汰                           │    │
│  │                                                       │    │
│  │ 采样池淘汰 (3.0+ 优化):                               │    │
│  │  维护一个淘汰候选池(默认16个key)                       │    │
│  │  每次采样放入池中 → 选择池中最久未访问的key 淘汰       │    │
│  │  → 准确度接近真实 LRU                                  │    │
│  │                                                       │    │
│  │ maxmemory-samples: 越大越接近真实 LRU (但更耗CPU)      │    │
│  │ 默认5 → 准确度足够, 生产可设为10                       │    │
│  └──────────────────────────────────────────────────────┘    │
│                                                               │
│  LFU (4.0+, Least Frequently Used):                            │
│  ┌──────────────────────────────────────────────────────┐    │
│  │ 每个 key 有一个访问计数器 (Morris Counter 对数计数)    │    │
│  │ 随时间衰减: lfu-decay-time (默认1, 分钟)              │    │
│  │ 对数增长: lfu-log-factor (默认10, 越大越慢增长)       │    │
│  │ 淘汰时选择计数器最小的 key                             │    │
│  └──────────────────────────────────────────────────────┘    │
└──────────────────────────────────────────────────────────────┘
```

### 🔥 高频面试题

**Q1: Redis 的 LRU 和真正 LRU 的区别？能实现真正的 LRU 吗？**

> A: Redis 的 LRU 是近似的，通过随机采样 N(maxmemory-samples)个 key + 淘汰池实现，准确度约 90%+。真正的 LRU 需要维护一个全局双向链表，每次访问都更新链表中 key 的位置，内存开销和 CPU 开销都很大。Redis 选择近似 LRU 作为性能与准确度的折中。4.0+ 推荐使用 LFU 策略，对于热点数据场景更准确。

**Q2: 当 Redis 内存满了怎么办？noeviction 和 allkeys-lru 选哪个？**

> A: 取决于业务场景。如果 Redis 作为缓存，选择 `allkeys-lru` 或 `allkeys-lfu`，自动淘汰冷数据，保证缓存命中率；如果 Redis 作为数据存储(如 Session)，选择 `noeviction`，拒绝新写入但保留已有数据，避免数据丢失。生产常见：缓存用 allkeys-lru，存储/队列用 noeviction + 监控告警。
