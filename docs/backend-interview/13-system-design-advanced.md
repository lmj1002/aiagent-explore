## 1. 系统设计总纲 —— 方法论与答题框架

### 1.1 系统设计面试的本质

系统设计面试考察的不是"你是否做过某个系统"，而是考察以下能力维度：

| 能力维度 | 考察点 | 面试官关注信号 |
|---------|--------|--------------|
| **需求澄清** | 能否主动澄清模糊需求 | 是否直接动手画图，没有确认范围 |
| **架构决策** | 在多个方案中做 trade-off | 是否只给出一个方案而不讨论替代方案 |
| **扩展性思维** | 能否支持从 QPS 1K 到 1M 的演进 | 是否只考虑单机方案 |
| **全局视角** | 是否考虑监控、容灾、成本 | 是否只关注功能实现 |
| **沟通表达** | 能否 5 分钟把架构讲清楚 | 是否结构混乱、跳步严重 |

### 1.2 标准答题框架 —— 六步法

面试中推荐按以下六个阶段递进展开，每阶段分配合理的时间：

```
阶段 1: 需求澄清 (3-5 min)     → 明确功能需求 + 非功能需求
阶段 2: 容量预估 (3-5 min)     → QPS / 存储 / 带宽 / 内存
阶段 3: 高层设计 (5-8 min)     → 画出核心架构图 + 数据流
阶段 4: 组件选型 (3 min)       → 数据库 / 缓存 / MQ / 负载均衡
阶段 5: 详细设计 (10-15 min)   → 深入核心链路的数据结构和算法
阶段 6: 扩展思考 (3-5 min)     → 容灾 / 监控 / 升级 / Trade-off
```

#### 1.2.1 阶段 1：需求澄清 (FR / NFR)

在动手画图之前，必须与面试官确认需求边界。

**功能需求 (Functional Requirements)：**

- 系统需要支持哪些核心功能？
- 哪些是 MVP 必须覆盖的？哪些可以后续迭代？
- 用户的使用场景是什么？（C端还是B端？国内还是海外？）

**非功能需求 (Non-Functional Requirements)：**

| 维度 | 问题 | 典型指标 |
|------|------|---------|
| **可用性** | SLA 要求是多少？允许停机时间？ | 99.9% ~ 99.999% |
| **延迟** | 接口平均延迟 / P99 要求？ | API < 200ms, 页面 < 2s |
| **一致性** | 强一致性还是最终一致性？ | 取决于业务场景 |
| **耐久性** | 数据允许丢失吗？ | 0 丢失 vs. 容忍秒级丢失 |
| **规模** | DAU / MAU？读写比例？ | 1M DAU / 10M DAU |
| **合规** | GDPR / 数据本地化要求？ | 数据不能出境 |

#### 1.2.2 阶段 2：容量预估

**QPS 预估路线图：**

```
DAU × 平均每天请求次数 ÷ 86400 ≈ 平均 QPS
平均 QPS × 峰值/平均比（通常 5-10 倍）≈ 峰值 QPS
```

经验参考值：

| 指标 | 参考基准 | 对应架构水平 |
|------|---------|-------------|
| 10 QPS | 单人博客 | 单机即可 |
| 1000 QPS | 中型电商 | 单机 + 缓存 |
| 10K QPS | 大型应用 | 集群 + Redis + DB 分片 |
| 100K QPS | 头部产品 | 多活 + 各层全面优化 |
| 1M+ QPS | 国民级应用 | 全球多活 + 自研基础设施 |

**存储预估：**

```
每条数据大小 × 每天写入量 × 存储天数 × 副本因子
```

**带宽预估：**

```
平均单次请求响应大小 × 峰值 QPS × 8 (bit 换算)
```

常见误区：
- 只算存储总量，不考虑写入 IOPS
- 忽略单条数据元数据开销（索引、日志等，通常额外占 30-50%）
- 带宽只算下行不算上行或忽略内网带宽限制

#### 1.2.3 阶段 3-4：高层设计与组件选型

高层设计要画出以下要素：

1. **客户端层**（浏览器 / App / SDK）→ CDN / DNS
2. **接入层**（负载均衡）→ Nginx / LVS / ALB
3. **网关层**（API Gateway）→ 限流 / 鉴权 / 路由
4. **业务服务层**（微服务集群）
5. **数据层**（缓存 / 数据库 / 消息队列 / 对象存储）
6. **基础设施**（监控 / 日志 / 配置中心 / 服务发现）

#### 1.2.4 阶段 5：详细设计关键点

详细设计要深入到能够指导编码的程度，典型关注点：

- **数据结构选型**：HashMap / SkipList / B+ Tree / Bloom Filter
- **一致性方案**：强一致 Two-Phase Commit / Paxos / Raft vs. 最终一致 TCC / Saga / 本地消息表
- **并发控制**：乐观锁（版本号 / CAS）vs. 悲观锁（行锁 / 分布式锁）
- **分片策略**：Hash 分片 / Range 分片 / 一致性 Hash + 虚拟节点
- **高可用**：主从 / 多副本 / 多活

#### 1.2.5 阶段 6：扩展思考与 Trade-off

面试加分项 —— 主动展示"架构师级"的思考深度：

- **成本分析**：这个方案每个月的机器成本是多少？
- **演进路径**：如果数量级再涨 10 倍，哪里会先成为瓶颈？
- **替代方案**：为什么不用 XXX（Kafka vs. Pulsar、MySQL vs. TiDB）？
- **容灾方案**：单机房挂了怎么办？跨区域复制延迟？

## 2. 经典系统设计题目深度剖析

### 2.1 设计短 URL 系统

**难度评级**：基础但考察全面，覆盖算法、存储、重定向、缓存多个维度。

#### 2.1.1 需求分析

| 维度 | 内容 |
|------|------|
| **FR** | 长 URL 转短 URL（返回 7 位短码）；短 URL 访问时 302 重定向到原 URL；支持过期时间；支持访问统计 |
| **NFR** | 写入 QPS ~1K；读取 QPS ~100K；P99 延迟 < 50ms；短码总量预估 10 亿级别 |
| **选问** | 是否支持自定义短码？短码永久有效还是定期过期？是否需要防碰撞？ |

#### 2.1.2 容量预估

| 指标 | 计算 |
|------|------|
| DAU | 100M |
| 每天短链生成量 | 100M × 0.01 = 1M/天 |
| 每天短链访问量 | 1M × 100 = 100M/天 |
| 平均 QPS 写入 | 1M / 86400 ≈ 12 QPS |
| 峰值 QPS 写入 | 12 × 10 = 120 QPS |
| 平均 QPS 读取 | 100M / 86400 ≈ 1157 QPS |
| 峰值 QPS 读取 | 1157 × 10 ≈ 12K QPS |
| 3 年存储总量 | 1M × 365 × 3 = ~1.1B 行 |
| 单行数据大小 | ~200 bytes（短码 + 长 URL + 创建时间 + 过期时间） |
| 总存储 | 1.1B × 200 = ~220 GB（不算索引） |

#### 2.1.3 方案核心：短码生成算法

面试中短码生成是最核心的考点，需要对比以下三种方案：

**方案 A：哈希 + 碰撞处理**

| 项 | 说明 |
|----|------|
| 算法 | MD5(url + 盐) → 取前 7 位 base62 编码 |
| 碰撞处理 | 若短码已存在且长 URL 不同，追加新盐值重新计算 |
| 优点 | 无中心化依赖，客户端可独立生成 |
| 缺点 | 存在理论碰撞概率；需要查 DB 确认碰撞；长 URL 相同短码不同（可做去重） |

可以用 Bloom Filter 加速碰撞检测，将查 DB 的 O(1) 降到内存 O(k)。

**方案 B：自增 ID + 62 进制编码**

| 项 | 说明 |
|----|------|
| 算法 | 发号器生成自增 ID (Snowflake / DB Sequence) → 转为 62 进制短码 |
| 62 进制 | `0-9a-zA-Z` 共 62 个字符，7 位可表示 62^7 ≈ 3.5 万亿组合 |
| 优点 | 天然无碰撞；实现简单 |
| 缺点 | 中心化发号器有性能瓶颈；短码可预测（安全问题） |

**方案 C：预生成池 + 随机分配**

| 项 | 说明 |
|----|------|
| 算法 | 提前批量生成短码放到 Redis 队列，请求时从队列取一个 |
| 优点 | 写入路径极快（只需 POP）；预生成可做去重检查 |
| 缺点 | 需要后台维护短码池水位；短码浪费 |

**面试推荐方案**：方案 B 做基础框架 + 对 ID 做异或混淆（`short_id = id XOR MASK`）防止短码可预测 + Redis 预生成池加速。

#### 2.1.4 重定向策略：301 vs 302

| 对比项 | 301 (Moved Permanently) | 302 (Moved Temporarily) |
|--------|------------------------|------------------------|
| 浏览器缓存 | 会缓存重定向结果，下次不再请求短链服务 | 不缓存，每次请求都经过短链服务 |
| 访问统计 | 无法精确统计访问量（浏览器不回调） | 可以精确统计每次访问 |
| 性能 | 后续访问更快（跳过短链服务） | 每次都要查短链 |
| 场景 | 已知永久不变的短链 | 需要统计、需要兜底策略的通用场景 |

**生产建议**：先用 302 做统计 + 安全检查（是否过期/是否恶意链接），对经过验证的热门链接可返回 301 减轻服务压力。

#### 2.1.5 详细设计

**数据存储：**

```sql
CREATE TABLE short_urls (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    short_code VARCHAR(7) NOT NULL UNIQUE,
    original_url TEXT NOT NULL,
    user_id BIGINT,
    created_at DATETIME NOT NULL,
    expires_at DATETIME,
    INDEX idx_short_code (short_code),
    INDEX idx_user_id (user_id)
);
```

**分库分表方案：**

10 亿数据量必须分片。按短码哈希分 256 个库 × 每个库 256 张表 = 65536 张表。

```
shard_key = hash(short_code) % 256  -- 选择库
table_id = hash(short_code) / 256 % 256  -- 选择表
```

**缓存策略：**

| 层级 | 说明 | 数据量 | 策略 |
|------|------|--------|------|
| L1 本地缓存 | Guava Cache / Caffeine | 热点 ~1M 条 | LRU + TTL（2 min） |
| L2 分布式缓存 | Redis Cluster | 全量 ~1B 条 | short_code → original_url + TTL（24 h） |
| 穿透防护 | Bloom Filter | 10M bits | 拦截不存在短码的查询 |

**重定向流程：**

```
用户点击短链接
  → DNS 解析到短链服务
  → Nginx 负载均衡
  → 查 L1 本地缓存 (快路径)
    → 命中: 返回 302/301
    → 未命中:
      → 查 L2 Redis
        → 命中: 返回 302 + 回写 L1
        → 未命中:
          → 查 Bloom Filter
            → 不存在: 直接返回 404
            → 存在: 查 DB
              → 查到: 返回 302 + 回写 L1/L2
              → 未查到: 返回 404
```

#### 2.1.6 扩展思考

1. **短码过期处理**：后台定时任务扫描过期记录 + 删除 Redis 缓存 + 重建 Bloom Filter
2. **恶意 URL 检测**：短码生成时调用安全扫描服务（Google Safe Browsing API）；用户举报 + 人工审核
3. **自定义短码**：预留一个特殊段位的 ID 范围给自定义短码，在哈希路由中做特殊处理
4. **访问统计**：使用消息队列异步写入访问日志（异步 + 削峰），下游用 Flink 做实时聚合统计

### 2.2 设计分布式 ID 生成器

#### 2.2.1 需求分析

| 需求 | 要求 |
|------|------|
| **全局唯一** | 所有 ID 在全局范围内不重复 |
| **趋势递增** | 数据库索引友好，B+ Tree 顺序写优于随机写 |
| **高可用** | 不能有单点故障，可用性 > 99.99% |
| **高性能** | 单机 10K+ TPS，整体 100K+ TPS |
| **有序性** | 生成 ID 大致有序（非严格递增），方便业务排序 |

#### 2.2.2 核心方案对比

**方案一：Snowflake（雪花算法）**

```
 0 | 41 bit timestamp | 10 bit workerId | 12 bit sequence
```

| 段 | 位数 | 含义 | 最大值 |
|----|------|------|--------|
| 符号位 | 1 bit | 固定 0 | - |
| 时间戳 | 41 bit | 毫秒级时间戳（相对于自定义纪元） | 2^41 ms ≈ 69 年 |
| 机器 ID | 10 bit | 数据中心 5bit + 机器 5bit (或统一 10bit) | 1024 台 |
| 序列号 | 12 bit | 同一毫秒内的自增序列 | 4096/ms |

**机 QPS 上限**：1000 × 4096 ≈ 409.6 万/秒（理论）

**时钟回拨问题**：Snowflake 严重依赖机器时钟，回拨会导致 ID 重复。

**三种回拨处理策略：**

| 策略 | 方案 | 优缺点 |
|------|------|--------|
| 等待 | 记录上次最大时间戳，回拨时等待时钟追回 | 简单但回拨严重时等待时间长 |
| 预留序号段 | 每毫秒只使用一半序号，回拨时用预留段 | 浪费一半序号空间，适应性好 |
| 机器 ID 轮换 | 回拨发生时注销当前 workerId 重新注册 | 需要注册中心协调 |

**美团 Leaf 改进方案：** 使用 ZK 管理 workerId 分配，运行时定期上报时间戳到 ZK，检测到回拨时拒绝服务并告警。

**方案二：号段模式（Leaf Segment）**

核心思想：从 DB 批量获取 ID 号段，在本地内存中生成。

```sql
CREATE TABLE id_allocator (
    biz_tag VARCHAR(32) NOT NULL,   -- 业务标识（order/payment/user...）
    max_id BIGINT NOT NULL,         -- 当前最大已分配 ID
    step INT NOT NULL,              -- 号段步长（如 1000）
    version INT NOT NULL,           -- 乐观锁版本号
    PRIMARY KEY (biz_tag)
);
```

分配流程：

```
1. 应用启动 / 号段用尽时，请求 DB 分配新号段：
   UPDATE id_allocator SET max_id = max_id + step, version = version + 1
   WHERE biz_tag = 'order' AND version = :old_version

2. 获取到新号段 (max_id - step + 1, max_id]，加载到内存

3. 应用本地自增分配，直至号段耗尽
```

| 优势 | 不足 |
|------|------|
| 无时钟依赖，无回拨问题 | 号段用完时产生尖刺延迟（双 buffer 优化可解决） |
| QPS 极高（内存分配，几乎无网络开销） | 重启丢失未使用的 ID（但 ID 不要求连续，可以接受） |
| 支持多业务独立发号（biz_tag 隔离） | 需要 DB 作为协调节点 |

**双 Buffer 优化（Leaf 核心改进）**：

```
                 trigger     trigger
segment[0]       |            |
segment[1]       |            |
                 <--使用中-->  <--预加载-->
```

- 维护两个号段 Buffer，一个使用一个预加载
- 当前号段消耗到 10%（可配置）时触发异步加载下一个号段
- 消除 DB 访问的尖刺延迟

**方案三：美团 Leaf 整体架构**

```
                      ┌─────────────────┐
                      │   Leaf Server   │
                      │  (集群无状态)    │
                      ├─────────────────┤
                      │  Segment 模式    │
                      │  / Snowflake 模式│
                      └────────┬────────┘
                               │
              ┌────────────────┼────────────────┐
              ▼                ▼                ▼
       ┌──────────┐    ┌──────────┐    ┌──────────────┐
       │  MySQL    │    │    ZK    │    │  Redis (可选)│
       │(号段存储) │    │(WorkerID)│    │(号段备选存储)│
       └──────────┘    └──────────┘    └──────────────┘
```

- Segment 模式：MySQL + 双 Buffer，适用于对 ID 趋势递增有要求、高 QPS 场景
- Snowflake 模式：ZK + 本地时钟，适用于对 ID 有序性要求不高、无中心化场景
- 两种模式通过 REST API 切换，互不干扰

#### 2.2.3 面试推荐方案

| 场景 | 推荐方案 |
|------|---------|
| 订单号 / 流水号（需趋势递增） | Leaf Segment + 双 Buffer |
| 日志 ID / 消息 ID（仅需唯一） | Snowflake + 时钟回拨兜底 |
| 超高 QPS（百万级） | Leaf Segment 内存分配 |
| 时序敏感（ID 约等于时间） | Snowflake（时间戳高位） |

#### 2.2.4 扩展思考

- **时钟回拨终极方案**：NTP 配置 `-x` 参数（渐进式调整非跳跃式）；物理机部署 TSM（Time Synchronization Module）硬件时钟卡
- **多数据中心**：Snowflake 的 workerId 按数据中心分段（如 数据中心 1: 0-255, 数据中心 2: 256-511）
- **异构兼容**：大厂通常自研 ID 服务 + 兜底方案（超出预期时降级为 UUID 前缀，业务侧通过额外字段区分）

### 2.3 设计分布式限流器

#### 2.3.1 需求分析

| 需求 | 说明 |
|------|------|
| **FR** | 限制单位时间内请求量；支持多种限流维度（API /用户/ IP / 全局）；超限时返回 429；支持动态调整阈值 |
| **NFR** | 限流决策延迟 < 1ms；不影响正常请求性能；高可用（限流器本身不能成为瓶颈）；支持分布式场景下精确限流 |

#### 2.3.2 限流算法对比

**1. 固定窗口计数器**

```
窗口：   [00:00 ----- 00:60]  count = 100
         ↑ 到达 100，拒绝后续请求
```

| 优点 | 缺点 |
|------|------|
| 实现简单，内存消耗极低 | 窗口边界流量尖刺：前一个窗口末尾 99 个请求 + 后一个窗口开头 99 个请求可在极短时间内通过 198 个请求 |

**2. 滑动窗口日志**

记录每个请求的时间戳，统计时间窗口内请求数。

| 优点 | 缺点 |
|------|------|
| 精确控制，无边界问题 | 需要存储窗口内所有请求时间戳，内存占用大 |

**3. 滑动窗口计数器（折中方案）**

将固定窗口再分为更小的细粒度窗口（如 1s 窗口分为 10 个 100ms 子窗口），统计当前时间所处的子窗口及之前的完整窗口。

```
[00:00] [00:10] [00:20] [00:30] [00:40] [00:50] [00:60]
  10      12      15      8       20     拒绝    拒绝
                          ↑ 现在
统计 = 10+12+15+8+20 = 65 (未超限)
```

**4. 令牌桶算法**

```
                          ┌──────────────┐
 请求 ←── 消耗令牌 ─────── │    令牌桶     │ ←── 定时填充（rate）
                          │ 容量 = burst  │
                          └──────────────┘
```

| 参数 | 含义 |
|------|------|
| `rate` | 令牌填充速率 = 平均 QPS |
| `burst` | 桶容量 = 允许的瞬时峰值 QPS |
| `tokens` | 当前可用令牌数 |

- 每次请求消耗 1 个令牌
- 定时器按 rate 往桶中放入令牌
- 桶满则不再增加
- 令牌不足时请求被限流

| 优点 | 缺点 |
|------|------|
| 允许一定程度的突发流量（burst 控制） | 需要定时器或延时计算，实现稍复杂 |
| 平均速率可控 | 长时间空闲后突发用光 burst 仍正常 |

**5. 漏桶算法**

```
                    ┌──────────────┐
  请求 → 排队进入 → │ 漏桶（队列）  │ → 以恒定速率流出
                    │ 容量 = queue  │
                    └──────────────┘
                    ↑ 队列满则丢弃
```

| 优点 | 缺点 |
|------|------|
| 输出速率严格固定，适合保护下游系统 | 无法应对突发流量，导致请求大量排队超时 |
| 实现简单 | 不适合需要允许突发的场景 |

**选型建议：**

| 场景 | 推荐算法 |
|------|---------|
| API 网关全局限流 | 令牌桶（允许突发但均值可控） |
| 用户维度细粒度限流 | 滑动窗口计数器（内存友好 + 精度可接受） |
| 保护下游脆弱系统 | 漏桶（输出严格控速） |
| 防刷 / 爬虫 | 固定窗口 + IP 黑名单联动 |

#### 2.3.3 分布式限流实现：Redis + Lua

**核心思路**：利用 Redis 单线程 + Lua 脚本原子性实现分布式计数器。

```lua
-- Lua 脚本：滑动窗口计数器限流
local key = KEYS[1]            -- 限流 key (e.g., "ratelimit:api:/order:user_123")
local window_ms = tonumber(ARGV[1])  -- 时间窗口 (ms)
local max_permits = tonumber(ARGV[2]) -- 最大请求数
local now = tonumber(ARGV[3])   -- 当前时间戳 (ms)

-- 移除窗口外旧数据
redis.call('ZREMRANGEBYSCORE', key, 0, now - window_ms)

-- 统计窗口内请求数
local count = redis.call('ZCARD', key)

if count < max_permits then
    redis.call('ZADD', key, now, now .. ':' .. math.random())
    redis.call('EXPIRE', key, window_ms / 1000 + 1)
    return 1  -- 通过
else
    return 0  -- 限流
end
```

**性能优化：**

| 优化手段 | 说明 |
|---------|------|
| Pipeline / Batch | 批量请求合并发送，省去多次网络 RTT |
| 本地 Token Bucket + 远程同步 | 每台机器维护本地令牌桶，定时与 Redis 同步配额 |
| 一致性哈希 | 相同限流 key 落同一 Redis 节点，避免原子性失效 |
| 降级方案 | Redis 挂了后降级到本地限流（允许一定误差） |

#### 2.3.4 Sentinel 原理剖析

Sentinel 是阿里开源的限流降级组件，核心架构：

```
            ┌──────────────────────┐
            │     Sentinel Core     │
            ├──────────────────────┤
            │ 责任链模式 (ProcessorSlotChain) │
            │  NodeSelectorSlot    │
            │  ClusterBuilderSlot  │
            │  StatisticSlot       │  ← 统计指标
            │  FlowSlot            │  ← 限流规则
            │  DegradeSlot         │  ← 降级规则
            │  SystemSlot          │  ← 系统保护
            └──────────────────────┘
```

**核心设计要点：**

| 特性 | 说明 |
|------|------|
| **数据统计** | 滑动窗口（LeapArray），秒级/分钟级统计 |
| **限流策略** | 直接拒绝 / Warm Up（冷启动预热） / 排队等待（匀速器） |
| **集群限流** | Token Server + Token Client 模式（类比令牌桶） |
| **指标采集** | 每秒采样，内存 RingBuffer 存储 60s 热数据 |
| **动态规则** | 通过控制台推送限流规则到应用侧（Push 模式） |

**Warm Up（冷启动）原理：**

```
      QPS
       │
  burst│  ┌────
       │  │     ────
   rate│──┘          ────
       │                   ────
       └─────────────────────────► 时间
               预热期
```

- 初始化时系统的令牌桶处于"冷"状态（桶容量小）
- 随着时间推移逐步增加流量（预热时间由 `warmUpPeriodSec` 控制）
- 基于 Guava `RateLimiter` 的 `SmoothRateLimiter` 实现（storedPermits 冷启动因子）

#### 2.3.5 扩展思考

- **动态限流**：根据后端服务 RT / 错误率自动调整限流阈值（类似 TCP 拥塞控制 AIMS）
- **按优先级降级**：VIP 用户使用独立的令牌桶，普通用户共享限流
- **限流监控**：限流次数作为核心监控指标，P1 告警；限流率（被限流数/总请求数）作为服务质量风向标
- **容量规划**：定期（如每周）进行压测 + 记录最大 QPS，自动调整限流配置
- **手动拒绝 vs 排队**：秒杀场景用排队（用户等待）；IO 密集场景用快速拒绝

#### 2.4.4 长连接网关设计

**核心职责：**

1. **连接管理**：维护与客户端的长连接（WebSocket / TCP），周期性心跳检测
2. **协议适配**：私有协议 ↔ 内部 RPC 协议转换
3. **连接路由**：用户 ID → 连接所在的 Gateway 节点映射（存储在 Redis）
4. **离线缓存**：短时间离线消息在 Gateway 内存中暂存

**状态路由表（Redis）：**

```
user_id → {
    "gateway_ip": "10.0.0.1:9001",
    "conn_id": "abc-123",
    "last_heartbeat": 1680000000123,
    "client_type": "ios",        // 多端登录
    "device_id": "device_xxx"
}
```

单用户支持多端登录（手机 + PC + Web），每条连接记录独立，消息投递时遍历所有设备。

**连接迁移（重连）**：客户端断线后重新连接可能落到不同 Gateway，需通过 Redis 路由表找到新位置。

#### 2.4.5 消息投递流程

**单聊消息：**

```
用户A → Gateway_A → Message Service
  → 查 Redis 用户B在线状态
    → 在线: 查 B 所在 Gateway_B → Push
    → 离线: 写入离线消息队列
  → 写入消息存储 (DB)
  → 返回 ACK 给 A
```

**群聊投递：扩散写 vs 扩散读**

| 对比项 | 扩散写 (Write Fanout) | 扩散读 (Read Fanout) |
|--------|----------------------|---------------------|
| 做法 | 群消息到达时给每个群成员写一条消息 | 群消息只存一份，用户拉消息时聚合 |
| 存储量 | O(N) 每消息 (N=群成员数) | O(1) 每消息 |
| 写入延迟 | 大群极高（群 1 万人 = 1 万次写入） | 低（只需写一次） |
| 读延迟 | 低（已对应到个人收件箱） | 高（需要聚合群消息 + 个人消息） |
| 场景 | 小群 (<1000 人) | 大群 (超大群 / 全员群) |

**主流方案：推拉结合**

```
小群 (<1000 人) → 扩散写（写入每个成员的收件箱）
大群 (≥1000 人) → 扩散读（消息只存一份，成员拉取时按时间范围读取）
                 + 在线成员 Push 通知（消息体精简或只含消息 ID）
```

#### 2.4.6 已读/未读设计

**单聊已读：**

```
用户B 拉取消息 → 发送已读回执 → 更新已读点 (Redis + 异步持久化)
    ↑ user_id + session_id + last_read_msg_id
```

用户A 查看会话列表时显示 "对方已读/未读"：
```
查 Redis: userB_read_point[会话ID]
比较: last_sent_msg_id <= userB_read_point ? "已读" : "未读"
```

**群聊已读：**

群聊中 "已读 X/Y 人" 的数据结构：

```
Redis Hash:
  group_read_count{msg_id} = {"count": 500, "members": []bitmap}

实现方式：
- Bitmap：群最大 1 万人，每位代表一个成员
- 已读标记：SETBIT group_read_flag{msg_id} member_index 1
- 计数：BITCOUNT group_read_flag{msg_id}
- 内存占用：1 条群消息的已读 Bitmap ≈ 1.25KB（万人群）
- 注意：已读 Bitmap 需要定期淘汰，避免占用过多内存
```

#### 2.4.7 离线消息

| 存储 | 内容 | 清理策略 |
|------|------|---------|
| Redis 最近 N 条 | 每个用户最近 200 条消息（滑动窗口） | 达到上限淘汰最早 |
| HBase / DB | 全量离线消息（按用户 + 时间戳排序） | 90 天过期（TTL） |
| 通知栏 (Push) | 仅推送通知标题（APNs / FCM / 国内厂商通道） | — |

用户上线时流程：

```
用户上线 → Gateway 注册
  → 推送 Redis 未读消息（N 条，即时消费）
  → 异步加载 DB 历史消息（按需拉取）
  → 同步最新群消息（通过群拉模式拉取离线期间增量）
```

#### 2.4.8 扩展思考

- **消息去重**：客户端生成消息 ID（UUID / Snowflake），服务端通过消息 ID 做幂等（Redis Set 记录 3 分钟内的已处理消息 ID）
- **消息顺序保证**：同一个会话的消息通过单线程分区处理（消息按 sender + send_time 排序），客户端还需要 Local 时间戳校正
- **端到端加密**：用户公钥在服务端注册，私钥只在客户端；服务端做密钥协商的转发但无法解密内容
- **多数据中心**：按照地域就近接入，消息跨域转发通过数据中心间的专用高速通道；群消息跨域复制

#### 2.5.6 库存扣减（核心设计）

**库存扣减是秒杀系统最核心也是最容易出错的环节。核心目标：不超卖 + 高性能。**

**方案一：Redis Lua 原子扣减（推荐方案）**

```lua
-- Lua 脚本：秒杀库存扣减
local key_stock = KEYS[1]          -- stock:{item_id}
local key_user_set = KEYS[2]       -- purchased_users:{item_id}
local user_id = ARGV[1]
local quantity = tonumber(ARGV[2]) -- 购买数量
local limit = tonumber(ARGV[3])    -- 用户限购数量

-- 1. 检查用户是否已购买
local purchased = redis.call('SISMEMBER', key_user_set, user_id)
if purchased == 1 then
    return {err = "already_purchased"}
end

-- 2. 检查库存
local stock = redis.call('GET', key_stock)
if not stock or tonumber(stock) < quantity then
    return {err = "insufficient_stock"}
end

-- 3. 原子扣减
redis.call('DECRBY', key_stock, quantity)

-- 4. 记录已购买用户（用于限购检查）
redis.call('SADD', key_user_set, user_id)
redis.call('EXPIRE', key_user_set, 86400)

return {success = true}
```

**为什么用 Lua 而不是事务（MULTI/EXEC）：**
- Lua 脚本整个在 Redis 中执行，原子 + 无网络往返
- 检查-执行在单线程中完成，没有并发问题
- 一次网络调用即可完成全部逻辑

**库存预热：**

```
活动开始前：
  1. 预热数据加载：DB 库存 → Redis (stock:{item_id} = count)
  2. 预热用户已购集合：清空 purchased_users:{item_id}
  3. 预热失败 → 活动熔断，不允许开启
```

**库存同步回 DB：**

```
异步方式 1：MQ 异步扣减
  抢单成功 → 发 MQ → 消费者 更新 DB 库存

异步方式 2：定时对账
  秒杀结束后，定时任务统计 Redis 实际扣减数量，一次性写入 DB
  UPDATE stock SET remaining = remaining - :sold_count WHERE item_id = :id AND remaining >= :sold_count

对账不一致时的处理：
  少卖：补库存
  多卖（极低概率）：通知人工介入 + 退款
```

#### 2.5.7 队列削峰（第三层防护）

```
抢单成功 → 发 MQ（消息包含：用户ID、商品ID、数量、Token）
                  ↓
            MQ Consumer（限速消费，如每秒处理 5000 单）
                  ↓
              订单创建服务
```

**MQ 选型考量：**

| 特性 | 要求 | 推荐 |
|------|------|------|
| 削峰填谷 | 支持大量堆积 | RocketMQ / Kafka |
| 事务保障 | 写 DB + 发 MQ 原子性 | RocketMQ 事务消息 |
| 顺序性 | 同一商品可并行 | 无需严格全局有序 |
| 高可用 | 不能丢失消息 | 同步刷盘 + 主从复制 |

**排队策略对比：**

| 方式 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| 内存队列 | JVM 内部阻塞队列 | 极低延迟 | 重启丢数据，容量受限 |
| MQ | 消息队列异步 | 持久化，可堆积 | 有网络延迟 |
| Redis List | LPUSH + BRPOP | 轻量，性能好 | 无消费者确认机制 |

#### 2.5.8 数据一致性保障

| 场景 | 问题 | 方案 |
|------|------|------|
| Redis 扣减成功但 MQ 发送失败 | 库存扣了但订单没创建 | 事务消息（RocketMQ 半消息）：MQ 准备就绪才真正扣库存 |
| 订单创建失败 | 库存已扣，需回滚 | 订单服务失败 → 发送回滚 MQ → 回滚库存（+ 释放已购记录） |
| MQ 重复消费 | 同一条消息创建多个订单 | 消费者幂等（订单号 = 秒杀 Token + 用户ID，唯一索引） |
| Redis 宕机 | 库存数据丢失 | Redis 主从 + AOF 持久化；重启后从 DB 重建库存 |

#### 2.5.9 扩展思考

- **极致优化**：热点库存 Key 在 Redis 集群中打散（`stock:{item_id}:{segment}`），避免单分片瓶颈
- **单体 Redis vs Redis Cluster**：秒杀场景推荐 Redis 集群，但热点 Key 仍然可能打满单分片；更优方案是本地缓存 + 库存分段
- **封禁恶意用户**：基于行为模式识别（同 IP 大量请求、相同 User-Agent 异常模式）、风控系统实时评分，高风险用户直接拒绝
- **大屏监控**：秒杀过程中实时监控 Redis 库存、MQ 堆积数、订单创建速率、下单成功率，设置自动化熔断条件
- **灰度发布**：先开放 10% 流量，验证系统稳定后逐步放量

### 2.6 设计 Feed 流系统

#### 2.6.1 需求分析

| 需求 | 说明 |
|------|------|
| **FR** | 用户发布内容（图文/视频）；关注用户的内容出现在 Timeline；Timeline 按时间倒序；支持分页拉取 |
| **NFR** | Timeline 加载延迟 < 500ms（P99）；写入延迟 < 200ms；支持 10 亿用户 / 百万级大 V；推送一致性（不丢不漏不重） |
| **选问** | Timeline 是严格时序还是允许少量乱序？是否需要推荐算法介入（非纯粹时序）？是否需要热点加权？ |

#### 2.6.2 核心方案对比

**推模式（Fanout on Write / 写扩散）**

```
用户A发布 → 遍历A的粉丝（100个）
           → 写入每个粉丝的收件箱（Redis List / Timeline Cache）
```

| 优点 | 缺点 |
|------|------|
| 读时延迟极低（直接读收件箱） | 大 V 粉丝千万级，一次发布写入千万次，延迟极高 |
| 读路径简单，易于分页 | 写入放大严重，沉默粉丝占大量存储 |
| 收件箱支持冷热分离 | 粉丝数线性增加写入开销 |

**拉模式（Fanout on Read / 读扩散）**

```
用户B刷Feed → 遍历B关注的所有人（500人）
            → 分别拉取每人最近 N 条 → 归并排序
```

| 优点 | 缺点 |
|------|------|
| 发布延迟低（只写一条） | 读取延迟高（需要拉取 + 归并） |
| 无写放大问题 | 关注的人多时归并排序复杂 |
| 存储成本低 | 分页实现复杂（必须记录每个源的游标） |

**推拉结合（混合方案）**

按粉丝数量分级：

| 用户类型 | 粉丝量 | 策略 |
|---------|--------|------|
| 普通用户 | < 10 万 | 推模式（写扩散到所有粉丝） |
| 大 V / 明星 | > 10 万 | 仅在线粉丝推 + 离线粉丝拉 |
| 超大 V | > 1000 万 | 拉模式（粉丝主动拉取） + 在线通知 |

**大 V 发布流程（推拉结合）：**

```
大V发布内容
  → 写入大V的个人发布箱（Redis ZSET：timestamp → content_id）
  → 查询在线粉丝列表（活跃用户）
    → 推送到在线粉丝的收件箱
  → 离线粉丝：登录时从其关注的大V个人发布箱拉取最新内容
```

#### 2.6.4 收件箱 / 时间线存储

**Redis 方案：**

```
收件箱 Key: inbox:{user_id}       (Sorted Set)
  Score = timestamp (或基于时间的 ID)
  Value = content_id

个人发布箱: outbox:{user_id}      (Sorted Set)
  同上
```

每个用户收件箱保留最近 1000 条（滑动窗口），超过的从 DB 拉取。

**分页一致性问题：**

| 问题 | 现象 | 解决方案 |
|------|------|---------|
| 翻页时新内容插入 | 第二页出现第一页已看过的内容 | 使用 Cursor 分页（返回 `last_timestamp`）代替 OFFSET |
| 翻页时内容被删除 | 丢失一条内容（可接受） | 不需要严格保证 |
| 大 V 发布后 Timeline 更新 | 用户看到半新半旧 | 客户端在刷新时重新加载首屏 |

**推荐分页方式：**

```
第一次请求  Timeline API:
  → 返回 content_ids + next_cursor (最后一条的时间戳)
第二次请求  GET /timeline?cursor=1680000000123&count=20
  → ZREVRANGEBYSCORE inbox:{uid} 1680000000123 0 LIMIT 20
```

#### 2.6.5 热度加权（非纯粹时序）

对于非纯粹时序的 Feed（如抖音/微博推荐流），需要引入热度加权排序：

```
Score = base_score × time_decay + interaction_score
       = f(content_type) × 0.5^(t / T) + (likes×0.3 + comments×0.5 + shares×0.2)
```

其中：
- `base_score`：基础分（原创 vs 转发、文本 vs 视频）
- `time_decay`：时间衰减函数（半衰期 T 可调节）
- `interaction_score`：互动分数

通常离线计算（Spark/Flink 定时刷新）而非实时写入，避免写路径复杂度暴增。

#### 2.6.6 扩展思考

- **"已读"位置恢复**：用户退出再进来，从上次读到的地方继续。方案：Redis 记录 `last_read_timestamp`，翻页时作为起始游标
- **存储冷热分离**：收件箱：Redis（近 1 天热数据）→ SSD Cache（近 7 天温数据）→ HBase/MySQL（全量冷数据）
- **大 V 冷启动**：新用户关注大 V 后，需要拉取大 V 最近 N 条内容写入新用户收件箱
- **消息合并**：同一用户短时间内关注多个人的发布，可以将拉请求合并批量处理
- **去重策略**：同一条内容（被转载/@）在 Timeline 出现多次，客户端根据 content_id 去重

#### 2.7.3 调度器设计

**时间轮算法（秒级调度）：**

```
           ┌───┐
           │ 0 │ ← tick (当前秒)
           ├───┤
           │ 1 │
           ├───┤
           │ 2 │
           ├───┤
           │...│
           ├───┤
           │59 │
           └───┘

每个槽位维护一个 Task List：
  槽位 0: [task_a, task_b]
  槽位 1: []
  槽位 2: [task_c]
  ...
```

- 心跳每秒走一格，执行当前槽位的所有任务
- 支持秒级、分钟级、小时级调度
- 分钟级任务注册到分钟时间轮（60 槽），小时级单独处理

**调度器高可用：**

```
多 Scheduler 节点通过分布式锁（ZK / Redis）竞选 Leader：
  Leader: 负责时间轮推进、任务分发
  Follower: 热备，Leader 心跳超时时抢锁接管

Leader 崩溃恢复：
  1. 新的 Leader 上线
  2. 从 DB 读取当前未完成任务列表
  3. 重建时间轮（跳过已调度的过去时间点）
  4. 开始正常调度
```

#### 2.7.4 任务分片

对于大数据量任务（如全量数据导出），需要进行分片并行执行。

```
任务定义：
  handler = "DataExportHandler"
  shard_param = "{\"total\": 10}"  // 分 10 片

分片执行：
  Worker 1 (shard 0): SELECT * FROM table WHERE id % 10 = 0
  Worker 2 (shard 1): SELECT * FROM table WHERE id % 10 = 1
  ...

分片策略：
  - 按 ID 取模（需要连续自增主键）
  - 按日期范围（date between '2024-01-01' and '2024-01-07'）
  - 按业务分桶（地域 / 用户分组）
```

#### 2.7.5 故障恢复与幂等

**执行器故障场景：**

```
场景 1: Worker 在执行中宕机
  → Scheduler 心跳超时检测到 Worker 失联
  → 将该 Worker 的任务标记为 "FAILED" 或 "RERUN"
  → 重新调度到可用 Worker

场景 2: 任务执行超时
  → Scheduler 设置超时时间（如 30 分钟）
  → 超时后强制标记失败，重试

场景 3: 任务执行成功但状态回写失败
  → Worker 写入执行结果超时
  → 通过"任务实例 ID"做幂等查询（是否已经执行成功）
```

**幂等执行设计：**

```sql
CREATE TABLE task_instance (
    id BIGINT PRIMARY KEY,
    task_id BIGINT NOT NULL,
    shard_id INT,
    status TINYINT NOT NULL,     -- 0: PENDING, 1: RUNNING, 2: SUCCESS, 3: FAILED
    execute_time DATETIME,
    worker_id VARCHAR(64),
    version INT DEFAULT 0,
    UNIQUE KEY uk_task_shard (task_id, shard_id, execute_time)
);
```

执行流程（CAS + 乐观锁）：

```
1. SELECT status FROM task_instance WHERE id = :instance_id
2. if status == SUCCESS → 跳过（幂等）
3. UPDATE task_instance SET status = RUNNING, version = version + 1
   WHERE id = :instance_id AND version = :old_version
   if affected_rows == 0 → 冲突，跳过
4. 执行任务逻辑
5. UPDATE task_instance SET status = SUCCESS WHERE id = :instance_id
```

#### 2.7.6 DAG 任务编排

**拓扑排序执行：**

```
任务 A ──→ 任务 B ──→ 任务 D
   │                  ▲
   └──→ 任务 C ──────┘

依赖关系（DAG 定义）：
  - B 依赖 A
  - C 依赖 A
  - D 依赖 B 和 C

执行流程：
  Step 1: 并发执行 A（入度为 0）
  Step 2: A 完成后，B 和 C 入度归零 → 并发执行 B, C
  Step 3: B, C 都完成后，D 入度归零 → 执行 D
```

**实现关键：**

- 每个任务实例记录 `depend_on` 列表（JSON 或关联表）
- 任务完成后扫描其"下游任务"，减少入度计数
- 入度归零的任务加入就绪队列（时间轮或立即分发）
- 循环依赖检测：创建 DAG 时做拓扑排序校验，有环则创建失败

#### 2.7.7 扩展思考

- **Executor 动态扩缩**：根据任务队列深度自动扩容 Worker 数量，配合 K8s HPA
- **任务优先级**：线上故障修复任务优先于离线报表任务，通过优先级队列实现
- **调度日志**：每个任务实例的全链路日志收集（ELK / Loki），支持按实例 ID 聚合查询
- **优雅关闭**：Worker 收到 SIGTERM 后拒绝新任务，等待当前任务执行完毕再退出
- **Quartz vs XXL-Job vs 自研**: XXL-Job 适合中小团队快速落地；自研适合对调度策略有深度定制需求的大厂

### 2.8 设计实时排行榜

#### 2.8.1 需求分析

| 需求 | 说明 |
|------|------|
| **FR** | 实时更新分数；查询排行榜 Top N；查询用户个人排名；支持多维度（日榜/周榜/月榜/总榜）；支持同分并列规则 |
| **NFR** | 更新延迟 < 100ms；排行榜查询延迟 < 50ms；支持百万级用户 / 秒级更新频率；榜单展示一致性（不允许跳变） |
| **选问** | 是计分制（加分）还是总分累加？是实时全量榜还是分段榜？需要历史快照吗？ |

#### 2.8.2 核心方案：Redis Sorted Set

Redis Sorted Set 是实时排行榜最核心的数据结构：

```
ZADD leaderboard:daily:2024-01-01 1500 user_a     -- 设置分数
ZINCRBY leaderboard:daily:2024-01-01 100 user_a   -- 增加分数
ZREVRANGE leaderboard:daily:2024-01-01 0 9        -- Top 10
ZREVRANK leaderboard:daily:2024-01-01 user_a       -- 查询排名
```

**时间复杂度：**

| 操作 | 复杂度 | 说明 |
|------|--------|------|
| ZADD | O(log N) | 跳表插入 |
| ZINCRBY | O(log N) | 更新分数 |
| ZREVRANGE | O(log N + M) | Top N (M=N) |
| ZREVRANK | O(log N) | 查排名 |

#### 2.8.3 单 Key 瓶颈与优化

**问题：单日榜 Key 当用户量达到 1000 万时，ZADD 仍有足够性能，但存在以下问题：**

1. **热 Key 问题**：单 Key 落在 Redis 集群一个分片上，更新压力集中
2. **大 Key 问题**：1000 万成员 × (20 B member + 8 B score + 跳表指针开销) ≈ 300~500 MB，大 Key 影响集群稳定性
3. **全量排名查询慢**：ZREVRANGE 深度翻页性能差（ZREVRANGE 底层是跳表，offset 大时遍历代价高）

**优化方案：分段排名**

```
┌──────────────────────────────────────┐
│  Top 100 段 (leaderboard:daily:top)   │ ← Redis ZSET (100成员)
│  └── 分数 >= 10000                    │
├──────────────────────────────────────┤
│  第 1 分段 (leaderboard:daily:seg_1)  │ ← Redis ZSET
│  └── 5000 <= 分数 < 10000            │
├──────────────────────────────────────┤
│  第 2 分段 (leaderboard:daily:seg_2)  │ ← Redis ZSET
│  └── 2000 <= 分数 < 5000             │
├──────────────────────────────────────┤
│  第 M 段 (leaderboard:daily:seg_M)    │ ← Redis ZSET
│  └── 分数 < 100                      │
└──────────────────────────────────────┘
```

**分段规则：**

```
分段依据：分数范围（推荐）或用户 ID 哈希
每个分段大小控制在 10 万成员以内
Top-N 段（Top 100/500）最多一个分段

用户分数更新时：
  1. 计算分数所在分段
  2. ZADD 到对应分段 + 移除旧分段中的记录
  3. 若分数进入 Top 段区间 → 同时写入 Top 段
```

**查询 Top N：**

- 直接查询 Top 段即可，ZREVRANGE 0 9 快速返回
- 若 N 小于 Top 段成员数，仅查 Top 段
- 若 N 大于 Top 段，需要合并 Top 段 + 后续分段

#### 2.8.4 时间维度榜单

| 榜单类型 | Key 设计 | 说明 |
|---------|---------|------|
| 日榜 | `leaderboard:daily:2024-01-01` | 精确到天的 Sorted Set |
| 周榜 | `leaderboard:weekly:2024-W01` | 每周一个 Key，或聚合 7 天日榜合并 |
| 月榜 | `leaderboard:monthly:2024-01` | 每月一个 Key |
| 总榜 | `leaderboard:total` | 累计，需要定期快照 |

**周榜/月榜的更新策略：**

| 策略 | 实现 | 优缺点 |
|------|------|--------|
| 实时写 | 每次更新同时写日榜 + 周榜 + 月榜 | 写放大 3x，但读快 |
| 异步聚合 | 只写日榜，T+1 离线聚合 7 天为周榜 | 写路径干净，但周榜有延迟 |
| 合并查询 | 在应用层查询 7 个日榜 Sorted Set 并归并 | 代码复杂，7 天数据量级大时性能差 |

**推荐：** 对实时性要求高的用"实时写"（日 + 周 + 月同时写入）；对一致性要求不高的用"异步聚合"（定时脚本计算）。

#### 2.8.5 同分处理

| 策略 | 实现 | 场景 |
|------|------|------|
| 先到先得 | ZREVRANGE 默认按字典序排同分 | 简单场景 |
| 时间戳辅助 | score = 分数 + (1 - 时间戳/1e13) 保证最终排序精确 | 需要精确排名 |
| 并行排名 | 同分用户排名相同，下一位跳过（如 1, 2, 2, 4） | 游戏/竞技场景 |

#### 2.8.6 历史榜单快照

```
需求：用户 A 在 2024-01-01 这一天排第 5 名，一个月后仍然能查到

方案：
  - 日榜数据在 Redis 保留 7 天后淘汰
  - 每日 23:59:59 定时脚本将日榜 Top 1000 持久化到 MySQL/HBase
  - 用户自己的排名数据异步写入 MySQL（user_rank_history 表）

返回历史排名：
  优先查 Redis（若仍在有效期内）
  查不到则从 DB 拉取历史快照
```

#### 2.8.7 百万级即时更新优化

对于高并发更新场景（如直播点赞、投票），Redis 单分片可能成为瓶颈：

**方案一：合并写入**

```
Worker (每台机器)
  ┌─────────────────┐
  │ 本地延迟队列      │
  │ (10ms 或 100条)  │
  └────────┬────────┘
           │ 批量 ZINCRBY
           ▼
  ┌─────────────────┐
  │ Redis SortedSet │
  └─────────────────┘
```

- 每个 Worker 累积用户的分数变更（Map<userId, delta>）
- 每 10ms 或累积 100 条发送一次批量 ZINCRBY
- 减少 Redis 网络 IO（约 10-100 倍）

**方案二：多级榜单**

```
L1: 本地内存排名（服务端维护 Top 100 近似榜）
L2: Redis Sorted Set（精确榜，承担写流量）
L3: DB 持久化（历史榜 + 兜底）
```

- Top N 查询直接走 L1 内存（0ms 延迟）
- 写入走 L2，L1 定时从 L2 同步全量 Top 100
- 用户查排名走 L2

#### 2.8.8 扩展思考

- **Redis 集群注意事项**：Sorted Set 大 Key 散列（hash tag 方式将不同榜单的 Key 散列到不同分片，但单榜大 Key 仍是痛点）
- **降级方案**：Redis 不可用时降级为"排行榜维护中"或展示本地缓存的上一周期榜单
- **实时 vs 准实时**：严格实时用 Redis Streams + ZINCRBY；准实时用 Flink 窗口聚合 + 定期写入 Redis
- **Bloom Filter 查排名**：对不进入 Top N 的用户，无需实时精确排名，可以每隔一段时间批量计算分段排名

## 3. 核心组件选型指南

### 3.1 负载均衡

| 组件 | 层级 | 性能 | 特点 | 适用场景 |
|------|------|------|------|---------|
| **LVS** | L4 (传输层) | 千万级 QPS | 基于 IP+端口转发，性能最高；支持 DR / NAT / TUN 模式；不能识别 HTTP 内容 | 四层入口，前端统一接入 |
| **Nginx** | L7 (应用层) | 十万级 QPS | 基于 HTTP/HTTPS 内容转发；支持健康检查、限流、缓存、SSL 卸载；配置灵活 | Web 服务反向代理、API 网关前置 |
| **Envoy** | L7 | 十万级 QPS | 基于 C++ 实现，现代架构设计；支持 HTTP/2、gRPC、服务发现、流量镜像、熔断；xDS 协议动态配置 | Service Mesh（Istio 数据面）、微服务网关 |
| **ALB/SLB** | L4/L7 | 托管弹性 | 云厂商托管，免运维；自动伸缩、SSL 托管、WAF 集成 | 云原生架构，减少运维成本 |

**选型建议：**

```
入口流量: LVS (L4) → Nginx (L7)  → 应用服务
微服务:   Envoy (Sidecar) 或 API Gateway
云原生:   云厂商 LB + Nginx Ingress Controller
```

### 3.2 缓存

| 组件 | 类型 | 性能 | 特点 | 场景 |
|------|------|------|------|------|
| **Redis** | 分布式缓存 | 单机 10万+ QPS | 丰富数据结构（String/Hash/ZSET/Stream）；支持主从/集群/哨兵；持久化（RDB/AOF） | 通用分布式缓存、分布式锁、排行榜、Session 存储 |
| **Local Cache** | 本地内存 | 纳秒级 | Guava Cache / Caffeine / Ehcache；不占网络 IO；容量受限于 JVM 堆内存 | 热点数据缓存（如配置项、字典表）、二级缓存（L1） |
| **CDN** | 边缘缓存 | 全球就近 | 静态资源分发；云厂商（CloudFront / Cloudflare / 阿里云 CDN） | 静态文件（HTML/CSS/JS/IMG）、流媒体、API 响应缓存 |

**缓存模式：**

| 模式 | 说明 | 场景 |
|------|------|------|
| **Cache Aside** | 读：先读缓存，miss 读 DB + 回写；写：先写 DB，再删除缓存 | 最通用 |
| **Read Through** | 缓存层代理 DB 读取，对应用透明 | 一致性要求不高的数据 |
| **Write Through** | 写操作同步写 DB + 缓存 | 需要缓存与 DB 强一致（通常配合分布式锁） |
| **Write Behind** | 写操作只写缓存，异步回写 DB | 写频繁且能容忍秒级不一致 |

**缓存一致性方案：**

1. **延迟双删**：写 DB → 删缓存 → 延迟 500ms → 再次删缓存（防并发读脏）
2. **订阅 Binlog**：Canal 订阅 MySQL Binlog → 同步更新/删除缓存（解耦，可靠）
3. **版本号**：每条数据维护版本号，缓存带版本，版本不一致时回源

### 3.3 数据库

| 组件 | 模型 | 一致性 | 水平扩展 | 场景 |
|------|------|--------|---------|------|
| **MySQL** | 关系型 | 强一致（单机/主从）；最终一致（异步复制） | 分库分表（需中间件） | 事务性强、强一致要求的结构化数据 |
| **PostgreSQL** | 关系型 | 强一致（MVCC） | 逻辑复制、PG-XL/Citus | 复杂查询、GIS、JSON 文档、需要 PG 特色的场景 |
| **TiDB** | NewSQL | 强一致（Raft + Percolator） | 原生分布式，自动分片 | 需要 MySQL 兼容 + 大容量 + 高可用的场景 |
| **HBase** | 列存 NoSQL | 强一致（单行） | Region 自动 split | 海量写入、稀疏宽表、时序数据 |
| **MongoDB** | 文档 NoSQL | 最终一致（默认）；可配置强一致 | 分片集群 | 文档灵活、快速迭代、非结构化数据 |

**MySQL 分库分表方案：**

| 组件 | 说明 | 优缺点 |
|------|------|--------|
| **ShardingSphere** | Apache 顶级项目，JDBC 层/Proxy 层 | 功能全面，社区活跃；Proxy 模式有网络损耗 |
| **MyCat** | 数据库中间件，代理层 | 较成熟，但社区活跃度下降 |
| **Vitess** | YouTube 开源，CloudSQL 兼容 | K8s 集成好，部署复杂 |
| **自研** | 大厂自研数据库中间件 | 高度定制，但维护成本高 |

**分库分表策略：**

```
垂直分库：按业务领域拆分（订单库/用户库/商品库）
水平分表：按数据量分片
  - Range 分片：按 ID 范围（0~1亿→ 库1, 1亿~2亿→ 库2）
  - Hash 分片：按 ID 取模（优点：分布均匀；缺点：扩缩容需要迁移）

通用分片规则：
  shard = hash(shard_key) % total_shards
```

**扩缩容方案：**

```
传统方案：停机迁移（停服 → 备份 → 按新规则迁移 → 切换配置 → 恢复）
在线方案：双写 + 迁移
  Step 1: 建立新库（新分片规则），开启双写（老库 + 新库）
  Step 2: 历史数据批量迁移（对比校验）
  Step 3: 读流量逐步切换到新库
  Step 4: 验证无误后停止双写

一致性 Hash 方案：减少迁移数据量（仅迁移环上前后节点）
  TiDB 等 NewSQL 内置自动分裂合并，无需手动分片
```

## 4. 架构评审能力

### 4.1 SLA / SLO / SLI

| 术语 | 含义 | 示例 |
|------|------|------|
| **SLI** (Service Level Indicator) | 可量化的服务表现指标 | 请求延迟 P99 < 200ms；可用性 99.99% |
| **SLO** (Service Level Objective) | 目标值或目标范围 | P99 延迟 <= 200ms（99.9% 的时间） |
| **SLA** (Service Level Agreement) | 服务等级协议（含后果赔偿） | 每月可用性 >= 99.9%，否则赔偿 10% 月费 |

**常见 SLI 指标：**

| 类别 | 指标 | 典型 SLO |
|------|------|---------|
| **可用性** | 请求成功率 | >= 99.9% |
| **延迟** | P50 / P95 / P99 延迟 | P99 < 500ms |
| **吞吐** | QPS / TPS | 支撑峰值流量 3x 冗余 |
| **错误率** | 5xx / 4xx 占比 | 5xx < 0.1% |
| **饱和度** | CPU / 内存 / 磁盘 / 连接数 | < 80% |

**可用性计算模型：**

```
可用性 = (总时间 - 不可用时间) / 总时间

N 个 9 的可用性：
  99%   → 全年停机 < 87.6 h
  99.9% → 全年停机 < 8.76 h
  99.99% → 全年停机 < 52.6 min
  99.999% → 全年停机 < 5.26 min
  99.9999% → 全年停机 < 31.5 s
```

**可用性计算公式（分布式系统）：**

`如果系统依赖 3 个组件，每个组件可用性 99.9%，则整体可用性 = 99.9%^3 ≈ 99.7%（全年停机 26 小时）`

### 4.2 容灾与多活架构

| 架构 | 模式 | RTO | RPO | 成本 | 适用 |
|------|------|-----|-----|------|------|
| **单机房** | 主从 / 主备 | 分钟级 | 秒级 | 低 | 非核心业务、SLA 要求低的场景 |
| **同城双活** | 两机房同时读写 + 数据同步 | 秒级 | < 5s | 中 | 金融核心、异地容灾 |
| **两地三中心** | 主 + 同城备 + 异地容灾 | 分钟级 | 分钟级 | 高 | 银行核心、监管要求 |
| **异地多活** | 多机房同时读写 + 双向同步 | 几秒 ~ 分钟 | < 1s | 极高 | 国民级应用（支付宝/微信/抖音） |

**RPO / RTO 定义：**

| 术语 | 含义 | 通俗理解 |
|------|------|---------|
| **RPO** (Recovery Point Objective) | 允许丢失的最大数据量（时间维度） | 数据丢了多久能找回 |
| **RTO** (Recovery Time Objective) | 允许恢复的最长时间 | 服务挂了多久能恢复 |

**同城双活架构要点：**

```
用户 → DNS (地域路由)
        → 机房 A (主) ←→ 专线同步 → 机房 B (主)
             │                            │
        ┌────┴────┐                 ┌────┴────┐
        │ Redis   │                 │ Redis   │
        │ 双写/CCR│                 │ 双写/CCR│
        └─────────┘                 └─────────┘
        ┌────┴────┐                 ┌────┴────┐
        │ MySQL   │                 │ MySQL   │
        │ 主主同步 │                 │ 主主同步 │
        └─────────┘                 └─────────┘
```

**关键挑战：**
- 数据库双主同步冲突（自增 ID、唯一约束冲突需要业务层处理）
- Redis 双向同步（跨机房延迟导致的数据不一致）
- 流量调度（机房级故障时 DNS/负载均衡切换，注意 DNS 缓存）

**异地多活**：更加复杂的架构，需要按用户维度做"单元化"——每个用户固定在某个机房读写，机房之间异步同步。

### 4.3 数据备份策略

| 级别 | RPO | 实现方式 | 说明 |
|------|-----|---------|------|
| **同步复制** | 0 | 主库写入 + 同步到从库（半同步/全同步） | 主从都落盘才返回成功，但跨机房时延高 |
| **异步复制** | 秒级 | 主库落盘返回，异步复制到从库 | 性能好，主宕机可能丢失少量数据 |
| **定时全量** | 天级 | 每天全量备份（mysqldump / XtraBackup） | 用于灾难恢复、误删数据恢复 |
| **Binlog 增量** | 分钟级 | 实时消费 Binlog 到备份库 / 对象存储 | 实现任意时间点恢复（PITR） |
| **跨区域复制** | 分钟级 | 通过 DTS/Canal 同步到异地 | 地理级容灾 |

**最佳实践：**

```
日备份：全量备份（保留 7 天） + Binlog 增量（保留 30 天）
周备份：全量备份（保留 1 个月）
月备份：全量备份（保留 12 个月）
年备份：全量备份（长期归档）

验证：定期（每月）执行备份恢复演练，验证备份可用
```

## 5. CAP/BASE 理论在架构设计中的实践

### 5.1 CAP 理论回顾

```
CAP 定理：分布式系统中，一致性（Consistency）、可用性（Availability）、
分区容错性（Partition Tolerance）三者最多同时满足两个。
```

**关键理解：**

- **P（分区容错）是必须的**：分布式系统必然存在网络分区，因此实际上是在 C 和 A 之间做选择
- **CP 系统**：放弃一部分可用性，保证强一致（ZooKeeper、etcd、TiDB）
- **AP 系统**：放弃强一致，保证可用性和最终一致性（DNS、CDN、Eureka）

### 5.2 CAP 在大厂实践中的演进

| 领域 | 一致性要求 | 实际选择 | 解释 |
|------|-----------|---------|------|
| 支付扣款 | 强一致 | CP（AP + 兜底补偿） | 支付领域需要强一致，但大厂也会通过逆向流程（退款）处理极端情况，实际是 AP 兜底 |
| 订单状态 | 最终一致 | 偏 AP | 订单创建 → 支付 → 履约，每一步通过 MQ 异步驱动，状态不一致通过定时对账修复 |
| 库存扣减 | 强一致 | 偏 CP | 库存扣减用 Redis 单线程 Lua（强一致），但库存回补容忍最终一致 |
| 用户资料 | 弱一致 | AP | 昵称修改后几秒内不同机房看到旧数据可以接受 |
| 朋友圈 | 最终一致 | AP | A 发了朋友圈，B 晚几秒看到也能接受 |

**实际架构设计的趋势：** 不追求严格的 CP 或 AP，而是**在数据模型层面做隔离**——关键数据走 CP 路径（短路径），非关键数据走 AP 路径（最终一致、异步补偿）。

### 5.3 BASE 理论

| 术语 | 含义 | 实践 |
|------|------|------|
| **BA** (Basically Available) | 基本可用 | 秒杀降级（部分用户返回"拥挤"而非 503） |
| **S** (Soft State) | 软状态 | 数据副本间存在短暂不一致的中间态 |
| **E** (Eventually Consistent) | 最终一致 | 经过足够长时间后，所有副本最终达到一致 |

**最终一致性的实现模型：**

```
1. 读时修复（Read Repair）：读取时检测到不一致，异步触发修复
2. 反熵（Anti-Entropy）：后台进程对比各副本差异并同步
3. 因果一致性（Casual Consistency）：基于向量时钟（Vector Clock）保证因果关系
```

### 5.4 分布式一致性协议选型

| 协议 | 性质 | 性能 | 场景 |
|------|------|------|------|
| **Raft** | 强一致（Leader 选举 + Log Replication） | 数千 TPS | etcd / Consul / TiDB / PolarDB |
| **Paxos** | 强一致（Multi-Paxos） | 数千 TPS | ZooKeeper (Zab) / Chubby |
| **Gossip** | 最终一致 | 百万级节点 | Cassandra / Redis Cluster / Consul |
| **2PC / 3PC** | 强一致（阻塞式） | 低（网络开销大） | TCC 分布式事务中的资源准备/提交阶段 |
| **TCC** | 最终一致（业务补偿） | 高 | 秒杀、跨服务扣款 |
| **Saga** | 最终一致（长事务） | 高 | 多步骤复杂业务流程 |

**面试高频问题："什么场景用强一致，什么场景用最终一致？"**

```
强一致（CP）：支付、库存、账户余额、分布式锁、配置中心
最终一致（AP + 补偿）：订单状态流转、通知推送、社交动态、缓存同步、日志收集
```

## 6. 面试答题框架与技巧

### 6.1 结构化表达：STAR 法则（面试自我介绍/项目介绍）

| 要素 | 说明 | 示例 |
|------|------|------|
| **S** (Situation) | 背景和前提 | "当时订单系统日订单量 100 万，MySQL 主库 CPU 持续 80%+" |
| **T** (Task) | 你的任务目标 | "我的任务是设计分库分表方案，支撑 3 年内日订单量增长到 500 万" |
| **A** (Action) | 你采取的行动 | "我选择了 ShardingSphere + 用户 ID 取模分 16 库 64 表，并做了数据迁移和灰度切换" |
| **R** (Result) | 量化结果 | "上线后主库 CPU 降到 20%，P99 延迟从 800ms 降到 50ms，支撑了 2 年业务增长" |

### 6.2 5 分钟说清楚一个架构

**时间分配：**

```
0:00 - 1:00   确认需求和范围（"我们设计一个支持 1000 万 DAU 的 XX 系统"）
1:00 - 2:30   画出核心架构图（三个核心组件 + 数据流向）
2:30 - 4:00   深入一个关键设计（数据库选型 / 一致性方案 / 高可用策略）
4:00 - 5:00   Trade-off 和扩展思考（"如果量级再涨 10 倍……"）
```

**关键提示：** 不要花 3 分钟讲业务背景（面试官已知）。直接从"我们面临的技术挑战是什么"开始。

### 6.3 画图技巧

面试中画图的原则（白板 / Excalidraw）：

| 原则 | 说明 |
|------|------|
| **从左到右** | 用户请求从左到右，数据从右到左（回包），符合阅读习惯 |
| **分层清晰** | 客户端 → 接入层 → 逻辑层 → 数据层，每层用矩形框括起来 |
| **标注关键数字** | 在组件旁标注 QPS / 数据量（如 "Redis: 10万 QPS"） |
| **画出数据流** | 用箭头表示请求路径和响应路径，标注 1, 2, 3 顺序 |
| **标注失败的路径** | "Redis 挂了怎么办？" → 画一条虚线箭头到降级方案 |

**架构图模板：**

```
┌───────┐      ┌──────────┐      ┌────────┐      ┌──────────┐
│ 客户端  │ ──→  │ 接入层    │ ──→  │ 逻辑层   │ ──→  │ 数据层    │
│ (App)  │ ←──  │ Nginx    │ ←──  │ API     │ ←──  │ MySQL    │
└───────┘      │ + LVS    │      │ Service │      │ + Redis  │
               └──────────┘      └────────┘      │ + MQ     │
                                                  └──────────┘
```

### 6.4 主动展示 Trade-off 思考

**面试官最在意的信号**——你是否有"架构师思维"。

| 好的信号（加分） | 差的信号（减分） |
|-----------------|-----------------|
| "这个方案选 Kafka 而不是 RabbitMQ 因为……" | "我们用 Kafka，因为大家都用" |
| "如果 QPS 到 10 万，瓶颈会在数据库连接数……" | "我们系统能支持 100 万 QPS"（无依据） |
| "这里我选最终一致，因为业务允许 3 秒内的不一致，但换来了 10 倍的吞吐提升" | "我们需要强一致"（不问场景直接下结论） |
| "这个方案的缺点是运维成本高，需要专门的 DBA 维护……" | 只说优点不说缺点 |
| "我们可以逐步演进：第一阶段做 A，第二阶段做 B……" | 一上来就设计一个"完美的"复杂系统 |

**Trade-off 分析常用句式：**

- "这里有两个方案：方案 A 是……优点……缺点……；方案 B 是……优点……缺点……。在目前的场景下，我倾向于选 A，因为……"
- "如果业务进一步发展，到下一阶段后，这个选择的弊端会显现，届时可以切换到方案 B"
- "这个方案在可用性和一致性之间做了取舍——我们选择了可用性优先，代价是极端情况下用户可能看到旧数据，但通过补偿机制可以在 3 秒内修复"

### 面试常见"一句话"速答

| 提问 | 要点 |
|------|------|
