## 1. 架构基础

### 1.1 微服务的定义与本质

微服务架构是一种将单一应用程序划分为一组小服务的架构风格，每个服务运行在自己的进程中，通过轻量级通信机制（通常是 HTTP/REST 或消息队列）协作。

**本质特征：**
- **围绕业务能力组织**：每个服务对应一个业务能力，而非技术分层
- **独立部署**：每个服务可独立构建、部署、扩缩
- **去中心化治理**：技术栈异构，各自选择最合适的语言/数据库
- **去中心化数据管理**：每个服务拥有自己的数据存储

### 1.2 横向对比：单体 vs SOA vs 微服务

```
+------------------+------------------+------------------+------------------+
|      维度        |      单体        |       SOA        |     微服务       |
+------------------+------------------+------------------+------------------+
| 服务粒度          | 单进程单部署     | 粗粒度业务服务   | 细粒度业务服务   |
| 通信方式          | 方法调用         | ESB/SOAP/WS      | REST/gRPC/消息   |
| 数据存储          | 单数据库         | 共享数据库       | 每服务独立数据库  |
| 部署方式          | 单 WAR/JAR       | 企业服务总线     | 容器化独立部署    |
| 技术异构          | 同构             | 有限异构         | 完全异构          |
| 治理              | 集中式           | ESB 集中式       | 去中心化自治      |
| 伸缩性            | 垂直伸缩         | 有限水平伸缩     | 精细水平伸缩      |
| 测试复杂度        | 低               | 中               | 高（契约测试）    |
| 运维复杂度        | 低               | 中               | 高                |
+------------------+------------------+------------------+------------------+
```

**高频面试题：**

> Q：微服务和 SOA 的根本区别是什么？
> A：微服务不是 SOA 的简单升级版，核心区别在于：
> 1. **粒度**：SOA 强调粗粒度业务服务复用，微服务追求极致的单一职责
> 2. **治理**：SOA 依赖 ESB 集中治理，微服务去中心化自治
> 3. **数据**：SOA 倾向于共享数据库，微服务强制每个服务独享数据库
> 4. **通信**：SOA 的 ESB 成为瓶颈和单点，微服务更轻量的点对点通信
> 5. **复杂度转移**：SOA 将复杂度集中在 ESB/编排层，微服务将复杂度分散到服务自身和基础设施层

### 1.3 服务拆分原则

#### DDD 限界上下文（Bounded Context）

DDD 的限界上下文是微服务拆分的最核心方法论。每个限界上下文对应一个微服务。

```
+---------------------------+     +---------------------------+
|  上下文 A：订单上下文      |     |  上下文 B：库存上下文       |
|                           |     |                           |
|  +------------------+    |     |  +------------------+    |
|  | Order            |    |     |  | Inventory        |    |
|  | - orderId        |    |     |  | - sku            |    |
|  | - items          |    |     |  | - quantity       |    |
|  | - totalAmount    |    |     |  | - warehouseLoc   |    |
|  | - status         |    |     |  | - reservedQty    |    |
|  +------------------+    |     |  +------------------+    |
|  | 领域事件: OrderCreated |   |     | 领域事件: StockReduced  |
|  | 领域服务: OrderService |   |     | 领域服务: InventoryService|
|  +------------------+    |     |  +------------------+    |
+---------------------------+     +---------------------------+
          |                                    |
          |        通过消息/事件通信             |
          +------------------------------------+
```

**拆分原则：**

1. **单一职责原则**：一个服务只做一件事，且做好
2. **闭包原则**：服务内的变更不应影响外部
3. **数据主权原则**：谁拥有数据谁负责
4. **团队适配原则**（康威定律）：服务边界 = 团队沟通边界
5. **事务边界原则**：高频强一致性操作尽量在一个服务内
6. **变更频率对齐**：同一节奏变更的业务放在一起
7. **资源独立性**：能够独立伸缩和部署

#### 康威定律（Conway's Law）

> "Organizations which design systems are constrained to produce designs which are copies of the communication structures of these organizations."
>
> 任何设计系统的组织，最终产生的设计方案等价于组织的沟通结构。

**架构启示：**
- 如果团队按前端/后端/数据库组织，你得到的会是分层架构
- 如果团队按订单/支付/用户组织，你得到的会是微服务架构
- 服务边界与团队组织结构对齐，否则沟通成本将急剧上升
- 逆康威定律：主动调整团队结构来驱动架构演进

#### 拆分决策树

```
开始拆分
│
├─ 业务逻辑是否可以在单一进程内清晰地划分界限？
│   ├─ 是 → DDD 事件风暴 → 识别限界上下文
│   │   └─ 每个上下文 = 候选微服务
│   ├─ 否 → 先梳理业务领域，再引入 DDD
│   └─ 不清晰 → 暂不拆分，重构统一语言（Ubiquitous Language）
│
├─ 候选服务之间调用频率是否极高（>1000次/秒）？
│   ├─ 是 → 考虑合并或使用异步解耦
│   └─ 否 → 可以拆分
│
├─ 候选服务之间是否需要强事务一致性？
│   ├─ 是 → 合并为一个服务，或引入 Saga/CQRS
│   └─ 否 → 可以拆分
│
├─ 候选服务的变更频率是否差异很大？
│   ├─ 是 → 拆分，让高频变更不影响低频服务
│   └─ 否 → 可以合并
│
├─ 候选服务能否独立部署和伸缩？
│   ├─ 是 → 拆分
│   └─ 否 → 先解决部署依赖再拆分
│
└─ 总结：先粗后细，逐步演进，避免过度拆分
```

### 1.4 生产踩坑经验

**踩坑 1：过度拆分灾难**
> 某团队将 6 人维护的中型系统拆成 20+ 微服务，结果每个服务代码仅几百行，但需要维护 20 个仓库、20 条 CI 流水线、20 个部署单元、N 个服务间调用。交付效率不升反降，从 1 周降至 1 个月。
> **教训**：团队规模 * 2 的服务数量上限法则（6人团队最多 12 个服务），过多微服务需要更成熟的基础设施和更大的团队支撑。

**踩坑 2：拆分后分布式复杂性被低估**
> 单体中的本地方法调用变成了网络调用，需要处理超时、重试、熔断、幂等、序列化、网络分区等问题。一个简单的 "查询订单同时展示用户信息" 在单体中是 JOIN，在微服务中变成了两次 RPC + 数据组装。
> **教训**：拆分前必须建立服务治理基础设施（注册中心、配置中心、熔断器、链路追踪），否则拆分后的系统比单体更脆弱。

**踩坑 3：共享数据库的诱惑**
> 为了"方便"，多个微服务共享同一个数据库，导致服务间耦合、数据库成为瓶颈、Schema 变更互相影响。
> **教训**：数据独占是微服务的核心原则，如果做不到就不要开始微服务改造。

**踩坑 4：忽略团队能力建设**
> 团队从未接触过 Docker/K8s、分布式事务、服务网格等，直接上马微服务。
> **教训**：微服务架构需要 DevOps 文化 + 容器化能力 + 分布式系统知识 + 全链路可观测性的配合，团队需要逐步建设这些能力，而非一蹴而就。

## 2. 通信机制

### 2.1 同步通信：REST vs gRPC

```
+-----------------------------+------------------------------------------+
|            REST             |                gRPC                     |
+-----------------------------+------------------------------------------+
| HTTP/1.1（通常）            | HTTP/2（必须）                           |
| 文本协议（JSON/XML）        | 二进制协议（Protobuf）                    |
| 人类可读                    | 机器可读，不可直接查看                    |
| 客户端-服务端耦合松散       | 强类型契约（.proto 文件）                 |
| 无内置流式支持              | 支持 server streaming / client streaming  |
| 浏览器原生支持              | 需要 gRPC-web 网关                        |
| 生态系统成熟                | 相对较新但生态快速成熟                     |
| 性能：中等                  | 性能：高（快 7-10 倍）                    |
| 工具：curl, Postman        | 工具：grpcurl, BloomRPC                   |
+-----------------------------+------------------------------------------+

          REST 调用流                          gRPC 调用流
  +--------+   HTTP/JSON    +--------+   +--------+   HTTP/2 Proto    +--------+
  | Client | ──────────────→| Server |   | Client | ─────────────────→| Server |
  +--------+                +--------+   +--------+                   +--------+
       ↑                        ↑            ↑                            ↑
       |   JSON 序列化/反序列化   |            |   Protobuf 编码/解码       |
       |   无类型检查            |            |   .proto 强类型契约         |
       |   运行时发现错误        |            |   编译期类型检查             |
       +------------------------+            +----------------------------+
```

**高频面试题：**

> Q：在微服务内部通信中，为什么 gRPC 比 REST 更适合？
> A：
> 1. **性能**：Protobuf 序列化比 JSON 小 3-10 倍，快 20-100 倍
> 2. **双向流**：gRPC 基于 HTTP/2 支持 Server Push、双向流，适合实时推送场景
> 3. **强类型**：.proto 文件既是接口定义又是文档，生成代码保证类型安全
> 4. **多语言**：自动生成 10+ 语言的客户端/服务端代码
> 5. **延迟**：HTTP/2 多路复用避免 HOL（Head-of-Line）阻塞
>
> 但 gRPC 不适合浏览器直接调用，网关层仍需 REST/graphQL 对外暴露。

> Q：Go 和 PHP 在微服务通信中各自的优劣势？
> A：
> **Go 微服务优势**：
> - 原生 gRPC 支持（protoc-gen-go），轻量 goroutine 处理高并发
> - 单一二进制部署，容器镜像小（<20MB）
> - 零依赖运行时，K8s 亲和性极佳
> - 内建 pprof/pprof 性能分析，Go net/http 可观测性好
>
> **PHP 微服务挑战与对策**：
> - PHP 传统上同步阻塞，Swoole/Fiber 后可协程化
> - gRPC 支持通过 protoc-gen-php-grpc 或 Swoole 扩展
> - 更适合 API 网关层、聚合层（BFF），对性能敏感的内核服务建议 Go
> - 建议采用 Go 写内核服务 + PHP 写 BFF/编排层的混合架构

### 2.2 异步通信：消息队列

```
                   +-----------------+
                   |   消息中间件      |
                   | (Kafka/RocketMQ/ |
                   |  RabbitMQ/NSQ)   |
                   +--------+--------+
                            |
            +---------------+---------------+
            |               |               |
       +----v----+    +----v----+    +-----v-----+
       | Producer|    | Producer|    | Consumer  |
       | (服务A)  |    | (服务B)  |    | (服务C)   |
       +---------+    +---------+    +-----------+

    消息模式对比:
    +--------------+---------------+--------------+
    |              |    RabbitMQ   |    Kafka     |
    +--------------+---------------+--------------+
    | 设计哲学       | 消息路由      | 日志存储      |
    | 吞吐量         | 万级/秒       | 百万级/秒     |
    | 延迟           | 微秒级        | 毫秒级        |
    | 消息顺序        | 不保证       | 分区内有序     |
    | 消息回溯        | 不支持       | 支持(offset)  |
    | 路由灵活性      | 高(Exchange) | 低(topic)    |
    | 典型场景        | 任务队列, RPC| 事件流, 日志  |
    | PHP 支持        | amqp扩展    | librdkafka   |
    | Go 支持          | amqp库      | sarama/confluent|
    +--------------+---------------+--------------+

    事件驱动架构：
    +-------------+          +------------+
    | 订单服务     | 发布事件  | 消息队列    |
    | (OrderCreated)|────────→| (Event Bus) |
    +-------------+          +-----+------+
                                    |
                     +--------------+--------------+
                     |              |              |
                +----v----+   +----v----+   +----v----+
                | 库存服务  |   | 通知服务  |   | 发票服务  |
                +---------+   +---------+   +---------+
```

**高频面试题：**

> Q：Kafka 如何保证消息不丢失？
> A：从三个层面保证：
> 1. **Producer 端**：acks=all（等待所有副本确认）+ enable.idempotence=true（幂等）+ retries>0
> 2. **Broker 端**：replication.factor>=3, min.insync.replicas=2, unclean.leader.election=false
> 3. **Consumer 端**：enable.auto.commit=false -> 处理完成后手动提交 offset
>
> 实际上不存在绝对的"不丢失"，而是把丢失概率压到工程可接受范围。

> Q：Swoole/Fiber 下的 PHP 如何消费 Kafka？
> A：PHP 传统消费模式是单进程阻塞式，Swoole 开启多个 Coroutine 并行消费，用 Channel 控制并发度。Fiber 在 PHP 8.1+ 提供协程支持。
>
> ```php
> // Swoole + Kafka 协程消费示例（伪代码）
> $consumer = new \longlang\phpkafka\Consumer(...);
> while (true) {
>     $messages = $consumer->consume(10000);
>     foreach ($messages as $msg) {
>         go(function () use ($msg) {
>             // 协程处理
>         });
>     }
> }
> ```

### 2.3 Protocol Buffers vs JSON

```
+---------------------------+---------------------------+
|        Protocol Buffers   |          JSON            |
+---------------------------+---------------------------+
| 二进制格式，不可读         | 文本格式，可读             |
| 需要 .proto 定义           | 无需预定义 Schema         |
| 编译期类型检查             | 运行时解析                 |
| 编码后体积小（约 JSON 1/3）| 体积大，含字段名           |
| 编码/解码快（C++ 原生）     | 解析慢（反射/动态）        |
| 向后兼容通过字段编号       | 字段增删兼容               |
| 不适合浏览器直用           | 浏览器原生                 |
| 强类型、无动态结构         | 灵活，可有动态 key         |
+---------------------------+---------------------------+

.proto 示例:
syntax = "proto3";
package order;

service OrderService {
    rpc CreateOrder (CreateOrderRequest) returns (OrderResponse);
    rpc GetOrder (GetOrderRequest) returns (OrderResponse);
    rpc StreamOrders (OrderFilter) returns (stream OrderResponse);
}

message CreateOrderRequest {
    string user_id = 1;
    repeated OrderItem items = 2;
    string coupon_id = 3;          // optional
}

message OrderItem {
    string sku = 1;
    int32 quantity = 2;
    int64 price = 3;               // 单位：分
}

message OrderResponse {
    string order_id = 1;
    int64 total_amount = 2;
    OrderStatus status = 3;
    int64 created_at = 4;
}

enum OrderStatus {
    ORDER_STATUS_UNKNOWN = 0;
    ORDER_STATUS_PENDING = 1;
    ORDER_STATUS_PAID = 2;
    ORDER_STATUS_SHIPPED = 3;
}
```

### 2.4 服务调用链追踪

```
请求入口 (API 网关)
    │
    ├─ Trace ID: a1b2c3d4e5f6
    │  Span ID: root (Parent Span)
    │
    ├─ 服务A：用户服务 ──────────────── 耗时 50ms
    │   │  Span ID: span-001 (Parent: root)
    │   │
    │   ├─ 缓存 Redis ── 耗时 10ms
    │   │   Span ID: span-002 (Parent: span-001)
    │   │
    │   └─ 数据库 MySQL ── 耗时 30ms
    │       Span ID: span-003 (Parent: span-001)
    │
    ├─ 服务B：订单服务 ──────────────── 耗时 120ms
    │   │  Span ID: span-004 (Parent: root)
    │   │
    │   ├─ gRPC 调用库存服务 ── 耗时 60ms
    │   │   Span ID: span-005 (Parent: span-004)
    │   │
    │   ├─ Kafka 发送消息 ── 耗时 5ms
    │   │   Span ID: span-006 (Parent: span-004)
    │   │
    │   └─ 数据库 MongoDB ── 耗时 40ms
    │       Span ID: span-007 (Parent: span-004)
    │
    └─ 总耗时: 170ms

    调用链数据结构（OpenTelemetry 格式）:
    {
      "traceId": "a1b2c3d4e5f6",
      "spans": [
        {
          "spanId": "span-001",
          "parentSpanId": "root",
          "name": "user-service.GetUser",
          "kind": "SERVER",
          "startTime": 1710000000000000,
          "endTime": 1710000000050000,
          "status": {"code": "OK"},
          "attributes": {
            "http.method": "POST",
            "http.url": "/v1/users/123"
          }
        }
      ]
    }

    链路上下文传播方式:
    ┌──────────────────────────────────────────────────────┐
    │                    HTTP 头传播                       │
    │  traceparent: 00-a1b2c3d4e5f6...-span-001-01        │
    │  tracestate: vendor1=value1,vendor2=value2           │
    │  (W3C Trace Context 标准)                            │
    └──────────────────────────────────────────────────────┘

    在 gRPC 中通过 Metadata 传播:
    metadata → "x-trace-id", "x-span-id", "x-parent-span-id"

    在 Kafka 消息头中传播:
    message.headers → "x-trace-id", "x-span-id"
```

**高频面试题：**

> Q：你的系统中有没有出现过"幽灵调用"——服务间调用链路在监控系统中不完整？
> A：常见的链路断裂原因：
> 1. **异步消息**：Producer 发送消息到 Kafka/RabbitMQ，Consumer 消费时丢失了 Trace Context（消息头未透传）
> 2. **线程池切换**：主线程的 TraceContext 没有传递给线程池中的子线程
> 3. **网关未透传**：API 网关没有将 x-trace-id 透传到下游
> 4. **gRPC 拦截器未配置**：客户端/服务端拦截器中未注入/提取 Metadata
> 5. **PHP 请求-响应模型**：每个请求结束后 TraceContext 被销毁，异步/批量任务需要手动 createSpan

> Q：PHP 如何实现链路追踪？
> A：PHP 在 FastCGI 模式下每个请求生命周期独立，链追踪通过：
> 1. **Swoole Tracker** / **OpenTelemetry PHP SDK** 在请求开始时创建 Root Span
> 2. 使用 curl handler 或 Guzzle 中间件自动注入 traceparent 头
> 3. PDO/Redis 等客户端通过 Hook 注入子 Span
> 4. 请求结束自动上报到 Jaeger/Zipkin
> 5. Go 相对简单，context.Context 天然支持跨 goroutine 传播

### 2.5 生产踩坑经验

**踩坑 1：gRPC 连接未复用导致连接爆炸**
> 每次 RPC 调用都创建新连接，导致服务端连接数飙升，触发 TCP 端口耗尽。
> **解决**：使用 gRPC 连接池（go-grpc-pool / 长连接复用），控制最大连接数。

**踩坑 2：Kafka 消费顺序问题**
> 订单事件先发后到，导致库存扣减先于订单创建执行，产生脏数据。
> **解决**：订单 ID 哈希分区确保同一订单的事件进入同一分区；消费端使用状态机检查前置状态。

**踩坑 3：Protobuf 兼容性事故**
> 生产环境一个服务升级了 .proto 文件，删除了一个废弃字段编号 7，但另一个服务仍在旧版本使用该编号发送数据，触发 Protobuf 解析异常。
> **教训**：被删除的字段编号必须 reserved 禁止复用：
> ```protobuf
> message Order {
>   reserved 7;       // 字段编号 7 已废弃
>   reserved "old_field";  // 字段名已废弃
> }
> ```

**踩坑 4：PHP 微服务中的连接泄漏**
> PHP-FPM 模式下每个请求创建 gRPC 连接，但进程不退出则连接不释放，导致 gRPC Server 端积累大量 TIME_WAIT。
> **解决**：PHP-FPM 最大请求数调低、开启 Swoole 常驻内存模式、使用连接池。

**高频面试题：**

> Q：为什么需要注册中心？直接用 DNS 做服务发现不行吗？
> A：传统 DNS 的局限：
> 1. **缓存问题**：DNS TTL 期间不可用节点无法被摘除
> 2. **健康检查缺失**：DNS 无法感知服务是否存活
> 3. **端口支持**：DNS SRV 记录支持度差，多数客户端只解析 A 记录
> 4. **更新延迟**：DNS 传播延迟秒级~分钟级
> 5. **无负载均衡策略**：DNS 轮询无法感知权重/熔断
>
> 注册中心提供实时健康检查、即时剔除、权重路由、区域亲和性等能力，是生产级的服务发现方案。

> Q：Consul 客户端发现与 K8s 服务发现的区别是什么？
> A：
> - **Consul 客户端发现**：消费者从注册中心获取 Provider 地址列表，由客户端实现负载均衡（如 Go 的 grpc-balancer、PHP 的 consul-php-SDK）
> - **K8s Service 发现**：通过 kube-proxy + iptables/IPVS 实现透明代理，Pod 通过 Service DNS 访问，Pod 重启 IP 变化不影响调用方
> - **本质区别**：Consul 是应用层服务发现，K8s Service 是网络层透明代理
> - **最佳实践**：K8s 内使用 K8s Service，跨集群/跨网络使用 Consul/Nacos

**高频面试题：**

> Q：Sentinel 的限流为什么比 Hystrix 更适合生产环境？
> A：
> 1. **限流能力**：Hystrix 定位是熔断器，限流能力薄弱；Sentinel 内置丰富的流量控制规则（QPS、线程数、关联规则、链路限流）
> 2. **实时监控**：Sentinel 控制台提供实时 QPS 曲线、RT 分布、拒绝统计
> 3. **动态规则**：结合 Nacos/Apollo 可实现规则热更新，无需重启
> 4. **适应云原生**：Sentinel Go 版本原生支持 K8s，Hystrix 已停止维护
>
> 但 Hystrix 的线程池隔离机制在某些场景仍是优势——当你需要完全隔离调用方和被调用方的线程资源时。

> Q：PHP 场景下如何实现熔断降级？
> A：PHP 每次请求在独立的 FPM 进程中处理，缺乏长驻内存状态，实现熔断器需要外部存储：
> 1. **Redis 滑动窗口**：在每个请求中统计成功/失败计数存储在 Redis Sorted Set 中
> 2. **熔断器库**：使用 `pkg/errors` + 自己维护的熔断器状态（如 `hystrix-php`）
> 3. **Swoole 常驻内存**：在 Swoole Worker 中维护内存熔断器状态，但跨进程共享需 Redis
> 4. **Sidecar 模式**：通过 Service Mesh（Istio）在网络层实现熔断，对 PHP 完全透明

### 3.5 生产踩坑经验

**踩坑 1：注册中心脑裂，服务大规模雪崩**
> Consul Server 集群网络抖动触发 Leader 选举，期间服务不可注册/发现，新 Pod 无法接入流量，旧 Pod 被误判为不健康切走流量。
> **解决**：Consul Server 至少 3 节点（奇数）部署在不同可用区；配置 `bootstrap_expect=3`；对注册中心进行独立网络隔离。

**踩坑 2：配置中心变更未通知到所有实例**
> Apollo 配置灰度推送，灰度列表漏了部分实例，导致生产环境部分配置不一致。
> **解决**：配置变更后通过控制台查看推送状态，确认所有实例已收到更新；配置灰度需谨慎，先灰度一台再全量。

**踩坑 3：Sentinel 规则配置过高限死了正常流量**
> 某团队为保护下游数据库，在 Sentinel 中设置了 100 QPS 限流。上线初期流量小没问题，做活动流量飙升后大量请求被限流，业务受损。
> **教训**：限流阈值需要基于压测数据设定，并配合弹性伸缩而非固定阈值；限流触发时要有友好的降级响应而非直接 429。

**踩坑 4：熔断器恢复抖动（Thundering Herd）**
> 熔断器 OPEN 状态结束后进入 HALF_OPEN，放行大量试探请求，此时下游仍不稳定，导致试探请求集体失败，熔断器快速回到 OPEN，加剧抖动。
> **解决**：HALF_OPEN 试探请求数量应从 1 开始逐步放开（如：1→2→4→8 指数回退），且试探请求设置独立的超时时间。

**踩坑 5：跨语言服务治理的兼容性问题**
> PHP 服务使用 consul-php 注册到 Consul，Go 服务使用 go-micro 的 consul registry，两边的健康检查格式不兼容，Go 服务认为 PHP 服务不可用。
> **教训**：多语言环境下需要统一 Service Registry 的元数据格式和健康检查接口，建议定义组织级的注册规范。

**高频面试题：**

> Q：为什么不推荐所有微服务共享一个数据库？
> A：共享数据库的问题：
> 1. **Schema 耦合**：一个服务的表结构变更会影响其他服务
> 2. **伸缩性冲突**：高频写入和复杂分析查询争抢数据库资源
> 3. **安全边界模糊**：服务 B 可以直接查服务 A 的表，绕过服务 A 的校验逻辑
> 4. **独立部署困难**：数据库 Schema 变更需要协调所有服务
> 5. **技术选型受限**：无法为不同服务选择最适合的存储（如订单用 MySQL，日志用 ES）
>
> 解耦方法：强制数据访问只能通过服务 API，不允许直接访问其他服务的数据库。

### 4.2 Saga 分布式事务

```
    Saga 模式: 两阶段补偿型长事务

    ┌─────────┐     ┌─────────┐     ┌─────────┐     ┌─────────┐
    │ 订单服务  │     │ 库存服务  │     │ 支付服务  │     │ 物流服务  │
    │ 创建订单  │────→│ 扣减库存  │────→│ 发起支付  │────→│ 创建物流  │
    └────┬─────┘     └────┬─────┘     └────┬─────┘     └────┬─────┘
         │                │                │                │
         │  失败补偿       │  失败补偿       │  失败补偿       │
         ↓                ↓                ↓                ↓
    ┌─────────┐     ┌─────────┐     ┌─────────┐
    │ 取消订单  │     │ 加回库存  │     │ 发起退款  │
    └─────────┘     └─────────┘     └─────────┘

    Saga 编排 (Choreography) vs 协调 (Orchestration):

    Choreography（事件驱动编排）:
    订单服务 ──发布OrderCreated──→ 库存服务 ──发布StockReduced──→ 支付服务
        ↑                                                         │
        └───────────────────── 发布PaymentFailed ────────────────┘

    Orchestration（中央协调器）:
                         ┌───────────────────────┐
                         │   Saga Orchestrator    │
                         │ (状态机 + 补偿逻辑)     │
                         └───┬───┬───┬───┬──────┘
                             │   │   │   │
                  ┌──────────┘   │   │   └────────────┐
                  │              │   │                │
             ┌────v──┐     ┌────v──┐  ┌────v──┐  ┌────v──┐
             │订单服务 │     │库存服务│  │支付服务│  │物流服务│
             └───────┘     └───────┘  └───────┘  └───────┘

    Saga 状态机示例:
    +-------+      payment     +------+    ship     +--------+
    | Pending |──────────────→│ Paid │────────────→│ Shipped│
    +-------+                 +------+             +--------+
        │                         │
        │ cancel                   │ refund
        ↓                         ↓
    +-------+                +--------+
    | Cancelled|             | Refunded|
    +-------+                +--------+
```

**高频面试题：**

> Q：Saga 和传统两阶段提交（2PC）的区别是什么？
> A：
>
> | 特性 | 2PC (XA) | Saga |
> |------|----------|------|
> | 一致性 | 强一致性 | 最终一致性 |
> | 锁 | 持有锁直到全局提交/回滚 | 无长锁，事务之间不阻塞 |
> | 性能 | 低（锁竞争 + 多次网络往返） | 高（无锁） |
> | 可用性 | 协调器单点，阻塞式 | 无阻塞，协调器故障可重试 |
> | 实现难度 | 框架支持（Atomikos, Seata） | 应用层编写补偿逻辑 |
> | 数据可见性 | 事务完成后才可见 | 中间状态可见（脏读） |
> | 适用场景 | 短事务、强一致性要求 | 长事务、高可用要求 |
>
> 微服务中几乎不使用 2PC，Saga 是主流方案。

> Q：Seata 的 AT 模式和 TCC 模式有什么区别？
> A：
> **AT 模式（Auto Transaction）**：
> - 侵入性低：通过代理数据源自动生成前后镜像（UNDO_LOG）
> - 两阶段：一阶段提交本地事务 + 二阶段异步清理/回滚
> - 适用：单库操作，对并发要求不高的场景
> - 问题：全局锁在大并发下成为瓶颈
>
> **TCC 模式（Try-Confirm-Cancel）**：
> - 侵入性高：业务需要实现 try/confirm/cancel 三个接口
> - 无锁：try 阶段不锁资源
> - 适用：高并发、资源预留场景（如库存预占、账户冻结）
> - 优势：性能优于 AT，资源利用率高
>
> 生产经验：90% 场景用 Saga（事件驱动）+ 10% 关键资金场景用 TCC。

> Q：PHP 如何实现 Saga？
> A：
> 1. **数据库本地事务 + 消息补偿表**：
>    - 先写业务数据 + 事务消息状态到本地表
>    - 后台进程轮询发送消息到 Kafka/RabbitMQ
>    - 消费方失败后写入补偿事件，生产者监听补偿事件执行回滚
> 2. **状态机框架**：
>    - 用 PHP Workflow 库（如 temporal-php、temporal-sdk-php）编排 Saga
> 3. **Seata-php-client**：
>    - 通过 HTTP 与 Seata Server 通信，PHP 实现 TCC 模式

### 4.3 CQRS 与 Event Sourcing

```
    CQRS: 命令查询职责分离

    传统 CRUD:                          CQRS:
    +────────────+                     +────────────+
    |   Order    |                     |   Command   |
    |  CRUD 混合  |                     │   (写入)    │
    |  读/写同模型 |                     │   Order    │
    +────────────+                     │ Service    │
                                       └─────┬──────┘
                                             │ 写 (INSERT/UPDATE)
                                             ↓
                                       ┌────────────┐
                                       │  写模型 DB  │
                                       │  (3NF 范式)  │
                                       └─────┬──────┘
                                             │ 同步 (事件/CDC)
                                             ↓
                                       ┌────────────┐
                                       │  读模型 DB  │
                                       │ (反范式/宽表)│
                                       └─────┬──────┘
                                             │
                                       ┌─────┴──────┐
                                       │   Query     │
                                       │   (读取)    │
                                       │   Order     │
                                       │   Query     │
                                       └────────────┘

    Event Sourcing:
    不存储当前状态，只存储事件流

    传统存储:                          Event Sourcing:
    Order(id=100, status="paid")      Event Stream for Order#100:
                                      1. OrderCreated { id:100, amount: 5000 }
                                      2. PaymentCompleted { id:100, method: "alipay" }
                                      3. OrderShipped { id:100, carrier: "SF" }
                                      4. OrderDelivered { id:100, time: "2024-03-01" }

                                      当前状态 = 重放所有事件
                                      Event Store (EventStoreDB / Kafka)
                                      ↓
                                      投影 (Projection) → 物化视图
                                      ↓
                                      读模型 (用于查询)
```

**高频面试题：**

> Q：什么时候应该使用 CQRS？什么时候不应该？
> A：
> **使用 CQRS 的场景**：
> - 读/写负载差异巨大（如写少读多，或写多读少）
> - 复杂查询和写模型无法共用同一数据模型
> - 团队需要为读和写分别优化数据库（写用 MySQL OLTP，读用 ES/ClickHouse OLAP）
> - 读写伸缩需求不同（写库 2 副本，读库 10 副本）
>
> **不要使用 CQRS 的场景**：
> - 简单的 CRUD 系统（CQRS 增加复杂度但无收益）
> - 一致性要求高的场景（CQRS 默认最终一致性）
> - 团队不熟悉事件驱动架构
>
> CQRS 不是银弹，大多数系统不需要 CQRS。引入前需要回答：CQRS 带来的复杂性是否值得？

> Q：Event Sourcing 如何处理 Schema 演化（事件结构变更）？
> A：
> 1. **向上兼容**：新增字段默认值，老事件回放时使用默认值
> 2. **Upcaster**：重写事件历史，将旧格式事件转换为新格式
>    ```php
>    // 将 v1 事件升级为 v2
>    class OrderV1Upcaster implements Upcaster {
>        public function canUpcast(DomainEvent $event): bool {
>            return $event instanceof OrderCreated && !isset($event->currency);
>        }
>        public function upcast(DomainEvent $event): DomainEvent {
>            // 添加 currency 默认字段
>            return $event->withAddedMetadata('currency', 'CNY');
>        }
>    }
>    ```
> 3. **事件版本化**：每个事件结构带 version 字段
> 4. **多版本共存**：新旧消费者各自处理自己能理解的事件版本

### 4.4 CAP 理论与 BASE 柔性事务

```
    CAP 定理:
    ┌───────────────────────────┐
    │       一致性 (C)           │
    │ 所有节点在同一时刻看到      │
    │ 相同数据                   │
    └──────┬────────────┬───────┘
           │            │
           │            │
    ┌──────v──────┐ ┌──v────────┐
    │ 可用性 (A)  │  │ 分区容错 (P) │
    │ 每次请求    │  │ 网络分区后   │
    │ 都能得到响应 │  │ 系统仍工作   │
    └─────────────┘  └───────────┘

    CP 系统（放弃 A）：etcd, ZooKeeper, Consul
    AP 系统（放弃 C）：Cassandra, DynamoDB, Eureka
    CA 系统（放弃 P）：单机数据库（在网络分区下不可用）

    在分布式系统中，P 是必选项，所以实际是在 C 和 A 之间权衡。

    BASE 柔性事务模型:
    ┌──────────────────────────────────────────────────┐
    │  Basically Available  基本可用                      │
    │      系统允许部分功能在故障时降级响应                    │
    │                                                    │
    │  Soft State          软状态                         │
    │      数据允许中间状态，不强求实时一致                    │
    │                                                    │
    │  Eventually          最终一致性                      │
    │  Consistent          经过一段时间后数据会达到一致        │
    └──────────────────────────────────────────────────┘

    一致性强弱对比:
    ┌───────────────┬─────────────────┬────────────────┐
    │  强一致性      │   最终一致性      │   弱一致性     │
    │               │                 │                │
    │ - 2PC/XA      │ - Saga          │ - DNS          │
    │ - 单机事务     │ - 异步复制       │ - CDN 缓存     │
    │ - ZK 线性一致  │ - CQRS          │ - 本地缓存     │
    │ - 分布式锁     │ - Eventual read  │               │
    └───────────────┴─────────────────┴────────────────┘

    数据一致性保证的几种模式:
    +------------------+---------------------------------------------+
    |      模式        |                    说明                     |
    +------------------+---------------------------------------------+
    | 幂等性           | 相同操作执行多次结果相同，通过唯一请求号去重   |
    | 补偿事务          | 正向操作失败后执行反向操作回滚                |
    | 重试机制          | 临时失败自动重试，配合指数退避                 |
    | 消息表            | 本地事务 + 消息表保证消息不丢                |
    | 版本号            | 乐观锁，更新时检查版本号防止覆盖              |
    | TCC              | 三个阶段：Try (预留) → Confirm/Cancel       |
    +------------------+---------------------------------------------+
```

**高频面试题：**

> Q：如何设计一个"严格最终一致性"的系统？请描述方案。
> A：
> **方案**：本地消息表 + 消息队列 + 幂等消费
>
> ```
> 1. 服务 A 开始本地事务
> 2. 写业务数据 + 写消息表（状态: INIT）
> 3. 提交本地事务
> 4. 后台进程轮询消息表 → 发送到 Kafka
> 5. Kafka 保证至少投递一次（At Least Once）
> 6. 服务 B 消费消息，执行业务逻辑
> 7. 执行成功 → 向 Kafka 发送确认（Commit Offset）
> 8. 执行失败 → 记录重试次数，达到上限后写入死信队列
> 9. 服务 A 监听死信队列，发起人工介入或补偿事务
>
> 幂等保证：
> - 每个消息携带全局唯一 ID（如 UUID）
> - 消费方记录已处理的消息 ID（去重表）
> - 重复消息被自动过滤
> ```
>
> **保障点**：
> - 本地消息表和业务数据在同一事务内 = 消息不会丢
> - 消息持久化到 Kafka = 不会丢
> - Consumer 幂等 = 不会重复处理
> - 死信队列 + 告警 = 不会永久卡死

### 4.5 生产踩坑经验

**踩坑 1：Saga 补偿逻辑不幂等导致数据错误**
> 某支付服务收到退款指令后执行退款，但由于网络超时，退款请求被重试两次，导致同一笔订单被退了三次。
> **解决**：补偿操作必须幂等，加上全局事务 ID 去重。

**踩坑 2：CQRS 读库延迟导致用户体验问题**
> 用户支付成功后跳转订单详情页，读库仍在同步，显示"未支付"，用户多次点击"支付"。
> **解决**：读库同步时间纳入 SLA 监控，核心场景走强一致读（直接查写库），或在写入后强制刷新读库缓存。

**踩坑 3：Event Sourcing 事件风暴——事件量爆发**
> 某物联网平台使用 Event Sourcing 存储设备状态变更事件，每天产生 5 亿事件，Event Store 存储膨胀到 TB 级别，事件回放耗时数小时。
> **解决**：引入 Snapshot（快照）机制——每 N 个事件生成一个状态快照，回放时从最近快照开始；对历史事件启用归档策略（超过 90 天的冷数据存到 S3）。

**踩坑 4：分布式事务跨越多团队——协调成本高**
> 一次订单流程涉及订单服务（A 团队）、支付服务（B 团队）、积分服务（C 团队）。线上出现订单成功但积分发放失败的情况，三个团队互相推诿。
> **教训**：Saga 协调器需要明确拥有者，SLA 明确每个参与方的超时和重试策略；建议引入一个独立的"事务编排服务"（Orchestrator Service）由平台团队统一维护。

**高频面试题：**

> Q：API 网关和 BFF 的区别是什么？什么场景下需要两者同时存在？
> A：
> **API 网关**：基础设施层面，负责统一的流量入口、路由、限流、认证、协议转换。由平台/基础设施团队维护。
> **BFF**：业务层面，为特定客户端（Web/Mobile/Third-party）定制的聚合层。由业务团队维护。
>
> **需要两者同时存在的场景**：
> - 大型系统，多种客户端需要不同的数据聚合逻辑
> - 网关层不应该有业务逻辑，BFF 负责业务聚合和裁剪
> - 安全分层：网关做全局认证和限流，BFF 做业务级别的权限和数据过滤
>
> **架构建议**：
> ```
> 客户端 → API 网关（统一入口） → BFF（业务聚合） → 后端微服务
> ```
> 小规模团队可以直接用 API 网关承担部分聚合逻辑，但需注意不要让网关变得"胖"。

> Q：网关层如何实现灰度发布？
> A：
> **方案一：基于请求头/参数路由**：
> ```
> 请求 → 网关
>        │
>        ├─ x-version: v2 → 路由到 v2 版本的服务
>        └─ 无版本头 → 路由到 v1（稳定版）
> ```
>
> **方案二：基于用户标识的一致性哈希**：
> ```
> 请求 → 网关
>        │
>        └─ userId % 100 < 5 → 灰度的 5% 用户被路由到新版本
> ```
>
> **方案三：基于权重**：
> - APISIX 的 `traffic-split` 插件：按权重分配流量到不同 Upstream
> - Kong 的 `canary` 插件：逐步增加新版本流量比例（5% → 20% → 50% → 100%）
>
> **灰度发布关键**：
> - 灰度期间同时监控两个版本的错误率、延迟、业务指标
> - 发现异常自动回滚（rollback）
> - 灰度用户标识需保持一致性（同一个用户始终走同一版本）

### 5.2 认证鉴权

```
    微服务认证架构:

**高频面试题：**

> Q：JWT 和 Session 认证在微服务架构中如何选择？
> A：
>
> | 维度 | JWT | Session |
> |------|-----|---------|
> | 无状态 | 是（服务端无需存储） | 否（需要 Session Store） |
> | 跨服务 | 天然支持，Token 自包含 | 需要共享 Session Store |
> | 撤销 | 难（Token 签发后无法撤销，需黑名单） | 易（删除 Session） |
> | 大小 | 大（Payload 携带信息，KB 级） | 小（只存 Session ID） |
> | 刷新 | 需 Refresh Token 机制 | 自动续期 |
> | 安全性 | Token 泄露后影响时间长 | Session ID 泄露影响时间短 |
> | 适合场景 | 跨服务认证、移动端、第三方 API | 传统 Web 应用、内网系统 |
>
> **微服务推荐组合**：
> - 内部服务间：JWT（公钥验证，无状态，高性能）
> - 用户会话：短生命周期 JWT（15min）+ Refresh Token（7天）+ Redis 黑名单
> - 高安全场景：JWT + Session 并存，JWT 做快速验证，Session 做撤销控制

> Q：服务间调用如何认证？如何防止一个服务仿冒另一个服务？
> A：服务间认证（M2M Authentication）方案：
>
> 1. **mTLS（双向 TLS）**：服务端和客户端互相验证证书，最安全但运维成本高
> 2. **JWT + Service Account**：每个服务有自己的 Service Account，使用 OAuth2 Client Credentials 获取 JWT
> 3. **API Key**：每个服务分配 API Key，通过 Header 传递，简单但不推荐用于生产
> 4. **Service Mesh（Istio）**：Sidecar 自动处理 mTLS，应用层无需关心
>
> ```yaml
> # Istio PeerAuthentication 示例
> apiVersion: security.istio.io/v1beta1
> kind: PeerAuthentication
> metadata:
>   name: default
>   namespace: production
> spec:
>   mtls:
>     mode: STRICT  # 严格模式，所有服务间通信必须用 mTLS
> ```

### 5.3 生产踩坑经验

**踩坑 1：JWT 过大导致 HTTP 头超限**
> 某系统 JWT 携带了完整的用户权限列表（500+ 权限），Token 超过 8KB，导致 Nginx 默认的 `large_client_header_buffers` 超限，请求被拒绝。
> **解决**：JWT 只放最小必要信息（user_id, role），权限通过网关查询权限服务获取；或将权限信息放在 Redis 中，JWT 只存 session_id。

**踩坑 2：API 网关成为瓶颈**
> 某电商大促期间，所有流量经过单节点 Kong 网关，网关 CPU 打满 100%，请求延迟从 10ms 飙升到 5s，所有业务不可用。
> **解决**：网关多副本部署 + 水平自动伸缩 + 网关前的全局负载均衡（SLB）；核心链路设置 Bypass 机制，网关故障时自动绕过部分规则直接访问后端。

**踩坑 3：BFF 和 API 网关职责混淆**
> 某团队在网关层写业务聚合逻辑，网关代码逐渐膨胀到 "上帝类"，修改网关需要全团队评审，发布网关严重影响所有业务。
> **教训**：牢记能力分层——网关只做跨横切关注点（认证、限流、路由），所有业务逻辑放在 BFF 或后端服务中。

**踩坑 4：灰度发布缺乏全链路标记**
> 灰度流量只标记了进入网关的请求，但请求在后端各个服务间调用时，灰度标记丢失，导致灰度流量和非灰度流量混合处理。
> **解决**：全链路传递灰度标记（Trace Context 的一部分），每个服务在调用下一个服务时传递 x-canary 头。

**高频面试题：**

> Q：Prometheus 的 Pull 模式和传统 Push 模式各有什么优劣？
> A：
> **Prometheus Pull 模式（拉取）**：
> - 服务端控制采集节奏，不会因为突发流量冲垮监控系统
> - 通过 Service Discovery 自动发现目标
> - 更适合 K8s 环境（Pod 自动发现/摘除）
> - 问题：监控系统需能直接访问目标端点（防火墙/网络策略限制）
>
> **Push 模式（推送 - Graphite/OpenTSDB）**：
> - 服务端不需要知道目标的存在
> - 适合短暂任务（Batch Job）和网络隔离环境
> - 问题：可能冲垮监控系统，数据重复推送
>
> **现代实践**：Push + Pull 结合
> - 常驻服务：Prometheus Pull
> - 短暂任务：Pushgateway → Prometheus
> - 跨网络：Prometheus Remote Write

> Q：在微服务中如何设计一个"够用"的告警规则，既能发现问题又不产生告警风暴？
> A：
> **原则**：
> 1. **告警要 actionable**：每个告警收到后都需要有人采取行动，否则降低级别
> 2. **避免多级告警同时触发**：根因告警（如数据库 down）产生一个告警即可，所有依赖它的服务告警自动静默
> 3. **引入告警抑制**：Prometheus Alertmanager 的 inhibit_rules 配置
>
> ```yaml
> # 抑制规则示例
> inhibit_rules:
>   - source_match:
>       severity: 'critical'
>       alertname: 'DatabaseDown'
>     target_match:
>       severity: 'warning'
>     equal: ['service']  # 相同的 service 标签
> ```
>
> **告警阈值设计**：
> - 不要固定阈值（100% CPU 才告警已经晚了）
> - 使用基于基线的动态阈值（与过去 7 天同期比较异常偏离）
> - 错误率用百分比而非绝对值（小流量服务错误率波动大需调整窗口）

### 6.3 生产踩坑经验

**踩坑 1：日志采集导致磁盘写满**
> PHP 服务每请求打印 50 行日志，Filebeat 未做限速，QPS 5000 时日志写入速率达到 200MB/s，打满了系统盘。
> **解决**：日志级别分层——ERROR 级别全量，WARN 按 1/10 采样，INFO 按 1/100 采样；日志轮转（Log Rotation）保留最近 7 天；日志和数据盘分离。

**踩坑 2：Prometheus 拉取导致服务雪崩**
> 大促期间 Prometheus 每 15s 拉取所有服务的 /metrics 端点，拉取请求与服务请求争抢连接池，触发服务雪崩。
> **解决**：给监控端点设置独立的线程池/连接池；降低拉取频率（15s → 30s）；大幅增加 Prometheus 的 scrape_timeout 配置；使用 Prometheus 的 HonorLabels 避免重写 metric 导致的性能开销。

**踩坑 3：链路追踪采样率设置不当**
> 100% 采样 → Jaeger 存储吃紧，每天产生几 TB 数据
> 1% 采样 → 查不到问题链路，定位故障困难
> **解决**：头部采样 + 尾部采样结合
> - **头部采样**（Head-based）：低采样率（1-5%）用于常规监控
> - **尾部采样**（Tail-based）：根据错误/高延迟等条件动态决定是否保存全量链路

**踩坑 4：告警疲劳——"狼来了"效应**
> 团队每周收到 1000+ 告警，但 90% 是误报或无需处理的噪音，最终告警被忽略。真出问题时告警没人看。
> **教训**：告警规则每季度重新审视和清理；每条告警必须有明确的解决步骤（Runbook）；引入"告警预算"——每月允许的告警数量，超过即视为告警基础设施需要优化。

**高频面试题：**

> Q：金丝雀发布和蓝绿部署各自的优缺点？
> A：
>
> **金丝雀发布**（Canary Release）：
> - 渐进式流量切换，风险可控
> - 可以精确控制流量比例（1%→5%→10%→...→100%）
> - 发现问题影响范围小
> - 需要全链路灰度标记（网关 → 服务 A → 服务 B 保持灰度上下文）
> - 发布周期较长（需要等待每个阶段的观察期）
>
> **蓝绿部署**（Blue-Green Deployment）：
> - 切换瞬间完成（秒级）
> - 回滚也瞬间（切回蓝环境）
> - 不需要灰度上下文传递
> - 资源消耗翻倍（同时运行两套完整环境）
> - 数据库兼容性问题：两个版本共用数据库，数据库 Schema 变更需向前/向后兼容
>
> **选择建议**：
> - 大版本更新、数据库 Schema 变更 → 蓝绿 + 数据库兼容
> - 小功能迭代、性能优化 → 金丝雀
> - 流量大、稳定性要求高 → 金丝雀（逐步放量 + 监控确认）
> - 资源紧张的团队 → 滚动更新（最省资源）

**高频面试题：**

> Q：GitOps 相比传统 CI/CD 的核心优势是什么？
> A：
> 1. **唯一真相源**：Git 仓库是整个系统的期望状态声明，所有变更通过 PR/MR 审查
> 2. **自动漂移修复**：如果有人在集群中手动改了资源（kubectl edit），GitOps 操作者会自动回滚到 Git 中的状态
> 3. **审计可追溯**：谁在什么时间改了什么是完全透明的，每个变更都有 commit
> 4. **回滚极简**：git revert 即可回滚整个部署
> 5. **环境一致性**：dev/staging/prod 的差异只有 overlay 中的配置差异
>
> **GitOps 不是银弹**：
> - 需要 Git 和 K8s 的深度理解
> - 处理 Secrets（密钥）需要配合 SealedSecrets / External Secrets / SOPS
> - 数据库 Schema 变更不适合纯 GitOps 模式（需要专门的迁移工具）

### 7.3 生产踩坑经验

**踩坑 1：镜像 Tag 用 latest 导致回滚困难**
> 每次构建都用 `latest` 标签，出现问题时无法快速回滚到上一个版本的镜像。
> **解决**：镜像 Tag 使用 Git commit SHA 或语义版本号：
> ```
> registry.example.com/order-service:v1.2.3-abcd1234
> ```
> 永远不用 `latest` 标签部署到生产环境。

**踩坑 2：蓝绿部署中数据库 Schema 不兼容**
> 绿环境部署了新代码，需要新的 `payment_method` 字段，蓝环境的旧代码因为没有这个字段写 NULL 而崩溃。
> **解决**：数据库变更遵从"向前兼容、向后兼容"原则：
> - Phase 1：新增字段（允许 NULL）+ 代码只读新字段
> - Phase 2：代码开始写新字段
> - Phase 3：设置 NOT NULL + 填充旧数据
> - Phase 4：废弃旧字段

**踩坑 3：GitOps 自动同步导致级联故障**
> 某个 ConfigMap 配置错误被提交，ArgoCD 自动同步到生产集群，导致所有 Pod CrashLoopBackOff。
> **解决**：生产环境 GitOps 启用 Manual Sync + 同步窗口限制；配置变更走预发布验证流程；使用 ArgoCD 的 Sync Waves 控制资源应用顺序。

**踩坑 4：灰度环境中全链路调用未隔离**
> 金丝雀服务 A 调用了稳定版服务 B，服务 B 又调用了金丝雀服务 C（由于注册中心随机负载），导致灰度请求在多个版本间随机跳跃，排查问题困难。
> **解决**：全链路灰度需要每个服务感知灰度的上下文，并通过 Header 传递灰度标签，在服务发现时根据灰度标签路由到对应版本。

### 8.4 迁移踩坑经验

**踩坑 1：大爆炸式重写（Big Bang Rewrite）**
> 某团队决定一次性重写整个单体为微服务，耗时 18 个月，投入 30 人。上线当天系统崩溃，因为新系统与其他系统的集成点遗漏了、性能表现不如预期、团队对分布式系统的理解在开发过程中发生了变化。
> **教训**：永远不要 Big Bang Rewrite。使用 Strangler Fig 模式，一次只替换一个功能模块，新旧系统并行运行，逐步切换流量。新功能在新系统开发，旧功能从旧系统逐步迁移。

**踩坑 2：新系统未考虑已有基础设施**
> 单体运行在物理机 + 手动部署 + Nagios 监控。微服务运行在 K8s + CI/CD + Prometheus。但在迁移期间，两个系统需要共用一个数据库、共享 Session、同域 Cookie。
> **教训**：迁移方案要包含共存期的"桥接方案"，如 Session 共享、数据双向同步、统一域名路由。

**踩坑 3：过度抽象通用库**
> 多个微服务共享一个"common-lib"，包含了数据库模型、DTO、工具类。每次修改 common-lib 都需要重新发布所有依赖的服务。
> **教训**：共享库应该是稳定的、通用的工具类。业务逻辑不要放在共享库，使用 Copy/Paste 的代码冗余也好过共享库导致的耦合。多语言环境下，共享库还需要跨语言维护（Go/PHP 各一套）。

**踩坑 4：高估了团队的 DevOps 能力**
> 团队习惯手动部署单体，切换到微服务后，每个服务都需要独立构建、测试、部署、监控。运维负担从"每周一次"变为"每天多次"，运维人员不堪重负。
> **教训**：在拆分第一个微服务之前，先建设好自动化基础设施（CI/CD、容器化、日志、监控、告警），确保每个服务可以"一条命令部署"。

**踩坑 5：PHP 微服务中的单库困境**
> PHP 项目拆分微服务后，仍然共用一个 MySQL 数据库。每个服务有自己的一组表，但 Schema 变更时仍然需要协调所有服务。
> **教训**：PHP 微服务的数据库隔离通常比 Go 更难实现，因为 PHP 生态的传统数据库操作模式和 ORM 设计往往假设单库。建议：
> - 优先使用独立的数据库实例（即使在同一台 MySQL 上也要用不同的 database）
> - API 网关到 PHP 后端之间增加一层 Redis 缓存，减少对数据库的直接穿透
> - PHP 服务的数据库连接配置通过环境变量注入，从架构层面强制隔离

**高频面试题：**

> Q：在 DDD 中，如何区分 Entity 和 Value Object？
> A：
>
> | 特征 | Entity（实体） | Value Object（值对象） |
> |------|---------------|----------------------|
> | 标识 | 有唯一标识（ID） | 无标识，靠属性值区分 |
> | 可变性 | 可变（mutable） | 不可变（immutable） |
> | 相等性 | 引用/ID 相等 | 所有属性值相等 |
> | 生命周期 | 有独立生命周期 | 附属于实体 |
> | 例子 | Order, User, Product | Address, Money, Quantity |
>
> 简单判断：如果你关心"这是哪个对象"，它是 Entity；如果你关心"它是什么样"，它是 Value Object。比如 100 元人民币，你不在乎是哪张纸币，你只在乎它的面值——所以 Money 是 Value Object。

> Q：如何判断一个聚合的边界是否合理？
> A：聚合设计的几个原则：
> 1. **聚合内强一致性，聚合间最终一致性**：一个事务只修改一个聚合
> 2. **聚合根保护不变条件**：外部只能通过聚合根操作聚合内的对象
> 3. **聚合不宜过大**：一个大聚合导致并发冲突高、事务失败率高
> 4. **参考事务边界**：如果两个实体在同一个业务场景中需要强一致事务，它们应该在一个聚合内
>
> 反例：Order 和 OrderItem 在一个聚合内（合理），Order 和 User 不在同一个聚合（合理），但有些人把 Order 的所有历史事件也放在 Order 聚合内导致聚合膨胀（不合理）。

**高频面试题：**

> Q：Service Mesh 的优势和代价是什么？什么时候应该采用？
> A：
> **优势**：
> 1. **业务与基础设施解耦**：治理能力下沉到 Sidecar，业务代码无需关注服务发现、熔断、mTLS
> 2. **多语言统一治理**：PHP/Go/Java 等服务使用一致的治理策略，无需每个语言 SDK 单独实现
> 3. **流量精细控制**：VirtualService/DestinationRule 实现灰度、流量镜像、故障注入
> 4. **零信任安全**：自动 mTLS，服务间通信默认加密
>
> **代价**：
> 1. **性能损耗**：每个请求多一次 localhost 代理转发（Envoy），延迟增加 2-5ms
> 2. **资源消耗**：每个 Pod 多一个 Sidecar 容器（约 50-100MB 内存）
> 3. **运维复杂度**：需要专门的 Service Mesh 运维团队
> 4. **调试困难**：多一跳，排查问题需要理解 Envoy 的数据面
> 5. **版本兼容性**：Istio 版本升级有时会引入不兼容变化
>
> **采用时机**：
> - 团队规模 > 50 人，服务数 > 20
> - 多语言技术栈（PHP + Go + Java + Python）
> - 对安全有较高要求（金融/医疗等合规场景）
> - 有专门的平台基础设施团队
>
> **不推荐场景**：
> - 小团队（< 10 人）
> - 非 K8s 环境
> - 对延迟极度敏感（高频交易等）
> - 服务全同构（如全是 Go）

### 9.4 Serverless 与微服务的边界

```
    Serverless 与微服务的对比:

    ┌─────────────────────────────┐   ┌─────────────────────────────┐
    │         微服务               │   │        Serverless           │
    ├─────────────────────────────┤   ├─────────────────────────────┤
    │ 常驻进程，持续运行             │   │ 事件触发，按需启动           │
    │ 手动/自动伸缩（秒级）          │   │ 从 0 到 N 自动伸缩（毫秒级） │
    │ 需要基础设施管理（K8s）        │   │ 完全托管，无需管理服务器     │
    │ 有冷启动问题（相对小）          │   │ 冷启动问题显著（秒级）       │
    │ 适合长时间运行的服务            │   │ 适合短时间运行的函数         │
    │ 开发者对运行时有完全控制权       │   │ 运行时环境受限               │
    │ 成本：24/7 运行               │   │ 成本：按调用次数计费         │
    └─────────────────────────────┘   └─────────────────────────────┘

    Serverless 什么时候替代微服务，什么时候互补:

    互补场景:
    ┌──────────────────────────────┐
    │     API 网关                  │
    ├──────────┬───────────────────┤
    │  路由     │ 触发规则            │
    └─────┬────┴─────┬─────────────┘
          │          │
    ┌─────v────┐ ┌───v──────────┐
    │ 核心微服务 │ │ Serverless 函数 │
    │ (常驻)    │ │ (事件处理)      │
    │ 订单服务  │ │ 图片压缩       │
    │ 用户服务  │ │ 数据 ETL       │
    │ 支付服务  │ │ 通知发送       │
    └──────────┘ │ 定时任务       │
                └──────────────┘

    Serverless 适合的微服务场景:
    1. 事件驱动的数据处理（Kafka Consumer → 云函数处理）
    2. 图片/视频处理（上传触发压缩/转码）
    3. 定时任务（Cron → 清理过期数据）
    4. Webhook 处理（第三方回调）
    5. 轻量 API（低流量、突发性强的接口）

    Serverless 不适合的场景:
    1. 延迟敏感的核心交易链路
    2. 长连接/WebSocket
    3. 有状态服务
    4. 需要 GPU/特殊硬件
    5. 高 QPS 稳定流量的服务（持续运行反而不如常驻服务便宜）
```

**高频面试题：**

> Q：你的服务应该用微服务还是 Serverless 函数？如何决策？
> A：
> **决策树**：
> ```
> 服务是否无状态？
> ├─ 否 → 微服务（Serverless 有限支持有状态）
> └─ 是
>    ├─ 请求延迟要求 < 100ms？
>    │  ├─ 是 → 微服务（Serverless 冷启动 > 100ms）
>    │  └─ 否
>    │     ├─ 流量是否稳定？
>    │     │  ├─ 是 → 微服务（持续运行成本更低）
>    │     │  └─ 否
>    │     │     ├─ 流量波动极大且突发性高？
>    │     │     │  ├─ 是 → Serverless（按需伸缩，避免资源浪费）
>    │     │     │  └─ 否 → 微服务
>    │     │     └─
>    │     └─
>    └─ 启动时间是否可接受（1-5s）？
>       ├─ 是 → Serverless
>       └─ 否 → 微服务（或 Provisioned Concurrency）
> ```
>
> **面试中建议的回答**：两者不是互斥关系，而是互补关系。核心业务用微服务保障稳定性和性能，周边业务/事件处理用 Serverless 降本增效。

**高频面试题：**

> Q：在 PHP + Go 混合架构中，服务间调用的最佳实践是什么？
> A：
> **通信方式**：
> - 核心服务间：gRPC（Go → Go），性能最优
> - BFF → 核心服务：gRPC（PHP → Go），使用 Swoole 协程 gRPC 客户端
> - 非关键链路：REST over HTTP（Python/PHP 等临时服务）
>
> **上下文传递**：
> - Trace ID、灰度标签、用户信息通过 gRPC Metadata 传递
> - PHP BFF 从 HTTP 请求头提取 → gRPC metadata 传递到 Go 服务
>
> **容错**：
> - PHP 侧使用超时 + 重试 + 熔断（Redis 滑动窗口）
> - Go 侧使用 gRPC 内置重试 + Istio 熔断
>
> **部署**：
> - PHP 服务：K8s Deployment + HPA，镜像 200-500MB
> - Go 服务：K8s Deployment + HPA，镜像 10-20MB

> Q：PHP 技术栈的团队在转向微服务架构时，最大的挑战是什么？
> A：
> 1. **无状态化改造**：PHP-FPM 天然无状态（每次请求独立），但 PHPer 习惯了在 $_SESSION 里存数据、在 $_SERVER 里获取配置。微服务需要显式传递上下文，无法依赖超全局变量
> 2. **连接池和长连接**：PHP-FPM 每次请求创建数据库/Redis 连接，微服务架构下连接数随实例数倍增。需要 Swoole 连接池或代理层（如 ProxySQL）
> 3. **异步思维转变**：PHP 同步阻塞是舒适区，微服务中需要处理异步消息、事件驱动、补偿事务
> 4. **DevOps 能力**：PHP 传统部署是 FTP/SFTP + git pull，微服务需要容器化 + CI/CD + K8s，对团队能力要求大幅提升
> 5. **社区生态**：虽然 PHP 微服务生态在快速发展（Hyperf, Swoole），但与 Go/Java 的微服务生态相比仍有差距
