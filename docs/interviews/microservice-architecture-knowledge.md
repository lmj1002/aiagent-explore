## 基础概念

### 1. 微服务架构定义与本质

微服务架构（Microservices Architecture）是一种将单一应用程序划分为一组小服务的架构风格。每个服务运行在独立的进程中，围绕业务能力构建，通过轻量级通信机制（通常是 HTTP REST 或 gRPC）协作。

**与单体架构的核心区别：**

| 维度 | 单体架构 | 微服务架构 |
|------|---------|-----------|
| 部署单元 | 单个 WAR/JAR | 独立进程 |
| 扩展粒度 | 整体复制 | 按服务独立扩展 |
| 技术栈 | 单一语言/框架 | 多语言异构 |
| 数据存储 | 单一数据库 | 数据库 per 服务 |
| 团队组织 | 职能团队 | 跨职能小团队 |

### 2. 服务拆分原则

#### 2.1 三大维度综合考量

- **业务维度（最重点）**：通过 DDD 识别限界上下文（Bounded Context），确定业务边界和聚合根
- **技术维度**：数据一致性要求、性能瓶颈、是否需要技术异构
- **组织维度**：康威定律——系统架构会复制组织的沟通结构

#### 2.2 五步"灵魂拷问"决策树

| 规则 | 判据 | 示例 |
|------|------|------|
| 规则1：名词归属法 | 看核心数据（实体）归谁管 | "修改收货地址" -> 地址属于用户，归属用户服务 |
| 规则2：生命周期一致性 | 看数据是否同生共死 | 订单明细和订单主表必须在同一个服务 |
| 规则3：事务强一致性边界 | 是否需要强事务支持 | 需要强一致性尽量放一个服务内 |
| 规则4：变更频率与原因 | 看修改代码的原因是否相同 | 促销规则天天变，剥离为独立服务 |
| 规则5：康威定律 | 看谁在维护 | 减少跨团队的强耦合调用 |

#### 2.3 服务粒度判断标准

服务不是越小越好——核心判断标准：**一个服务能由一个小团队（2-5人）独立负责，且职责边界清晰**。

**过早拆分的六大代价：**
1. 分布式通信带来的网络延迟
2. 分布式事务的复杂性
3. 测试与调试难度增加
4. 运维复杂度和资源开销上升
5. 数据一致性保障困难
6. 重构成本放大

### 3. DDD 限界上下文

**限界上下文（Bounded Context）** 是 DDD 的核心战术模式，每个限界上下文对应一个独立的微服务。上下文之间通过**防腐层（Anti-Corruption Layer）** 进行通信，降低耦合。

```
                     ┌──────────────────┐
                     │   订单上下文      │
                     │ ┌──────────────┐ │
                     │ │   Order (聚合根)│ │
                     │ └──────────────┘ │
                     └────────┬─────────┘
                              │ 防腐层
                     ┌────────▼─────────┐
                     │   商品上下文      │
                     │ ┌──────────────┐ │
                     │ │  Product(聚合根)│ │
                     │ └──────────────┘ │
                     └──────────────────┘
```

**聚合根示例（伪代码）：**
```java
public class Order {
    private String orderId;
    private List<OrderItem> items;
    private OrderStatus status;

    // 聚合根保证一致性
    public void addItem(Product product) {
        if (this.status == OrderStatus.PAID)
            throw new IllegalStateException("已支付订单不可修改");
        this.items.add(new OrderItem(product));
    }
}
```

### 4. 康威定律

> **"设计系统的组织，其产生的设计和架构等价于组织间的沟通结构。"**

**关键要点：**
- 如果团队按照业务线划分，微服务也往往按业务域进行划分
- 一个限界上下文不应超过一个团队的职责范围
- 两个团队维护一个微服务，必然将该服务拆成事实上的两个服务

**沟通成本公式（《人月神话》）：** 沟通成本 = n(n-1)/2

| 团队规模 | 沟通渠道 |
|---------|---------|
| 5人 | 10条 |
| 15人 | 105条 |
| 50人 | 1225条 |

### 5. CAP 理论与 BASE 理论

**CAP 定理：** 分布式系统最多只能同时满足一致性（Consistency）、可用性（Availability）、分区容错性（Partition Tolerance）中的两项。由于网络分区不可避免，实际是在 C 和 A 之间做取舍。

| 场景 | 选择 | 典型组件 |
|------|------|---------|
| 电商/社交 | AP（最终一致即可） | Eureka、Nacos(AP模式) |
| 金融/支付 | CP（强一致优先） | Consul、ZooKeeper、Nacos(CP模式) |

**BASE 理论（微服务基石）：**
- **BA**（Basically Available）：基本可用，允许降级
- **S**（Soft State）：软状态，允许中间状态
- **E**（Eventually Consistent）：最终一致性

## 核心原理

本章深入剖析微服务架构七大核心领域的原理。

### 1. 服务注册与发现

#### 1.1 核心流程

```
                   注册中心
                   ┌─────┐
   ┌──────────────►│ Nacos│◄──────────────┐
   │               └─────┘                │
   │ 1.启动注册        ▲                  │ 2.定时拉取/长轮询
   │                   │                  │
┌──▼──┐          ┌────┴───┐          ┌───┴───┐
│Provider│          │        │          │Consumer│
└──────┘          └────────┘          └───────┘
   │                                        │
   └────────── 3. 直接 RPC/REST ──────────┘
```

**核心步骤：**
1. 服务提供者启动时向注册中心注册自身元数据（IP、端口、健康检查URL）
2. 服务消费者从注册中心获取服务实例列表
3. 消费者根据负载均衡策略选择一个实例发起调用
4. 注册中心通过心跳/健康检查剔除不可用实例

#### 1.2 主流组件 CAP 对比

| 组件 | CAP模型 | 一致性协议 | 心跳频率 | 剔除超时 | 配置中心 |
|------|---------|-----------|---------|---------|---------|
| Eureka | **AP** | Peer to Peer 最终一致 | 30秒 | 90秒 | 不支持 |
| Nacos | **AP/CP可切换** | Distro(AP) / Raft(CP) | 5秒 | 30秒 | 内置 |
| Consul | **CP** | Raft 共识算法 | 可配置 | 服务端主动探测 | KV存储 |
| ZooKeeper | **CP** | ZAB 原子广播 | Session过期 | 依赖网络 | 不支持 |

#### 1.3 Nacos 核心特性（2025/2026 首选）

- **分级存储模型**：Namespace -> Group -> Service -> Cluster -> Instance
- **保护阈值**：健康比例低于阈值时，返回所有实例（含不健康的），防止雪崩
- **临时 vs 永久实例**：临时实例心跳维持，永久实例服务端主动探测
- **gRPC 通信**（2.x+）：性能大幅提升，延迟从秒级降至毫秒级
- **服务变更主动推送**：区别于 Eureka 的客户端定时拉取
- **多环境隔离**：Namespace 级物理隔离

#### 1.4 Eureka 自我保护机制

当 15 分钟内超过 85% 的心跳失败时，Eureka 认为自身网络异常，**不再剔除任何实例**。宁可放过一千也不错杀一个，避免因网络波动导致大规模误删。这在网络不稳定的场景下是必要的保护手段，但也意味着不可用实例可能被调用。

#### 1.5 服务下线感知机制

| 方式 | Eureka | Nacos |
|------|--------|-------|
| 主动下线 | 调用 deregister + 等待下一次拉取(30s) | 调用 deregister + gRPC 立即推送 |
| 故障下线 | 90秒心跳超时 | 15秒未检测到健康 -> 30秒剔除 |
| 消费者感知延迟 | 最长 30 秒（客户端拉取间隔） | 亚秒级（gRPC 推送） |

### 2. API 网关

#### 2.1 六大核心能力

```
                   ┌─────────────────────────────────┐
                   │           API 网关               │
                   │ ┌─────┬─────┬─────┬─────┬─────┐│
     Client ──────►│鉴权  │限流  │路由  │协议  │可观测│├────► 微服务A
                   │JWT   │令牌桶│灰度  │转换  │链路  ││     微服务B
                   │OAuth2│滑动窗│蓝绿  │HTTP  │日志  ││     微服务C
                   │APIKey│漏桶  │AB    │gRPC  │监控  ││      ...
                   │WAF   │     │      │      │      ││
                   └─────┴─────┴─────┴─────┴─────┴─────┘
```

1. **统一入口与路由转发**：反向代理、动态路由、灰度/蓝绿/AB发布
2. **全链路安全防护**：JWT/OAuth2.0/API Key鉴权、WAF、黑白名单、TLS/SSL
3. **流量治理与精细化管控**：限流熔断降级、流量镜像、负载均衡
4. **协议适配**：HTTP 转 gRPC、协议转换
5. **可观测性**：日志、指标、链路追踪
6. **业务增强**：缓存、Mock、请求/响应转换

#### 2.2 四大限流算法

| 算法 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|---------|
| 固定窗口 | 固定时间窗口内限制最大请求数 | 实现简单、资源占用低 | **窗口边界临界突刺问题** | 简单全局粗粒度限流 |
| 滑动窗口 | 拆分为多个小窗口，滑动统计 | 精度高、无临界突刺 | 实现复杂度高 | 精细化核心接口限流 |
| 漏桶算法 | 固定速率处理请求，超额排队/拒绝 | 严格平滑流量 | 无法应对突发流量 | 下游处理速率固定的流量整形 |
| 令牌桶算法 | 固定速率生成令牌，请求需获取令牌通过 | 支持突发流量、灵活性高 | 实现复杂度略高 | **大多业务场景，网关首选** |

**固定窗口临界突刺问题说明：** 如限制 QPS=100，在第59秒的窗口内请求达到100后全被拒绝，但在第60秒进入新窗口，集中涌入100请求。更极端的场景：第59.9秒100个请求 + 第60.0秒100个请求 = 0.2秒内200请求压垮系统。

#### 2.3 主流网关性能对比（APISIX 3.0 vs Kong 3.0）

| 场景 | APISIX 3.0 (QPS) | Kong 3.0 (QPS) | APISIX优势 |
|------|:---:|:---:|:---:|
| 1条路由，无插件（纯代理） | 14,104 | 9,857 | 140% |
| 1条路由 + 限流插件 | 9,154 | 4,810 | 190% |
| 1条路由 + 限流 + 鉴权 | 8,933 | 3,977 | 220% |
| 5000条路由 | 13,787 | 9,840 | 140% |

APISIX 使用**基数树（Radix tree）** 进行高效路由匹配；Kong 通过遍历所有路由进行逐一匹配。

#### 2.4 选型建议

- 追求高性能 + 国产化生态 -> **APISIX**
- 追求全球化生态 + 商业支持 -> **Kong**
- Java 技术栈 -> **Spring Cloud Gateway**
- 服务网格架构 -> **Istio（Envoy）**

#### 2.5 灰度路由策略

| 策略 | 原理 | 典型场景 |
|------|------|---------|
| 权重路由 | 按比例分配流量到不同版本 | 灰度发布 10% -> 50% -> 100% |
| 标签路由 | 按 Header/Cookie/参数匹配 | 内部测试用户先体验新功能 |
| 蓝绿部署 | 两套独立环境，一键切换 | 在线服务零停机发布 |
| A/B 测试 | 精细化流量分配，验证新功能 | 转化率对比实验 |

### 3. 配置中心

#### 3.1 解决的核心痛点

| 痛点 | 传统方案 | 配置中心方案 |
|------|---------|------------|
| 配置分散 | 每个服务独立管理配置文件 | 集中统一管理 |
| 修改需重启 | 改完重新打包部署 | 动态热更新，秒级生效 |
| 环境不一致 | 手动同步易出错 | Namespace/Env 多环境隔离 |
| 权限缺失 | 无细粒度控制 | RBAC + 审计日志 |
| 无法追溯 | 无变更记录 | 版本管理 + 一键回滚 |

#### 3.2 动态刷新原理（长轮询 Long Polling）

```
客户端                         配置中心
  │                              │
  │── 长轮询 (携带本地 MD5) ────►│
  │                              │── 比对 MD5
  │    ◄── 无变更，挂起30~60s ──┤
  │       （或变更立即返回）       │
  │                              │
  │── 收到变更通知 ──────────────│
  │── 拉取最新配置 ─────────────►│
  │◄── 返回最新配置内容 ─────────│
  │                              │
  │── 更新 Spring Environment ──►│
  │── 触发 @RefreshScope 刷新 ──►│
```

**Nacos 2.x 升级为 gRPC 双向流**，相比 HTTP 长轮询更轻量，实时性更好。

#### 3.3 主流配置中心对比

| 维度 | Nacos | Apollo | Spring Cloud Config |
|------|-------|--------|-------------------|
| 动态刷新 | 长轮询/秒级 | 长轮询/秒级 | 需配合 Bus |
| **灰度发布** | 基础（开源版） | **完善**（IP/标签） | 不支持 |
| 权限控制 | 基础 RBAC | 细粒度 + 审批流程 | 依赖 Git |
| 配置继承 | 不支持 | 支持环境/集群继承 | 不支持 |
| 管理界面 | 简洁 | **功能丰富** | 无界面 |
| 服务发现 | 内置 | 仅配置 | 仅配置 |
| 部署复杂度 | 低（单 Jar） | 中（4组件） | 中 |
| 回滚 | 支持版本回滚 | 支持 | 依赖 Git 回滚 |

#### 3.4 多环境配置方案

| 方案 | Nacos | Apollo |
|------|-------|--------|
| 环境隔离 | Namespace（dev/test/prod） | Env（DEV/FAT/UAT/PROD） |
| Profile 方式 | DataId 拼接 profile（`order-dev.yaml`） | 不推荐 |
| 物理隔离 | 不同集群独立部署 | 可独立部署多套 |

#### 3.5 敏感配置处理

1. **Jasypt 加密**：配置中心存储密文，密钥通过环境变量传递
2. **环境变量注入**：优先级高于配置中心，适合密码等敏感信息
3. **KMS 集成**：对接云厂商密钥管理服务
4. **最佳实践**：不同环境使用不同密钥，定期轮换

### 4. 服务间通信

#### 4.1 三大通信模式对比

| 维度 | REST（HTTP/1.1） | gRPC | 消息队列（MQ） |
|------|-----------------|------|--------------|
| 通信方式 | 同步 | 同步 | **异步** |
| 数据格式 | JSON（文本） | Protobuf（二进制） | 多样 |
| 性能 | 中等 | **最高** | 较高 |
| 强类型 | 弱 | **强** | 弱 |
| 流式支持 | 不支持 | **双向流** | 单向 |
| 典型框架 | Feign/OpenFeign | gRPC-Java/grpc-go | Kafka/RocketMQ |
| 适用场景 | 外部API | 内部高性能 | 解耦/最终一致性 |

**选型原则：**
- 外部 API / 异构系统互通 -> REST
- 内部高性能低延迟场景 -> gRPC（比 HTTP/1.1 延迟低约 30%）
- 解耦 / 削峰填谷 / 最终一致性 -> 消息队列

#### 4.2 服务熔断与降级

**熔断器三态转换：**

```
                         ┌─────────────┐
                         │   CLOSED    │
                         │  (正常调用)   │
                         └──────┬──────┘
                                │ 失败率/慢调用比例 达到阈值
                                ▼
                         ┌─────────────┐    超时窗口
                         │    OPEN     │ ─────────────►
                         │  (快速失败)  │
                         └──────┬──────┘
                                │ 超时窗口结束，放行试探
                                ▼
                         ┌─────────────┐
                         │  HALF_OPEN  │
                         │   (试探恢复) │
                         └──────┬──────┘
                       成功◄────┘────►再次失败
```

#### 4.3 Sentinel vs Resilience4j

| 对比维度 | Sentinel | Resilience4j |
|---------|---------|-------------|
| 设计理念 | 流量治理为核心，全方位保护 | 轻量级、模块化、函数式 |
| 隔离策略 | 信号量隔离 | 信号量 + 线程池隔离 |
| 限流粒度 | 服务/方法/参数/链路级 | 服务/方法级 |
| 动态配置 | **原生支持，控制台动态配置** | 需额外集成配置中心 |
| 实时监控 | **强大的可视化控制台** | 基础，需集成第三方 |
| 集群支持 | 原生集群限流 | 有限 |
| 适用场景 | 大型系统、复杂流量治理 | 中小型微服务 |
| 选型建议 | Spring Cloud Alibaba 技术栈 | Spring Cloud 官方推荐 |

**@SentinelResource 示例：**
```java
@SentinelResource(
    value = "getOrder",
    fallback = "getOrderFallback",
    blockHandler = "getOrderBlockHandler",
    exceptionsToIgnore = {IllegalArgumentException.class}
)
public Order getOrder(Long orderId) {
    // 业务逻辑
}
```

**降级方案：**
- 快速失败：直接返回 503
- 兜底数据：从缓存读取旧数据返回
- 降级开关：主动关闭非核心功能，保证核心链路可用
- Mock 数据：返回静态模拟数据

#### 4.4 重试风暴防护

重试不当会导致**重试风暴**——调用方在服务短暂故障时大量重试，放大流量压垮下游。

防护措施：
1. **指数退避** + 随机抖动（Jitter）：`sleep = min(cap, base * 2^n + random(0, jitter))`
2. **最大重试次数**限制（通常 2-3 次）
3. **令牌桶限流**：对重试请求也应用限流
4. **分组分批恢复**：故障恢复后逐步放开流量

### 5. 分布式链路追踪

#### 5.1 核心概念

```
Trace: 一次完整的请求链路
  │
  ├── Span: 客户端/浏览器 (客户端)
  │     └── Span: API 网关
  │           └── Span: 订单服务
  │                 ├── Span: 用户服务 (RPC调用)
  │                 │     └── Span: 数据库查询
  │                 └── Span: 支付服务 (MQ消息)
  │                       └── Span: 第三方支付
```

- **Trace**：一次完整请求链路，全局唯一 TraceID
- **Span**：一次操作单元（方法调用/DB查询/RPC），有开始和结束时间
- **SpanContext**：包含 TraceID、SpanID、Sampled 标志、Baggage 数据
- **W3C TraceContext 标准**：`traceparent` 头格式 = `version-traceid-spanid-flags`

#### 5.2 OpenTelemetry 架构

```
┌─────────┐   ┌─────────┐   ┌─────────┐
│ 应用A    │   │ 应用B    │   │ 应用C    │
│ OTel SDK │   │ OTel SDK │   │ OTel SDK │
└────┬────┘   └────┬────┘   └────┬────┘
     │              │              │
     └──────────────┼──────────────┘
                    ▼
          ┌─────────────────┐
          │ OTel Collector    │
          │  ├  Batch/Sampling│
          │  └  Export        │
          └────────┬────────┘
                    │
         ┌─────────┼─────────┐
         ▼         ▼         ▼
     ┌──────┐ ┌──────┐ ┌────────┐
     │Jaeger│ │Zipkin│ │Prometheus│
     └──────┘ └──────┘ └────────┘
```

OpenTelemetry 是 CNCF 可观测性标准，统一了 Metrics（指标）、Logs（日志）、Traces（链路）三大数据类型 API/SDK，支持多后端导出。

#### 5.3 跨进程上下文传播

| 通信方式 | 传播机制 |
|---------|---------|
| HTTP | 通过 Header 传递 `traceparent` / `tracestate` |
| gRPC | 通过 Metadata 传递 |
| MQ（Kafka/RabbitMQ） | 消息 Header 中注入 TraceContext，消费者提取重建 |
| 异步任务/线程池 | 显式传递 Context，避免 `context.Background()` |

#### 5.4 采样策略

| 策略 | 原理 | 适用场景 |
|------|------|---------|
| **Head Sampling（头部采样）** | 请求入口处决定是否采样，概率/Parent-based | 实现简单，适合常规场景 |
| **Tail Sampling（尾部采样）** | Collector 缓存完整 Trace 后按规则保留 | 高精度保留感兴趣的 Trace |
| **概率采样** | 固定比例（如 1%、10%）随机采样 | QPS 极高时控制成本 |
| **Parent-based 采样** | 子 Span 继承父 Span 的采样决策 | 保持 Trace 完整性 |
| **Rate Limiting 采样** | 每秒固定采样数量 | 控制固定费用 |
| **自适应采样** | 按端点/错误率/延迟动态调整 | 智能优化采样率 |

**黄金法则：** 生产环境不是全量采集。核心做法是**高比例采样错误/慢 Trace + 低比例采样正常 Trace**。

**成本计算：** Span 数量 = QPS x 平均 Span 数每条 Trace x 采样率。1000 QPS 的服务，平均 20 个 Span/Trace，100% 采样 = 每秒 20,000 Span。

#### 5.5 工具对比

| 工具 | 特点 | 适用场景 |
|------|------|---------|
| OpenTelemetry | CNCF 标准，统一 API/SDK | 标准化建设、多后端输出 |
| Jaeger | CNCF 项目，原生支持 OTel | 中小规模、Uber 生态 |
| SkyWalking | **国产化、自动埋点、功能全面** | 信创环境、Java 生态主力 |
| Zipkin | 最早的开源方案 | 历史项目维护 |

### 6. 数据一致性方案

#### 6.1 方案全景图

```
分布式事务
├── 刚性事务 (CP, ACID)
│   ├── 2PC (两阶段提交) — XA 规范
│   └── 3PC (三阶段提交) — 增加准备+超时
│
└── 柔性事务 (AP+BASE, 最终一致性)
    ├── TCC (Try-Confirm-Cancel) — 业务补偿
    ├── Saga — 长事务+补偿事务
    ├── 可靠消息最终一致性（MQ事务消息）
    ├── 本地消息表 — eBay 经典方案
    └── 最大努力通知 — 定期校对
```

#### 6.2 TCC 模式详解

**三阶段执行流程：**

```
Try（预留资源）           Confirm/Cancel（确认/回滚）
┌───┐                    ┌───┐
│账户A│── 冻结 100 元 ────►│账户A│── 扣除 100 元（Confirm）
│服务│                    │服务│── 释放冻结（Cancel）
│订单│── 创建预订单 ─────►│订单│── 状态→已支付（Confirm）
│服务│                    │服务│── 取消预订单（Cancel）
│库存│── 预扣库存 1 ────►│库存│── 扣减库存（Confirm）
│服务│                    │服务│── 释放预扣（Cancel）
└───┘                    └───┘
```

**TCC 三大注意事项：**

1. **空回滚**：Try 没执行就收到 Cancel（超时等场景），Cancel 不能报错
2. **防悬挂**：Try 返回超时后 TM 直接发 Cancel，但 Try 最终执行了——Cancel 需要能判断是否有 Try 执行过
3. **幂等性**：Confirm/Cancel 可能被多次调用，必须幂等

**幂等方案：** 使用唯一事务ID（XID），每次操作前查询是否已执行。

#### 6.3 Saga 模式

- 将一个长事务分解为多个本地事务 + 补偿事务
- 无锁、高性能、适合长流程
- 编排方式有两种：**Choreography（事件协作）** vs **Orchestration（中央编排）**

```
订单 Saga 示例（Choreography 模式）：
  创建订单 ──► 扣减库存 ──► 扣减账户 ──► 确认订单
     │            │            │            │
   取消订单 ◄── 回滚库存 ◄── 回滚账户 ◄── 回滚订单
```

#### 6.4 本地消息表（eBay 经典方案）

```
┌────────────┐         ┌────────────┐
│ 订单服务    │         │ 积分服务    │
│ ┌────────┐ │         │            │
│ │业务操作 │ │         │            │
│ └───┬────┘ │         │            │
│     │      │         │            │
│     ▼      │         │            │
│ ┌────────┐ │         │            │
│ │本地消息 │ │         │            │
│ │  表插入 │────────►  │            │
│ └────────┘ │         │            │
│ ┌────────┐ │         │            │
│ │定时轮询 │ │ 重试 ──►│            │
│ │未确认消息│ │         │            │
│ └────────┘ │         │            │
└────────────┘         └────────────┘
```

**流程：**
1. 在同一个本地事务中执行业务操作 + 插入消息表
2. 通过定时任务轮询状态为"待发送"的消息
3. 发送消息到 MQ，消费者成功后回调确认
4. 超时未确认则重试，保证最终成功

#### 6.5 Seata 分布式事务框架

**三大核心组件：**

| 组件 | 全称 | 职责 |
|------|------|------|
| **TC** | 事务协调者 | 维护全局和分支事务状态，驱动全局提交/回滚 |
| **TM** | 事务管理器 | 定义全局事务的范围，开启/提交/回滚全局事务 |
| **RM** | 资源管理器 | 管理分支事务的资源，向 TC 注册分支和报告状态 |

**Seata 四种模式：**

| 模式 | 侵入性 | 原理 | 场景 |
|------|--------|------|------|
| **AT** | 低（自动补偿） | 一阶段提交 + undo_log，二阶段自动回滚 | 通用业务，无改造意愿 |
| **TCC** | 高（手动实现） | 程序员实现 Try/Confirm/Cancel | 高并发核心链路 |
| **Saga** | 中（状态机） | 状态机驱动异步执行 | 长流程业务 |
| **XA** | 低（数据库原生） | 数据库级 XA 事务 | 要求强一致性 |

#### 6.6 选型建议

| 场景 | 推荐方案 | 原因 |
|------|---------|------|
| 强一致性 + 简单业务 | Seata AT | 无侵入，自动回滚 |
| 高并发 + 资金链路 | TCC | 性能最优，业务可补偿 |
| 长流程 + 跨系统 | Saga | 无锁、异步、适合编排 |
| 异步通知 + 最终一致 | 本地消息表 / MQ事务消息 | 解耦好，性能高 |
| 被动通知 + 对账兜底 | 最大努力通知 | 通知失败由接收方主动查询 |

## 高频面试题

### 基础级

#### Q1：微服务架构与传统单体架构相比有哪些优缺点？

**答案要点：**
- **优点**：独立部署（不影响其他服务）、技术异构（不同服务可用不同语言/框架）、独立扩展（按需扩容热点服务）、故障隔离（一个服务崩溃不影响整体）、团队自治（小团队独立负责）
- **缺点**：分布式复杂性（网络延迟、数据一致性问题）、运维成本高（服务数量膨胀）、调试困难（跨进程难以断点）、测试复杂度高（需要契约测试）、资源开销大（进程级隔离）

#### Q2：什么是 CAP 定理？在微服务注册中心中如何体现？

**答案要点：**
- CAP 定理：一致性（C）、可用性（A）、分区容错性（P）三者不可兼得，网络分区必须选择 P
- Eureka 选 AP（最终一致，自我保护机制）
- Consul 选 CP（Raft 强一致，分区时少数派不可用）
- Nacos 支持 AP/CP 切换（默认 AP，可用 Distro 协议）
- 注册中心适合选 AP——短暂不一致不影响服务发现正确性

#### Q3：限流算法有哪些？各自优缺点是什么？

**答案要点：**
- 固定窗口（简单但有临界突刺）
- 滑动窗口（精度高、无突刺）
- 漏桶算法（流量严格平滑、无法应对突发）
- 令牌桶算法（支持突发流量、业务首选）
- 生产推荐：**令牌桶** + **滑动窗口**组合使用

#### Q4：简述 gRPC 相对于 REST 的优势。

**答案要点：**
- 基于 HTTP/2 多路复用，支持双向流
- Protobuf 二进制序列化，更小更快
- 强类型接口定义（`.proto` 文件自动生成客户端/服务端）
- 延迟比 HTTP/1.1 REST 低约 30%
- 适用内部服务间高性能通信

#### Q5：什么是链路追踪中的 TraceID 和 Span？

**答案要点：**
- TraceID：全局唯一标识一次完整请求链路，贯穿所有服务
- Span：标识一次操作单元（一个方法调用、一次 DB 查询），有开始/结束时间
- ParentSpanID：构建调用树，形成父子关系
- W3C TraceContext 标准：通过 `traceparent` Header 传递 `version-traceid-spanid-flags`

### 进阶级

#### Q1：服务拆分粒度如何把控？给出一套可操作的判断方法。

**答案要点：**
- **黄金标准**：一个服务能否由 2-5 人的小团队独立维护
- DDD 五步决策法（名词归属/生命周期一致性/事务边界/变更频率/组织架构）
- **不宜过早拆分**——从适度规模的单体起步，根据业务发展逐步拆分
- **不宜拆分过细**——警惕分布式事务成本、调用链复杂、运维负担增大的信号
- 参考指标：每个服务的接口数/表数量在合理范围内（如每服务 5-15 张表）

#### Q2：Eureka 的自我保护机制的原理和优缺点是什么？

**答案要点：**
- **触发条件**：15 分钟内超过 85% 的心跳失败
- **触发后的行为**：不再剔除任何实例（即使真的宕机了），宁可放过不错杀
- **优点**：防止因网络抖动导致大规模误删，集群网络恢复后自动恢复
- **缺点**：宕机的实例不会剔除，消费者可能调用到无效实例
- **适用场景**：网络不稳定的环境；不适用于对服务准确率要求极高的场景

#### Q3：配置中心配置变更后如何实现不重启生效？简述完整链路。

**答案要点：**
- 基于**长轮询（Long Polling）** 机制：
  1. 客户端发起长轮询请求，携带本地配置 MD5
  2. 服务端比对 MD5，无变更则挂起请求（30-60s 超时）
  3. 配置变更时服务端立即返回响应
  4. 客户端拉取最新配置，更新 Spring Environment
  5. `@RefreshScope` 注解的 Bean 重新初始化
  6. 立即发起下一次长轮询
- Nacos 2.x 已升级为 **gRPC 双向流**，实时性更高
- **不生效的常见原因**：静态变量接收配置、未加 `@RefreshScope`、`final` 修饰字段、配置优先级被覆盖

#### Q4：Sentinel 与 Resilience4j 的选型依据是什么？

**答案要点：**
- **选 Sentinel**：Spring Cloud Alibaba 技术栈、需要可视化控制台动态调整规则、需要热点参数限流/链路限流/集群流量管理等高级特性、大流量复杂治理场景
- **选 Resilience4j**：Spring Cloud 官方推荐、追求轻量级模块化、已有注册中心/配置中心不需要 Sentinel 重复功能、中小规模服务
- 两者都能实现熔断降级限流，但 Sentinel 功能更全面、运维更便捷

#### Q5：分布式事务场景下，如何选择 TCC、Saga、Seata AT？

**答案要点：**
- **Seata AT**：无侵入（插入 undo_log），适合通用业务，但性能不如 TCC，低并发的内部系统首选
- **TCC**：业务侵入强（需手动实现 Try/Confirm/Cancel），但性能最优。适合高并发核心链路（如扣库存、扣账户余额）
- **Saga**：无锁、异步、状态机驱动，适合长流程（如旅行预订），支持复杂编排，但补偿逻辑需要仔细设计
- **核心矛盾**：事务的侵入性与性能呈反比——越少的侵入性意味着数据层面的性能开销越大

### 高级级

#### Q1：设计一个支持亿级日活的微服务架构，你会如何从零开始做架构选型和拆分？

**答案要点：**
- **入口层**：CDN -> DNS -> Nginx/LVS 四层负载 -> APISIX/Kong API 网关
- **服务层**：按 DDD 领域拆分为核心域（订单/支付）、支撑域（库存/物流）、通用域（通知/认证）
- **基础设施**：
  - 注册/配置中心：Nacos（AP 模式，多集群部署）
  - 服务通信：内部 gRPC，异步 MQ（Kafka）
  - 限流降级：Sentinel（网关层 + 服务层双层级）
  - 可观测性：OpenTelemetry + SkyWalking + Prometheus + Grafana
- **存储**：每个服务独立 DB，核心链路 Redis 缓存，ES 做宽表异构查询
- **数据一致性**：核心资金链路 TCC，通用场景 MQ 最终一致性
- **部署**：K8s + Istio ServiceMesh（渐进式接入）
- **关键考量**：流量突发时的限流降级策略、异地多活方案、灰度发布机制

#### Q2：微服务架构中，如何保证跨 10+ 个服务的全链路数据一致性？（不限于 TCC/Saga）

**答案要点：**
- **分层递进策略**：
  1. **服务内**：本地事务 + 业务约束，保证领域内强一致
  2. **核心链路**：TCC 或 Seata AT，保证强一致
  3. **非核心链路**：MQ 最终一致性 + 幂等（消息去重表）
  4. **兜底方案**：定时对账 + 补偿 Job + 人工介入
- **设计原则**：
  - 尽量减少跨服务强一致场景——有时可以接受用户看到"处理中"的短暂中间状态
  - 核心-非核心隔离：核心流程不走 MQ（同步调用），非核心走 MQ
  - 幂等设计：每次操作携带幂等键，去重表保证同一条消息只处理一次
  - 对账系统：日终/小时级对账发现不一致并触发自动修复

#### Q3：生产环境中某服务故障导致雪崩，请描述排查过程和根治方案。

**答案要点：**
- **排查流程**：
  1. 查看全局监控大盘，确认故障范围
  2. 查看链路追踪（SkyWalking/Jaeger），定位故障根服务
  3. 查看根服务的 CPU/内存/GC/慢 SQL 和日志
  4. 确认是哪种故障模式（资源耗尽、慢调用积压、下游依赖超时）
- **根治方案**：
  - **熔断**（Sentinel/Resilience4j 配置熔断阈值，防止级联故障）
  - **隔离**（线程池隔离或舱壁隔离，让故障局限在单个线程池内）
  - **限流**（网关层 + 服务层双层限流，入口控制流量）
  - **降级**（非核心功能关闭，返回兜底数据）
  - **容量规划**（定期压测 + 自动扩缩容 + 预热机制）
  - **重试策略**（指数退避 + 随机抖动，防止重试风暴）

#### Q4：在已有的 Spring Cloud Netflix 架构中，如果要迁移到云原生架构（K8s + Istio），需要考虑哪些问题？

**答案要点：**
- **注册中心迁移**：Eureka -> K8s Service + Istio 服务发现。Istio 无需独立注册中心，使用 K8s DNS 或 xDS 协议
- **配置中心迁移**：Spring Cloud Config -> ConfigMap / Nacos / Apollo（取决于是否保留 Java 侧配置）
- **网关迁移**：Zuul/Spring Cloud Gateway -> Istio Ingress Gateway / APISIX
- **熔断降级**：Hystrix -> Istio DestinationRule（流量治理下沉到 Sidecar）+ Sentinel（应用级兜底）
- **链路追踪**：Sleuth+Zipkin -> OpenTelemetry + Jaeger/SkyWalking
- **迁移策略**：渐进式——非核心服务先迁，服务网格 Sidecar 逐步注入，保留双注册中心灰度过渡
- **注意事项**：mTLS 双向认证配置、Sidecar 资源开销（每个 Pod 多一个 Envoy 容器）、网络延迟增加

#### Q5：如何在实现"最终一致性"的同时保障系统的可用性和高吞吐？

**答案要点：**
- **异步化**：核心流程同步 + 非核心流程异步（MQ 削峰填谷）
- **幂等性**：全局唯一幂等号 + 去重表，保证消息最多消费一次
- **可靠投递**：生产端确保消息一定到达 MQ（本地消息表或 MQ 事务消息）
- **消费确认**：手动 ACK + 死信队列兜底
- **最大努力重试**：失败消息按指数退避重试，达到最大次数后进死信队列人工处理
- **对账补偿**：定时对账扫描不一致数据，触发补偿任务
- **多级降级**：正常情况下走异步；写入压力大时降级为只写本地表+定时扫描；极端情况下降级为直接写入 DB 保底
- **设计哲学**：先保证写入成功，后保证状态一致。用户看到"处理中"比"系统错误"更能接受

## 最佳实践

### 生产环境配置清单

#### 注册中心最佳实践

- **Nacos 集群部署**：至少 3 节点（奇数），VIP/VIP4 或 SLB 做负载
- **保护阈值**设置：0.7-0.8（健康实例比例低于此值时返回所有实例）
- **命名空间隔离**：每个环境（dev/test/prod）使用独立的 Namespace
- **临时实例策略**：默认使用临时实例（心跳维持），避免手动管理
- **健康检查**：配置合理的健康检查接口，返回服务真实可用性

#### 网关最佳实践

- **双层级限流**：网关层（全局流控）+ 服务层（精细化流控）
- **限流算法选择**：令牌桶（入口）/ 滑动窗口（精细化接口）
- **鉴权下沉**：统一在网关完成 JWT 校验/OAuth2 认证，去掉服务内部的重复鉴权
- **超时设置**：网关请求超时时间应比服务调用超时时间长，避免误熔断
- **分级限流**：全局限流 -> 应用级限流 -> 用户/IP 级限流

#### 配置中心最佳实践

- **多级缓存**：内存缓存 -> 本地文件缓存（配置中心宕机时兜底）
- **敏感配置加密**：Jasypt + 环境变量传递密钥
- **配置变更审批**：生产环境配置变更走发布审批流程
- **灰度发布**：先灰度（10% 实例）-> 观察 -> 全量发布
- **回滚能力**：每次发布都保留版本，支持一键回滚

#### 服务通信最佳实践

- **内部通信优先 gRPC**：性能更高，接口更规范
- **异步解耦用 MQ**：单一消费者失败不影响其他消费者
- **超时设置**：每个服务调用设置合理超时（通常 50ms-500ms），不同等级服务不同
- **重试机制**：仅对幂等接口开启重试，指数退避 + 随机抖动
- **熔断配置**：动态配置熔断阈值，结合链路追踪实时调整

#### 链路追踪最佳实践

- **采样策略**：错误 Trace 100% 采集 + 正常 Trace 按比例（如 1-10%）
- **多级采样**：Collector 做 Tail Sampling 兜底
- **Tag 规范化**：限制自定义 Tag 的数量和基数，避免高基数标签导致存储膨胀
- **保留策略**：热数据（7天）ES 快速查询，冷数据（30天）归档对象存储
- **告警关联**：Trace 关联告警，点击跳转到对应链路的详细追踪页面

#### 数据一致性最佳实践

- **"能不分布式事务就不分布式事务"**——核心设计哲学
- **核心链路选 TCC**：支付、扣库存等需要高一致性的场景
- **非核心链路走 MQ**：发短信、积分、通知等
- **对账系统兜底**：每天定时对账，发现不一致触发修复
- **幂等设计必须全面**：每个接口都需要考虑幂等性

### 常见反模式（Anti-Patterns）

| 反模式 | 描述 | 后果 | 正确做法 |
|--------|------|------|---------|
| 过度拆分 | 每个类一个微服务 | 通信成本爆炸，延迟高 | 业务聚合，2-5人团队能维护的粒度 |
| 共享数据库 | 多个服务直连同一库 | 服务间强耦合，无法独立演进 | 每个服务独享数据库 |
| 忽视可观测性 | 只有日志没有链路/监控 | 故障时无从下手排查 | 日志+指标+链路三件套 |
| 分布式事务滥用 | 所有跨服务都走分布式事务 | 性能极差，吞吐量断崖下降 | 尽量减少强一致场景 |
| "为微服务而微服务" | 小项目盲目上微服务 | 复杂度远超收益 | 从单体起步，按需演进 |
| 网关太重 | 网关塞入过多业务逻辑 | 网关成为瓶颈和单点故障 | 网关只做路由/鉴权/流控 |
| 重试风暴 | 所有调用方同时重试 | 放大了故障范围 | 指数退避 + Jitter + 限流 |
| 无健康检查 | 注册中心不知道服务真实状态 | 流量打到不可用实例 | 配置合理的健康检查接口 |

### 设计决策清单

以下是在架构设计中需要反复权衡的核心决策点：

1. **服务粒度**：服务太大 -> 难以演进；服务太小 -> 通信成本高。判断标准：团队能否独立维护
2. **一致性 vs 可用性**：核心链路强一致 vs 非核心链路最终一致
3. **同步 vs 异步**：同步理解简单但耦合高；异步解耦好但调试困难
4. **有状态 vs 无状态**：优先无状态，方便水平扩展
5. **API 优先 vs 代码优先**：gRPC 天然 API 先定义；REST 需要额外规范
6. **数据库共享 vs 分离**：一个服务一个数据库，禁止直接跨服务访问数据库
7. **事件驱动 vs 请求驱动**：请求驱动理解简单；事件驱动解耦更好
8. **编排 vs 编排（Orchestration vs Choreography）**：集中编排易于管理但存在单点；事件协作更灵活但难追踪
9. **单体优先 vs 微服务优先**：对业务理解不足时先单体，分清楚边界后再拆

## 进阶拓展

### ServiceMesh + Istio

#### 什么是 Service Mesh

Service Mesh（服务网格）是将微服务通信逻辑从应用代码中提取到基础设施层的架构模式。核心思想是**Sidecar 代理**模式——每个服务实例旁边部署一个代理（通常是 Envoy），所有进出流量都经过这个代理。

```
传统微服务（SDK 模式）：                Service Mesh（Sidecar 模式）：
┌─────────────┐                       ┌─────────────┐
│ 服务 A       │                       │ 服务 A + Sidecar│
│ 注册发现      │                       │              │
│ 熔断降级      │                       │   sidecar ◄───► sidecar
│ 负载均衡      │                       │  (Envoy)    │  (Envoy)
│ 链路追踪      │                       └─────────────┘  └─────────────┘
│ 配置管理      │                           管理面 (Control Plane)
└─────────────┘                                │
                                           Istiod (Pilot+Galley+Citadel)
```

#### Istio 架构组成

| 组件 | 职责 | 说明 |
|------|------|------|
| **Envoy Proxy**（数据面） | 流量代理 | 每个 Pod 的 Sidecar，拦截所有进出流量 |
| **Pilot**（控制面） | 服务发现 + 流量管理 | 将路由规则分发到 Envoy |
| **Citadel**（控制面） | 安全 | mTLS 证书签发和轮换 |
| **Galley**（控制面） | 配置验证和分发 | 验证 Istio CRD 配置 |
| **Kiali**（可观测） | 可视化服务拓扑 | 服务间调用关系图 |
| **Jaeger/Zipkin**（可观测） | 链路追踪 | 分布式链路追踪集成 |

#### Istio 的核心能力

- **流量管理**：金丝雀发布、灰度路由、流量镜像、熔断器、超时重试
- **安全**：mTLS 自动双向认证、AuthorizationPolicy 细粒度访问控制
- **可观测性**：自动生成 Metrics/Traces/Logs，无需修改代码
- **多集群/多网络**：跨集群服务发现和流量治理

#### Istio 核心 CRD 资源

| CRD | 用途 |
|-----|------|
| VirtualService | 定义路由规则：匹配条件->目标子集 |
| DestinationRule | 定义流量策略：负载均衡、熔断、连接池 |
| Gateway | 定义入口/出口网关 |
| ServiceEntry | 将外部服务注册到网格 |
| PeerAuthentication | 服务间 mTLS 配置 |
| AuthorizationPolicy | 细粒度访问控制 |

**VirtualService + DestinationRule 示例：**
```yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: reviews
spec:
  hosts:
  - reviews
  http:
  - match:
    - headers:
        end-user:
          exact: jason
    route:
    - destination:
        host: reviews
        subset: v2
  - route:
    - destination:
        host: reviews
        subset: v1
---
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: reviews
spec:
  host: reviews
  trafficPolicy:
    connectionPool:
      tcp:
        maxConnections: 100
    outlierDetection:
      consecutive5xxErrors: 5
      interval: 30s
  subsets:
  - name: v1
    labels:
      version: v1
  - name: v2
    labels:
      version: v2
```

#### Service Mesh 与 API Gateway 的关系

| 维度 | API Gateway | Service Mesh（Istio） |
|------|------------|----------------------|
| 流量方向 | 南北向（外部->内部） | 东西向（内部服务之间） |
| 代理位置 | 集群入口 | 每个 Pod 内 Sidecar |
| 管理方式 | 配置中心/控制台 | 声明式 CRD 配置 |
| 改造难度 | 接入即可 | 需 K8s + 网格基础 |
| 侵入性 | 低 | 代码零侵入 |

**最佳组合**：API Gateway（APISIX/Kong）处理南北向流量 + Istio 处理东西向流量。

### 云原生微服务

#### 云原生 vs 传统微服务

| 维度 | 传统微服务 | 云原生微服务 |
|------|-----------|-------------|
| 部署平台 | 虚拟机 | 容器（Docker + K8s） |
| 服务发现 | Eureka/Nacos/Consul | K8s Service + Istio |
| 配置管理 | 配置中心 | ConfigMap + 配置中心 |
| 网关 | Spring Cloud Gateway/Kong | Istio Ingress Gateway + APISIX |
| 弹性伸缩 | 手动/脚本 | HPA + 自动伸缩 |
| 可观测性 | 独立搭建 | Prometheus + Grafana + OTel |
| 基础设施管理 | 运维手动 | IaC（Terraform/Pulumi） |

#### 云原生架构的演进路线

```
第一阶段：容器化
  应用打包为 Docker 镜像，K8s 部署
  引入：Docker + K8s + HPA

第二阶段：服务治理下沉
  将注册发现、熔断降级从 SDK 下沉到 Sidecar
  引入：Istio Service Mesh

第三阶段：可观测性标准化
  统一 Metrics/Logs/Traces 采集
  引入：OpenTelemetry + Prometheus + Loki + Tempo

第四阶段：GitOps + 平台工程
  声明式基础设施，内部开发者平台
  引入：ArgoCD/Flux + Backstage/Terraform

第五阶段：Serverless + eBPF
  极致弹性，无服务器体验
  引入：Knative + eBPF 网格优化
```

#### 2025-2026 趋势

1. **Ambient Mesh**：Istio 无 Sidecar 模式，使用每个节点的代理（ztunnel），降低资源开销
2. **Gateway API**：Kubernetes 官方网关标准，统一 Ingress + Service Mesh 路由
3. **eBPF 优化**：使用 eBPF 技术替换 Envoy 部分功能，提升 Service Mesh 性能
4. **多集群/多网格**：跨数据中心/跨云的统一治理
5. **平台工程**：构建内部开发者平台（IDP），降低微服务开发门槛
6. **AI 驱动的运维**：AI 异常检测、智能扩缩容、智能故障定位
