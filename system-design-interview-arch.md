# 高级系统设计面试知识架构

> 以"四步方法论 + 六大经典设计题 + 架构原则 + 进阶拓展"为骨架，构建系统设计面试的完整知识体系。

---

## 基础概念

### 系统设计面试的本质

系统设计面试考察的不是"标准答案"，而是候选人的**结构化思维、工程权衡能力和沟通协作水平**。面试官关注的核心维度：

| 考察维度 | 说明 |
|---------|------|
| 结构化思维 | 能否从模糊需求梳理出清晰的架构脉络 |
| 容量感 | 是否具备量化的数字直觉（QPS、存储、带宽） |
| 权衡能力 | 面对多个方案能否理性分析优缺和取舍 |
| 广度与深度 | 是否熟悉常见组件（Redis、MQ、DB）的原理与边界 |
| 沟通表达 | 能否清晰地画图、写伪代码、讲推理过程 |

### 系统设计的三种常见题型

| 题型 | 特点 | 代表题目 |
|------|------|---------|
| 高并发型 | 峰谷差异大，QPS 极高 | 秒杀、直播、抢红包 |
| 海量数据型 | 存储量大，关注数据模型和分片 | 飞书/Notion类文档、评论系统 |
| 复杂逻辑型 | 业务规则多，状态转换复杂 | 订票系统、支付系统、审批流 |

## 核心原理（设计方法论）

### 系统设计四步法

业界广泛认可的系统设计面试框架（源自 ByteByteGo / Alex Xu）：

```
第一步：需求澄清（3-5 分钟）
第二步：高层架构与容量估算（5-10 分钟）
第三步：深入设计核心模块（10-20 分钟）
第四步：总结与瓶颈分析（3-5 分钟）
```

#### 第一步：需求澄清

在动手之前，必须明确两个维度：

- **功能性需求**：
  - 核心功能是什么？（例如短链：生成短链 + 重定向访问）
  - 有哪些边界场景？（自定义短链、过期策略、统计）
- **非功能性需求**：
  - QPS 目标（日均/峰值）
  - 延迟要求（P99 < 100ms）
  - 数据一致性级别（强一致 / 最终一致）
  - 可用性要求（99.9% / 99.99%）

> **常见陷阱**：跳过澄清直接画架构图，导致方向错误。

#### 第二步：高层架构与容量估算

**信封背面估算（Back-of-the-envelope）**：

估算入口（以 Twitter / 微博为例）：

```
日活跃用户（DAU）：2 亿
每个用户每天发 1 条 Feed → 2 亿条/天
每个用户每天刷 100 次 Timeline → 200 亿次/天
写入 QPS ≈ 2 亿 / 86400 ≈ 2300
读取 QPS ≈ 200 亿 / 86400 ≈ 23 万
峰值 QPS ≈ 平均 × 5～10 倍
存储估算：
  每条 Feed 约 1KB → 2 亿 × 1KB ≈ 200GB/天
  5 年 ≈ 365TB
带宽估算：
  写入带宽 ≈ 2300 × 1KB ≈ 2.3MB/s
  读取带宽 ≈ 23 万 × 1KB ≈ 230MB/s
```

**关键数字速查表**：

| 场景 | 典型数值 |
|------|---------|
| MySQL 单表瓶颈 | 500 万～1000 万行 |
| 单机 QPS（通用业务） | 500～2000 |
| Redis 单实例 QPS | 5万～10万 |
| Redis 复杂 Lua 脚本 QPS | 5000～20000 |
| Kafka 单分区吞吐 | ~10MB/s |
| Nginx 单机连接数 | 5万～10万 |
| 一次网络往返（同机房） | ~0.5ms |
| B+树 3 层索引查询 | ~10ms |
| 1 TB 硬盘顺序读 | ~200MB/s |

#### 第三步：深入设计核心模块

一般选取 **1-2 个核心模块** 展开深入讨论，而非平均用力。

**衡量标准**：哪些模块是这个系统的"核心挑战"？

- 秒杀系统 → 库存扣减 + 流量整形
- IM 系统 → 长连接管理 + 消息可靠性
- Feed 流系统 → 推拉模型选择 + 明星热点处理
- 短链系统 → 发号器 + 高性能重定向

#### 第四步：总结与瓶颈分析

主动指出系统中仍存在的瓶颈及优化方向：

1. **单点故障**（SPOF）：是否有单点？如何消除？
2. **数据一致性**：最终一致性是否满足业务？是否需要补偿？
3. **扩展性**：能否通过加机器线性提升性能？
4. **监控与告警**：关键指标如何监控？链路追踪？

### 通用架构模式速查

| 模式 | 适用场景 | 关键组件 |
|------|---------|---------|
| 分层架构 | 大多数系统 | 接入层 → 业务层 → 数据层 |
| 微服务拆分 | 大型复杂业务 | API Gateway, 服务注册发现 |
| CQRS | 读写负载差异大 | 写入端 RDB/事件存储, 读取端 ES / 读视图 |
| 事件驱动 | 跨服务解耦 | 事件总线, 订阅者 |
| 读写分离 | 读多写少 | 主库写 / 从库读, binlog 同步 |
| 分片 | 单表数据超千万 | 按 ID 哈希 / 按时间范围 |
| 多级缓存 | 读密集、延迟敏感 | L1 本地缓存 → L2 Redis → L3 DB |

## 高频面试题（6 个经典设计题）

### 一、秒杀系统

#### 设计要点

**核心问题**：少量的商品库存被大量用户抢夺，在极短时间内产生极高流量。

**架构分层**（自顶向下）：

```
前端层       → 按钮防抖、验证码、CDN 静态化
接入层       → Nginx+Lua 限流、IP 黑名单、请求签名鉴权
业务层       → Redis+Lua 原子扣库存、MQ 异步下单、DB 持久化
数据层       → 分库分表、读多写少、最终一致性对账
```

**库存扣减核心方案对比**：

| 方案 | 抗压能力 | 一致性 | 复杂度 | 推荐场景 |
|------|---------|--------|--------|---------|
| 数据库乐观锁（CAS） | 500～2000 QPS | 强 | 低 | 低并发，小规模 |
| Redis+Lua 原子扣减 + MQ 异步下单 | 5000～20000+ QPS | 最终 | 中 | 标准秒杀 |
| 分段锁（Segmented Locking） | 5万～10万 QPS | 最终 | 高 | 极高并发爆款商品 |

**Redis+Lua 原子扣减核心逻辑**：

```lua
-- KEYS[1]: 库存 key, ARGV[1]: 扣减数量
local stock = tonumber(redis.call('GET', KEYS[1]) or '0')
if stock >= tonumber(ARGV[1]) then
    redis.call('DECRBY', KEYS[1], ARGV[1])
    return 1   -- 成功
end
return 0       -- 库存不足
```

**流量削峰的三层防线**：

1. **验证码/答题** → 人机识别，打散请求到达时间
2. **令牌桶/漏桶限流**（网关层） → 控制入口流量
3. **MQ 异步削峰** → 请求入队即返回"排队中"，消费者拉模式消费

#### 关键数字

| 指标 | 数值 |
|------|------|
| 商品库存 | 通常 1000～10000 件 |
| 瞬时并发 | 10万～100万（峰谷比可达 1000:1） |
| Redis 单实例抗压 | 5万 QPS（简单命令） |
| MQ 单机削峰能力 | 5万～10万条/秒 |
| 订单超时释放时间 | 15～30 分钟 |
| 支付率 | 60%～80%（未支付订单需释放库存） |

#### 实战追问清单

- **超卖如何防止？** → Redis+Lua 原子操作保证扣减不超库存
- **少卖如何防止？** → 定时补偿任务扫描超时未支付订单，回滚 Redis 库存
- **"少卖"场景：Redis 扣减成功但服务崩了怎么办？** → MQ 确保最终落库，失败有重试和补偿
- **Redis 挂了怎么办？** → 降级：入口拦截 99% 流量，1% 走数据库悲观锁备选路径
- **如何防一人多刷？** → 用户维度的分布式锁 + 幂等性 Token + 数据库唯一索引
- **热点 SKU 怎么抗？** → 分段锁（`skuId mod N` 分散到 N 个 key）
- **订单超时怎么处理？** → RabbitMQ TTL 死信队列 / Redis ZSET 过期检测（不要做全表扫描）

### 二、IM 即时通讯系统

#### 设计要点

**核心问题**：消息的实时性、可靠性、有序性，以及群聊场景下的扩散风暴。

**四层架构**：

```
接入层（Connection）  → 长连接网关集群，百万级连接管理
路由层（Router）      → 一致性哈希路由，用户状态管理（在线/离线）
消息层（Logic）       → 消息推送、群发扩散、ACK 确认
存储层（Storage）     → 消息持久化、冷热分离
```

**长连接管理**：

| 技术选型 | 优劣 |
|---------|------|
| WebSocket | 兼容 HTTP 协议、浏览器友好、全双工 |
| TCP 长连接（Netty / Go netpoll） | 性能更高、控制力强，需自定义应用层协议 |
| MQTT | IoT 场景标准，轻量级，QoS 分级 |

**关键机制**：

- **心跳检测**：30～60 秒间隔，自适应心跳（根据 NAT 淘汰时间调整），超时剔除
- **断线重连**：指数退避（1s → 2s → 4s → 8s → 上限 60s），带随机偏移防雪崩
- **连接分组**：一致性哈希 + 连接池，便于定点推送

**消息可靠性（三段 ACK）**：

```
客户端 A → 接入层（ACK-1）
接入层 → 消息服务层（ACK-2）
消息服务层 → 目标客户端接入层 → 目标客户端（ACK-3）
```

**可靠性保障组合**：

| 场景 | 方案 |
|------|------|
| 消息不丢 | 服务端持久化 + 客户端 ACK + 重试 |
| 消息不重 | 全局唯一 MessageID + Redis 或 DB 去重索引 |
| 消息不乱 | 单调递增 seqID（每个会话独立），服务端按序投递 |
| 消息必达 | 离线存储 + 上线后拉取 + 推送补全 |

**已读/未读设计**：

- **单聊**：最后一条已读消息的 seqID 同步给对方
- **群聊**：Redis ZSET 存储每个群成员的"已读游标"，ZSET 设置最大条目上限（如 100 条）
- **未读数统计**：Redis 计数器，点进会话后清零

**群聊扩散模型**：

| 模型 | 原理 | 适用场景 | 优缺点 |
|------|------|---------|--------|
| 写扩散 | 发送者写一条消息到群，服务端复制到每个成员的收件箱 | 小群（≤1000 人） | 读快写慢，群人数增加时扩散放大严重 |
| 读扩散 | 发送者写一条消息到群消息表，成员拉取时按时间线聚合 | 大群（万人以上） | 写快读慢，读取时需合并 |
| 混合模式 | 活跃成员写扩散，非活跃/大群成员读扩散 | 大中小群通用 | 兼顾读写性能，实现复杂 |

**存储设计**：

```
消息表（Message）：
  message_id (PK, BIGINT)
  session_id (VARCHAR, 会话/群ID)
  sender_id  (BIGINT)
  content    (TEXT)
  seq_id     (BIGINT, 会话内单调递增)
  created_at (DATETIME)

收件箱表（Inbox）—— 写扩散模式：
  user_id    (PK)
  message_id (PK)
  session_id
  seq_id
  is_read    (BOOL)

索引：session_id + seq_id → 拉取消息
```

#### 关键数字

| 指标 | 数值 |
|------|------|
| 单机长连接数 | 10万～50万（取决于内存和 OS 配置） |
| 心跳间隔 | 30～60s |
| 消息延迟目标 | P99 < 100ms |
| 消息体大小 | < 64KB（大文件走对象存储） |
| 写扩散上限 | 群人数 ≤ 1000 |
| 读扩散推荐 | 群人数 > 10000 |
| 离线消息保留 | 7～30 天 |

#### 实战追问清单

- **群聊"消息风暴"怎么解决？** → 大群用读扩散 + 消息队列广播 + 本地缓存
- **消息有序性怎么保证？** → 单会话内 serial 队列投递，seqID 在存储层保证单调递增
- **百万并发登录怎么抗？** → 分批登录（客户端随机延迟 1-5s），Redis 连接池预热
- **客户端网络切换时怎么处理？** → 连接迁移 + seqID 续传，断线重连后拉取断点后的消息
- **发送方撤回消息如何处理？** → 服务端下发撤回指令（非物理删除），更新已投递消息状态

### 三、Feed 流系统

#### 设计要点

**核心问题**：用户刷 Timeline 时如何快速聚合已关注用户的动态，并处理好"明星/大 V"带来的写入放大。

**推拉模型对比**：

| 对比项 | 推模式（写扩散） | 拉模式（读扩散） | 混合模式 |
|--------|----------------|----------------|---------|
| 写入路径 | 发帖时写入每个粉丝的收件箱 | 发帖时只写自己时间线 | 普通用户推，大V 用拉 |
| 读取路径 | 直接从收件箱读取 | 读取时合并所有关注对象 | 合并收件箱 + 大V 拉取 |
| 粉丝少时 | 表现良好 | 表现一般 | 表现良好 |
| 明星发帖 | 大量写入（写风暴） | 读取时轻微 | 明星走拉模式 |
| 新鲜度 | 高 | 中（取最新 N 条） | 高 |

**混合模式具体实现**：

```
混合模式（Hybrid）：
  1. 普通用户（粉丝 < 1 万）→ 推模式写入粉丝收件箱
  2. 明星大V（粉丝 > 1 万）→ 拉模式，不写入粉丝收件箱
  3. 用户刷 Timeline 时：
     a) 从自己的收件箱取出"普通关注人"的 Feed（已有）
     b) 从大V 的发布列表中拉取最新 N 条
     c) 按时间合并排序
```

**缓存策略**：

```
L1: 本地缓存（Caffeine）→ L2: Redis SortedSet（ZSET）
  - Score = 时间戳
  - Member = FeedID
  - TTL = 7 天 / 按活跃度淘汰
```

- **游标分页**：使用 `last_timestamp + last_feed_id` 防止分页跳跃
- **NoMore 标记**：SortedSet 中放入特殊标记表示"数据库无更多数据"，防止缓存穿透
- **空值缓存**：布隆过滤器快速判断 Key 是否存在

**热点 Feed 处理（明星效应）**：

```
场景：某明星发一条微博，1 亿粉丝要看到
问题：推模式需要 1 亿次写入，写入风暴

解决方案：
  1. 明星发帖 → 写入明星的"发布列表"（Redis List / MySQL）
  2. 粉丝拉 Timeline 时 → 从明星发布列表拉取最新动态（拉模式）
  3. 活跃粉丝（近 7 天登录过的）→ 也可以选择推模式写入收件箱
  4. 非活跃粉丝 → 回归时再拉取
```

#### 关键数字

| 指标 | 数值 |
|------|------|
| DAU（以微博/Twitter 为参考） | 1亿～3亿 |
| 写入 QPS（按每个用户每天 1 条） | ~2300 |
| 读取 QPS（按每个用户每天刷 100 次） | ~23万 |
| 收件箱 TTL | 7 天 |
| 单条 Feed 存储大小 | ~1KB |
| 推模式粉丝上限 | 10000 人 |
| 混合模式里拉取明星的 QPS | 取决于明星数量 × 粉丝刷 Timeline 频率 |

#### 实战追问清单

- **如何防"僵尸粉"发 Feed 冲垮收件箱？** → 粉丝过滤，只推给最近活跃用户
- **收件箱太大怎么处理？** → SortedSet 设置上限（保留最近 N 条），数据库兜底
- **明星发布延迟怎么保证？** → 明星走拉模式，粉丝刷 Timeline 时实时拉取明星动态
- **Feed 被删除后怎么让全量粉丝知道？** → 消息队列广播删除指令，收件箱标记删除（软删除）
- **系统刚上线冷启动怎么做缓存预热？** → 按粉丝活跃度排序，分批预热 Top N 用户的收件箱

### 四、短链系统

#### 设计要点

**核心问题**：快速将一个长 URL 转换为短字符串，并能在高并发访问时快速重定向。

**核心流程**：

```
1. 用户访问短链（https://short.url/abc123）
2. 短链服务根据短码查询映射关系
3. 返回 301/302 重定向到原始长 URL
4. 浏览器跳转到目标页面
```

**301 vs 302 重定向**：

| 方向 | 含义 | 缓存 | 统计能力 | 推荐场景 |
|------|------|------|---------|---------|
| 301 | 永久重定向 | 浏览器缓存，下次不再请求短链服务 | 无法统计 | 不关心分析的场景 |
| 302 | 临时重定向 | 每次请求都会到短链服务 | 可统计点击数、来源、设备 | 需要数据分析的通用场景 |

> **面试结论**：大多数商业场景选择 **302**，因为需要点击数据分析。纯工具类场景可选 301。

**发号器方案对比**：

| 方案 | QPS | 优缺 | 推荐 |
|------|-----|------|------|
| 数据库自增 ID | 几百 QPS | 实现简单，性能最低 | 低并发 |
| Redis INCR | 5万+ QPS | 远程调用有网络开销 | 中等并发 |
| **号段模式**（预取号段） | **10万+ QPS** | 本地发号，内存缓存，DB 兜底 | ✅ 推荐 |
| **雪花算法**（Snowflake） | **400万+ QPS** | 去中心化，趋势递增，本地生成 | ✅ 推荐 |
| UUID | 极高 | 无序，太长，不适合索引 | ❌ 不推荐 |

**Base62 编码**：

```
62^6 ≈ 568 亿个组合 → 6 位短码足够

编码表：0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz

举例：ID=123456789 → Base62 编码后约 4-5 位字符
```

**高性能优化策略**：

| 层级 | 方案 |
|------|------|
| 发号器 | 雪花算法本地生成，零 IO |
| 重定向 | 多级缓存：CDN → Redis（热点短链） → 本地 Caffeine |
| 防穿透 | 布隆过滤器（Bloom Filter）判断短码是否存在 |
| 数据分片 | 按短码哈希分库分表 |
| 无状态 | 水平扩展，Nginx 做负载均衡 |

#### 关键数字

| 指标 | 数值 |
|------|------|
| 短码长度 | 6～8 位（568 亿～218 万亿） |
| 单次重定向延迟 | < 10ms（缓存命中） |
| 发号器 QPS（单机雪花） | ~400 万 |
| 布隆过滤器误判率 | 1% 以内（合理位数组大小） |
| 数据库单表上限 | 500 万～1000 万行 |
| redis 缓存全部热码 | 亿级短链轻松容纳 |

#### 实战追问清单

- **同一个长 URL 是否生成相同短链？** → 需要去重，查映射表或布隆过滤器；也可不做（用户不关心）
- **哈希冲突怎么解决？** → MurmurHash + 冲突时拼接特殊字符串重算 + 唯一索引兜底
- **短链过期怎么办？** → 过期时间字段 + 定时清理 + 布隆过滤器重建
- **如何防止恶意生成？** → 用户维度限流 + 白名单 + 人工审核
- **自定义短链怎么实现？** → 用户直接指定短码，校验唯一性后写入

### 五、分布式 ID 生成器

#### 设计要点

**分布式 ID 四大要求**：

1. **全局唯一**：分布式环境下不重复
2. **趋势递增**：适配数据库 B+ 树索引，减少页分裂
3. **高可用**：无单点故障，支持水平扩展
4. **高性能**：单机万级以上 QPS

**雪花算法（Snowflake）结构**（64 位 Long 型）：

```
 0 ─ 0000000000 0000000000 0000000000 0000000000 0 ─ 00000 ─ 00000 ─ 000000000000
|1| ───── 41 位时间戳 ───── |5位|5位| ── 12 位序列号 ──
```

| 位段 | 长度 | 说明 |
|------|------|------|
| 符号位 | 1 bit | 固定为 0，保证正数 |
| 时间戳 | 41 bit | 毫秒级，从自定义纪元起可用约 69 年 |
| 机器 ID | 10 bit | 5 位数据中心 + 5 位节点，最大 1024 个实例 |
| 序列号 | 12 bit | 单毫秒内自增，最大 4096 个/ms |

**单节点 QPS**：4096 × 1000 = **~400 万**

**号段模式（Segment Pattern）**：

```
核心原理：数据库预分配连续 ID 区间，本地内存自增发号
            ↓
       db.update max_id = max_id + step
            ↓
       本地分配 [old_max_id+1, old_max_id+step]
```

```sql
CREATE TABLE id_segment (
    biz_tag     VARCHAR(64) PRIMARY KEY,
    max_id      BIGINT NOT NULL,
    step        INT NOT NULL DEFAULT 1000,
    update_time DATETIME
);
```

**号段模式的优化 - 双 Buffer 预加载**：

```
当前号段用到 10% → 后台线程异步加载下一个号段
                   ↓
消除 IO 阻塞毛刺，TP999 < 1ms
```

**三大工业级方案对比**：

| 方案 | 核心技术 | 时钟回拨 | QPS | 依赖 | 推荐 |
|------|---------|---------|-----|------|------|
| 原生雪花 | 时间戳 + 机器ID + 序列号 | 极高风险 | 400万+ | 无 | ❌ |
| **Leaf-segment**（美团） | 双 Buffer 号段 + 动态步长 | 无此问题 | 100万+ | MySQL | ✅✅ |
| **Leaf-snowflake**（美团） | ZK 自动分配 WorkerID + 回拨防护 | 极低 | 100万+ | ZK + MySQL | ✅✅ |
| **CachedUidGenerator**（百度） | RingBuffer 无锁预生成，大 WorkerID | 极低 | 600万+ | MySQL | ✅✅✅ |
| 原生号段 | 数据库预分配 | 无此问题 | ~10万 | MySQL | 低流量场景 |

**时钟回拨解决方案全图**：

```
时钟回拨 NTP 修正策略：
  1. 使用 NTP 的 slew 模式（平滑调整）而非 step 模式（跳跃回拨）
  
当回拨已发生后：
  2. 轻微回拨（≤5ms） → 阻塞等待时间追上
  3. 中等回拨 → 换绑机器ID（重启重新分配 WorkerID）
  4. 严重回拨 → 拒绝服务 + 报警 + 手动介入

美团 Leaf 三层防御：
  Layer 1：启动校验 — 系统时间 < ZK 持久化历史时间 → 拒绝启动
  Layer 2：运行时等待 — 回拨 ≤5ms 时 sleep 等待时间追平
  Layer 3：持久化备份 — ZK 上持久化最大时间戳，回拨到备用方案
```

#### 关键数字

| 指标 | 数值 |
|------|------|
| 雪花算法单节点 QPS | ~400 万 |
| 雪花算法支持节点数 | 1024 |
| 时间戳可用年限 | ~69 年（41 bit） |
| Leaf-segment 延迟（TP999） | < 1ms |
| CachedUidGenerator QPS | 600 万+ |
| UUID 与 Long ID 写入性能差异 | UUID 比 Long 慢 80%+（因 B+ 树页分裂） |

#### 实战追问清单

- **雪花算法能生成多少 ID？** → 4096/ms/节点，400万 QPS/节点，支持 1024 节点
- **为什么 UUID 不适合做主键？** → 36 位无序字符串，B+ 树频繁页分裂，写入性能差
- **时钟回拨怎么解决？** → 启动校验 + 运行时等待 + 持久化 + 备用方案（5 层防御）
- **号段模式为什么用双 Buffer？** → 消除 IO 阻塞毛刺，实现 TP999 < 1ms
- **美团 Leaf 和百度 UidGenerator 的区别？** → Leaf 更稳（号段+雪花），UidGenerator 更快（600万 QPS）

### 设计决策清单 Checklist

在面试中每做一个决策前，快速自问：

```
[ ] 这个决策解决了什么具体问题？
[ ] 它引入了什么新风险？（复杂度、一致性、成本）
[ ] 在什么条件下需要改变这个决策？
[ ] 有没有更简单的替代方案？
```

### 架构韧性检查清单

```
[ ] 单点故障消除：每个组件是否有冗余？宕机一个是否影响整体？
[ ] 降级策略：依赖服务变慢时是否能回退到缓存或部分结果？
[ ] 恢复机制：服务恢复上线时，如何防止被积压请求冲垮？
[ ] 数据一致性：最终一致性业务是否可以接受？补偿机制是否就绪？
[ ] 监控与可观测性：关键指标（QPS、延迟、错误率）是否可追踪？
[ ] 限流与熔断：突发流量是否有第一道防线？
```

### 常见陷阱

| 陷阱 | 表现形式 | 正确做法 |
|------|---------|---------|
| 跳过需求澄清 | 直接画图，方向错误 | 花 5 分钟提问明确功能和性能需求 |
| 平均用力 | 每个组件都讲同样深度 | 找到核心挑战深入展开 1-2 个模块 |
| 缺乏数字感 | "高并发"、"海量数据"无数字 | 做信封背面估算，用 QPS/存储/带宽驱动决策 |
| 过度设计 | 小系统用 CQRS + ES + 微服务 | MVP 先用最简单的方案，指出后续优化方向 |
| 死记硬背 | 背诵标准答案，不展示思考 | 把面试当作对话，主动分析 trade-off |
| 忽略边界场景 | 只讨论正常流程 | 主动提及超时、重试、幂等、补偿 |
| 不响应 hints | 面试官给提示后仍固执己见 | 认真倾听 hint，它是方向的指引 |

### 面试加分行为

1. **主动做估算**：在高层设计完成后，主动给出 QPS、存储、带宽等关键数字
2. **主动谈权衡**："我选择方案 A 是因为 XXX，如果场景变成 YYY，我会换成方案 B"
3. **主动问边界**：面试官给出需求后，追问"是否有大小限制？是否考虑过期？是否需要统计？"
4. **画清架构图**：用清晰的层级结构展示组件和交互关系
5. **诚实承认盲区**："这部分我没有生产经验，但我认为应该考虑 XYZ"

## 进阶拓展

### DDD（领域驱动设计）落地实践

**战略设计的四步法（Event Storming 工作坊）**：

```
Step 1: 列出领域事件（Domain Events）—— 发生了什么？
Step 2: 找到触发事件的命令（Command）—— 谁触发的？
Step 3: 识别聚合根（Aggregate）—— 哪个实体负责？
Step 4: 划分限界上下文（Bounded Context）—— 如何分组？
```

**战术设计核心要素**：

| 概念 | 说明 | 示例 |
|------|------|------|
| 实体（Entity） | 有唯一标识，可变 | 订单（Order），通过 ID 区分 |
| 值对象（Value Object） | 无唯一标识，不可变，描述属性 | 地址（Address）、金额（Money） |
| 聚合根（Aggregate Root） | 实体中负责保护业务不变量的根节点 | 订单聚合中 Order 是根 |
| 领域事件（Domain Event） | 发生在领域内的业务事实 | OrderCreated, PaymentReceived |
| 限界上下文 | 模型应用的边界 | 订单上下文 / 库存上下文 |

**DDD 分层架构**：

```
应用层（Application）     → 业务流程编排，事务控制
领域层（Domain）          → 核心业务逻辑，聚合根、领域服务
基础设施层（Infrastructure）→ 技术实现：DB、MQ、缓存
接口层（Interface）       → REST API、gRPC 端点
```

**落地效果参考**（某 OTA 平台案例）：需求响应从 3 个月缩短至 2 周，可用率 99.99%

### 事件驱动架构

**核心模式**：

```
服务 A 产生事件 → 事件总线（Kafka / RocketMQ）→ 服务 B、C、D 订阅处理
```

**Transactional Outbox（发件箱模式）**：

```
问题：DB 操作和 MQ 发送无法在一个分布式事务中完成
方案：
  1. 在本地事务中将"事件"写入 outbox 表
  2. 后台进程（CDC / 定时扫描）读取 outbox 表
  3. 确保事件被发送到 MQ
  4. 消费方幂等处理
```

**SAGA 模式（长事务补偿）**：

| SAGA 类型 | 原理 | 适用 |
|-----------|------|------|
| 编排式（Choreography） | 每个服务发布事件触发下一个 | 简单链路 |
| 协调式（Orchestrator） | 中心协调器控制流程 | 复杂业务、有分支 |

**CQRS + Event Sourcing 组合**：

```
Command API → 事件存储（Event Store）→ 事件处理器 → 读模型（Read View）→ Query API
                        ↓
                最终一致性：读模型可能延迟更新
```

**DDD + 事件驱动典型架构**：

```
                       ┌──────────┐
                       │  API GW  │
                       └────┬─────┘
                     ┌──────┴──────┐
                     │  Application │
                     │  (编排层)    │
                     └──────┬──────┘
                     ┌──────┴──────┐
                     │  Domain     │ Aggregate → Domain Events
                     └──────┬──────┘
                     ┌──────┴──────┐
                     │  Event Bus  │ (Kafka / RocketMQ)
                     └──────┬──────┘
              ┌──────────────┼──────────────┐
              ▼              ▼              ▼
        Service B    Service C    Event Store(持久化)
```

### CDN / 多级缓存最佳实践

**经典三级缓存架构**：

```
L1: 浏览器缓存（Cache-Control / ETag）
    → 命中率约 30～40%，完全零网络开销

L2: CDN 边缘节点缓存
    → 命中率约 40～50%，延迟降至 10ms 以内

L3: 应用层本地缓存（Caffeine / Guava）+ 分布式缓存（Redis）
    → 本地命中：<1ms，分布式命中：1～5ms

L4: 数据库（MySQL / PostgreSQL）
    → 10ms+，最后一道防线
```

**缓存更新策略**：

| 策略 | 写操作 | 读操作 | 一致性 | 适用 |
|------|--------|--------|--------|------|
| Cache-Aside | 先写 DB，后删缓存 | 先查缓存，miss 则读 DB 回填 | 最终一致 | 通用场景 |
| Write-Through | 同步更新缓存和 DB | 直接从缓存读 | 强 | 写频繁场景 |
| Write-Behind | 只写缓存，异步刷 DB | 直接从缓存读 | 弱 | 写吞吐极高 |
