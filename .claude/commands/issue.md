# /issue — Issue 深度分析与结构化输出

读取并深度分析指定的 GitHub Issue，输出标准化的分析报告，作为后续 Plan 和开发的输入。

---

## 用法

```
/issue <issue-number>              # 分析指定 Issue（本地或远程）
/issue <issue-url>                 # 通过 URL 分析
/issue --remote <issue-number>     # 强制从 GitHub 拉取最新
```

---

## 执行流程

### Step 1：获取 Issue 内容

**本地模式**（默认）：直接读取用户提供的 Issue 内容。

**远程模式**（`--remote` 或传入 URL）：
```bash
gh issue view <issue-number> --json number,title,body,labels,assignees,state,createdAt,updatedAt,url,comments
```

如果存在评论（comments 不为空），也一并拉取：
```bash
gh issue view <issue-number> --comments
```

### Step 2：结构化分析（必须按此模板输出）

输出文件：`docs/issues/analysis-{issue_number}.md`

---

---

### Step 3：缺失信息检测

如果 Issue 模板中的关键字段为空，必须在报告中明确标注：

- ⚠️ **缺少验收标准** → 阻塞，需要 Issue 创建者补充
- ⚠️ **缺少复现步骤** → 阻塞 Bug 修复
- ⚠️ **缺少涉及模块** → 增加分析不确定性
- ⚠️ **缺少技术限制** → 可能导致方案不符预期

---

## 输出

- 标准化分析报告: `docs/issues/analysis-{issue_number}.md`
- 终端摘要（用户可直接看到）

---

## 禁止

- 不要跳过"缺失信息检测"
- 不要在验收标准不明确时进入 Plan 阶段
- 不要修改 Issue 原始内容
